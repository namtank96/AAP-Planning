01

기업 AI 활용의 현 주소

① PoC 이후 정체 지속

활용 가능성은 확인했지만, 운영 전환은 아직

생성형 AI PoC는 기업의 AX 전환 가능성을 충분히 보여주었습니다. 다만 실제 업무 프로세스에 붙어 지속적으로 운영되는 사례는 아직 제한적입니다.

② 단일 기능 중심의 단순 도구로 활용

챗봇·문서 요약·정보 검색 등 개별 업무 단위로 제한

FAQ 챗봇, 문서 요약, 사내 검색, RPA 보조 등은 개별 업무의 생산성을 높여 주고 있습니다. 다만 업무 흐름 전체를 이해하고 다음 단계까지 이어서 처리하는 수준으로는 확장되지 못하고 있습니다.

③ 계속 높아지는 인적 의존성

판단·승인·시스템 반영은 여전히 사람에게 집중

AI가 답변과 초안을 만들어도 정책 판단, 예외 처리, 승인, 시스템 입력은 여전히 현업 실무진이 담당합니다. 결국 AI와 업무 시스템 사이에 남은 한 단계를 사람이 채우는 모양이 됩니다.

AAP의 출발점은 모든 업무를 AI로 자동화하는 것이 아닙니다.
AI가 처리할 수 있는 판단·실행 영역과 사람이 검토해야 하는 영역을 구분하고, 이를 고객의 업무 시스템과 통제 체계 안에서 안전하게 운영하는 구조로 전환하는 데 있습니다.
02

실제 운영으로 이어지기 어려운 이유

AI 기술 자체의 문제가 아닙니다. 실제 업무 운영으로 이어지기 위해 필요한 맥락·실행 흐름·통제 체계·학습 구조가 함께 연결되지 못하기 때문입니다.

요소현황한계
① 맥락사내 문서·FAQ 기반 검색과 답변, 일반적 요약·초안 작성고객사의 정책·프로세스·권한·예외 기준까지는 이해하지 못해 실제 업무 판단에 그대로 쓰기 어려움
② 실행AI가 답변·추천·문서 초안을 만들어 사용자에게 전달ERP·CRM·SCM·그룹웨어·결재 시스템에 실제 처리로 이어지는 실행 경로가 없음. 결국 사람이 다시 시스템에 입력
③ 통제검색 결과·답변·근거 문서를 제시어떤 근거로 판단했는지, 언제 사람이 검토해야 하는지, 어떤 실행을 막아야 하는지에 대한 기준과 이력 체계가 미정리
④ 학습사용자 피드백, 프롬프트 개선, 검색 품질 튜닝 등 단위 개선은 가능판단 근거·실행 결과·예외 처리·사람 검토 이력이 업무 지식·정책 개선·Agent 스킬로 재사용되는 구조는 부족
개별 AI 기능을 추가하는 방식만으로는 해결되기 어렵습니다. 업무 의미를 정리하고, Agent 실행 흐름을 설계하며, 시스템 반영과 운영 통제를 함께 묶어야 합니다.
03

왜 지금 AAP가 필요한가

기존 데이터 플랫폼을 대체하는 것이 아닙니다. 이미 갖춰진 Asset 위에 실제 업무 맥락·실행·통제·학습을 연결하는 운영 실행 계층을 결합하는 접근입니다.

3-1. 지금 Agentic AI가 필요한 세 가지 이유

기존 자산은 쌓였지만, 업무 실행과는 분리되어 있음

데이터 플랫폼과 업무 시스템에는 이미 많은 데이터·문서·처리 이력이 축적되어 있습니다. 다만 이 자산이 정책 판단, 승인, 예외 처리, 시스템 반영까지 이어지는 실제 업무 흐름과는 아직 충분히 연결되지 못하고 있습니다.

AI PoC는 가능성을 보였지만, 반복 운영 구조는 부족함

생성형 AI PoC는 문서 요약, 검색, 초안 작성의 가능성을 확인시켜 주었습니다. 이제 필요한 것은 단일 기능 검증이 아니라, 업무 기준·사람 검토·시스템 실행·운영 이력이 함께 돌아가는 지속 가능한 운영 구조입니다.

Agentic AI는 답변을 넘어 업무 흐름을 연결할 수 있음

Agentic AI는 단순 답변 생성에 머무르지 않고, 업무 단계를 계획하고 필요한 도구를 호출하며 처리 경로를 제안할 수 있습니다. 기업 환경에서는 이를 권한·감사·HITL 기준 안에서 실행 보조와 사람 검토로 연결하는 구조가 필요합니다.

3-2. AAP 정의와 구성 원칙

AAP는 단일 AI 기능이나 특정 솔루션이 아니라, 고객의 기존 자산·업무 실행 흐름·사용자 접점·보안 통제 기준을 하나의 운영 구조로 연결하는 접근입니다.

정의 고객의 데이터·문서·시스템을 업무 맥락으로 연결하고,
Agent가 업무 기준에 따라 판단 후보 생성·실행 보조·사람 검토 연계·시스템 반영까지 안전하게 이어지도록 만드는 기업용 Agentic AI 운영 체계
① 사용자 접점

Service Experience

현업 사용자·검토자·승인자가 Agent와 상호작용하는 접점입니다. 업무 요청, 근거 확인, 검토·승인, 결과 확인이 역할별 화면과 절차로 연결됩니다.

② 실행 흐름

AAP Execution Loop

입력 연결 → 업무 의미화 → 판단 후보·처리 경로 생성 → 사람 검토/시스템 반영 → 운영 이력 축적으로 이어지는 실행 흐름입니다. 운영 피드백은 다시 의미·판단 단계로 환류됩니다.

③ 연결 대상

Customer Environment

고객이 이미 보유한 데이터 플랫폼, 업무 시스템, 문서·KMS, 결재·그룹웨어 자산입니다. AAP는 이를 대체하지 않고 업무 실행 흐름에 연결합니다.

④ 공통 기준

Trust · Security · Governance

권한, 감사, HITL, 보안, 폐쇄망 요건은 전 영역에 적용되는 공통 기준입니다. Agent 실행, 사람 검토, 시스템 반영이 이 기준 안에서 운영됩니다.

3-3. Data Platform 상에서의 AI 역할 확장

기업 데이터 위에서 AI의 역할은 분석 보조에서 생성형 업무 보조로, 이제는 업무 실행 흐름을 연결하는 방향으로 확장되고 있습니다.
AAP는 앞선 단계를 대체하는 것이 아니라, 이미 갖춰진 데이터·분석·AI 자산 위에 업무 판단 후보·실행 보조·사람 검토·시스템 반영을 연결하는 업무 운영 구조를 더하는 접근입니다.

4-3. 수행 모델

05

적용 레퍼런스

두 검증 자산이 AAP 오퍼링으로 합쳐지는 방식 · Case 1 (금융 자문형)은 Data·Agentic·Service 축을, Case 2 (산업 도메인)는 Semantic 축을 검증했습니다. 남은 영역을 보완하면 반복 적용 가능한 운영형 AAP 패턴이 됩니다.

5-1. 금융 자문형 Agent 검증 사례

Case 1 · 금융 자문형 사례

멀티에이전트 기반 금융 자문 검증

도메인 자산운용 · 영역 투자 의사결정

시장 데이터·뉴스·매크로 지표를 통합 수집하고, 멀티에이전트가 단계적으로 분석해 투자 자문 초안을 생성하는 구조입니다. 데이터 수집부터 자문 산출까지의 End-to-End 흐름이 작동 가능함을 검증했습니다.

핵심 내용외부 데이터 수집 → 멀티에이전트 분석 → 자문 생성까지 End-to-End 검증. Vertical Agent 형태의 서비스 가능성 확인
검증된 AAP 영역· Enterprise Data & Inputs (외부 데이터 통합)
· Agentic Reasoning (멀티에이전트 분석)
· Service Experience (자문 생성·전달)
보완이 필요한 영역· Business Semantic Modeling (투자 정책·룰·권한 체계)
· Enterprise Integration & Action (주문·기록 시스템 반영)
· Operational Learning · Trust (Decision Log·HITL·감사)
AAP 인사이트AAP의 데이터 통합·Agentic Reasoning 축이 실제 구성 가능함을 입증했습니다. 여기에 Semantic·Integration·Learning을 더하면 자문형 업무를 운영 체계로 확장 가능한 적용 패턴이 됩니다.

5-2. 산업 도메인 모델링 검증 사례

Case 2 · 산업 도메인 모델링 사례

제조 도메인 산업단체 영향도 분석

도메인 공공·산업단체 · 영역 제조 도메인 영향도 분석

국제 정세·통상 규제·환율 변화를 산업 도메인 온톨로지로 정리해, 회원사별 영향도 분석과 통보 후보를 생성하고 담당자의 판단·후속 조치를 지원하는 의사결정 보조 사례입니다.

핵심 내용산업 도메인 개념·품목·회원사·규제 관계를 구조화하고, 영향도 판단 룰을 설계해 회원사별 차등 통보 가능성을 검증
검증된 AAP 영역Business Semantic Modeling (도메인 온톨로지·룰 설계) · 도메인 SME 협업 기반 모델링 방법론
보완이 필요한 영역Enterprise Data & Inputs (외부 데이터 자동 수집) · Agentic Reasoning (단계적 판단·도구 호출) · Enterprise Integration & Action (회원사 통보·리포팅 Workflow) · Trust · Governance (감사·보안·운영 로그)
AAP 인사이트AAP의 핵심 차별화 영역인 Business Semantic Modeling이 실제 산업 도메인에서 작동함을 입증했습니다. 데이터 수집·Agent 실행·통보 Workflow·운영 학습을 결합하면 공공·산업단체형 적용 패턴으로 재사용 가능합니다.

5-3. 두 레퍼런스가 함께 보여주는 것

축금융 자문형 사례산업 도메인 모델링 사례AAP 통합 시 의미
Data & Inputs외부 데이터 통합 검증외부 데이터 연계 필요외부 데이터 자동 수집과 고객 자산 연계 구조를 재사용 가능한 표준 패턴으로 정리
Semantic투자 정책·룰 구조화 필요산업 도메인 온톨로지 검증산업·업무별 온톨로지와 판단 룰 설계 방법론을 재사용
Agentic멀티에이전트 분석 검증Agentic 보조 판단 구조 필요멀티에이전트 기반 판단·계획 구조를 공통 운영 모델로 확장
Action / Service Outcome주문·기록 반영 필요회원사 통보 Workflow 필요시스템 반영·통보 Workflow와 사용자 검토 화면을 결합해 실제 업무 서비스로 전환
Learning · TrustDecision Log·감사 필요감사·보안·운영 로그 필요Decision Log·HITL·감사 체계를 붙여 운영형 AAP로 전환
두 사례는 각각 다른 AAP 축을 검증했습니다. AAP는 이 조각들을 하나의 통합 구조로 일반화해, 다른 고객·산업에 재적용 가능한 오퍼링으로 만드는 접근입니다. 두 사례의 보완 영역을 결합하면, 단순 분석 결과가 아니라 현업 사용자가 검토·승인·실행 결과를 확인하는 Service Experience까지 확장할 수 있습니다.

5-4. 가까운 시일의 도전

두 사례는 이미 서로 다른 AAP 축을 검증했습니다. 다음 단계는 각 사례의 빈 영역을 보완해, 반복 가능한 적용 패턴으로 일반화하는 것입니다.

  • Ops Rule · Decision Log·HITL 운영 기준 문서화
  • Pattern · 금융 자문형 AAP 적용 패턴 정리

산업 도메인 모델링 사례 다음 단계

  • Method · 온톨로지 모델링 방법론 표준화·재사용 가이드 정리

  • Ops Log · 감사·운영 로그 구조 정의
  • Pattern · 공공·산업단체형 AAP 적용 패턴 정리
공통 산출물 · 검증 자산을 보완 PoC로 확장하고, 금융 자문형·공공·산업단체형 적용 패턴으로 일반화해 다른 고객·업무에 재적용

고객 유형별 진입 패턴

공통 구조는 유지하되, 고객의 AI 성숙도와 통제 요구 수준에 따라 제안 메시지를 다르게 가져갑니다.

Type A. '실행형' AX로의 전환 및 확산이 어려운 고객

Data Platform·AI 조직 등 자체 자산은 갖췄으나 운영 전환에서 멈춤
고객사 상황
(1) PoC는 다수 진행했으나, 업무 시스템 반영·승인·감사까지 이어지는 운영 단계로 확장되지 못함
(2) 내부 AI 조직과 현업 사이 책임·범위가 불분명함
제안 접근 전략
기존 자산을 대체하지 않고 그 위에 Agentic 운영 계층(판단·실행·검토 연결)을 결합
고객 AI 조직과 역할을 나눠 운영 전환 가능한 1~2개 업무부터 단계적으로 확장
주요 타깃 영역
Enterprise Integration & Action · Operational Learning · HITL & Action Policy
초기 진입 방안
기존 PoC·Copilot 중 운영 전환 가치가 높은 1~2개를 선정해 Writeback·HITL·운영 학습을 더하는 확장 Pilot으로 시작

Type B. AX Readiness가 약한 고객

AX 전환을 위한 니즈는 있으나, 데이터·AI 자산이 부족함
고객사 상황
처리해야 할 반복 업무는 명확하지만, Data Platform·AI 운영 조직·Agent 구축 경험이 부족해 어디서 어떻게 시작해야 할지 결정이 어려움
제안 접근 전략
큰 플랫폼 구축이 아니라 업무 1개에 집중한 운영형 Pilot으로 시작. 업무 의미·정책·시스템 연계를 함께 정리해 운영 전환 가능한 구조를 작은 단위로 검증한 뒤 점진적으로 확장
주요 타깃 영역
(1) Business Semantic Modeling
(2) Service Experience (Vertical Agent)
(3)Enterprise Integration & Action
초기 진입 방안
반복 빈도·정책 명확성이 높은 단일 업무를 선정해 2~4주 Discovery + 8~12주 운영형 Pilot으로 시작. 검증된 업무 패턴은 다음 업무로 재사용

Type C. AX 허들이 지나치게 높은 고객

통제·감사·폐쇄망 요건이 AI 도입을 결정함(공공·금융·국방 등)
고객사 상황
업무 니즈는 분명하나, 권한·감사·망분리·외부 SaaS 사용 제한이 도입 조건. 자동 처리 범위가 매우 제한적이며 사람 검토·승인 비중이 높음
제안 접근 전략
Private/폐쇄망 환경 + HITL·감사·승인 체계가 기본 전제인 통제형 AAP로 접근. 자동 처리 영역과 사람 검토 영역의 경계를 사전 정의하고, 모든 처리 이력을 감사 가능한 형태로 보관
주요 타깃 영역
(1) Trust · Security · Governance · Private/Closed Deployment · HITL & Action Policy · Audit & Explainability
초기 진입 방안
폐쇄망 환경에서 운영 가능한 매뉴얼·규정 검색 또는 보고서 초안 작성처럼 위험도가 낮고 감사 가능성이 높은 업무부터 시작. 자동 실행 영역은 후속 단계에서 단계적으로 확대
