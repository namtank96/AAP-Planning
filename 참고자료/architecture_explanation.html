01전체 그림 — 한 화면으로 보는 아키텍처

AI Foundry는 5개 영역으로 구성됩니다 — 사용자 접점 (Experience Layer), 자산화 시스템 (Asset System), 의사결정 처리 흐름 (Decision Flow), 자기진화 제어 (Control System), 그리고 이 모든 것을 떠받치는 Foundation Layer.

핵심 명제 (Value Proposition). AI Foundry는 단순한 "AI 에이전트 도구"가 아닙니다. 기업이 매일 내리는 의사결정 — 정책, 규정, 절차, 판단 — 을 구조화된 자산으로 축적하고, 이 자산이 다음 의사결정에 자동으로 활용되도록 설계된 자기진화 Agentic 플랫폼입니다. 시간이 지날수록 시스템이 더 정확해지는 구조가 핵심입니다.

023축 구조의 핵심 — State · Runtime · Self-Improving

대부분의 AI 시스템이 "Runtime" 하나에만 집중합니다 (질문 → 답변). AI Foundry는 여기에 두 축을 더해 폐쇄 루프를 만듭니다.

State
Asset System

의사결정에 필요한 모든 자산을 관리합니다 — 정책, 지식, 스킬, 의사결정 기록, 암묵지. "무엇을 알고 있는가"의 영역.

Runtime
Decision Flow

실제 의사결정이 처리되는 흐름 — 정보 수집(Input) → 판단(Reasoning) → 실행(Execution). "어떻게 결정하는가"의 영역.

Self-Improving
Control System

품질 진단, 평가, 인간 검토, 학습 루프, 실험 관리. "어떻게 더 나아지는가"의 영역.

🔁자기진화 폐쇄 루프 — 시스템이 시간과 함께 똑똑해지는 이유

이 3축은 단방향이 아니라 폐쇄 루프로 연결됩니다:

  • Asset → Flow: Decision Flow가 실행될 때마다 Asset System의 정책·지식·스킬을 참조합니다.
  • Flow → Asset: Execution 결과가 Decision Log로 자동 기록되고, 새로운 Skill Package·Knowledge로 자산화됩니다.
  • Flow → Control: 실행 결과가 Quality Diagnostics·Evaluation으로 평가되고, HITL 검토를 거칩니다.
  • Control → Flow / Asset: 평가·학습 결과가 정책·프롬프트·온톨로지에 반영되어 다음 의사결정의 품질을 높입니다.
왜 이 구조인가? 일반 AI는 "한 번의 추론"만 잘하면 됩니다. 기업 의사결정은 다릅니다 — 같은 종류의 결정이 반복되고, 이전 결정과 일관성이 있어야 하며, 감사·설명·재현이 가능해야 합니다. State·Runtime·Self-Improving 3축은 이 요구를 모두 충족하는 구조입니다.

03Asset System — 의사결정 자산화 시스템

Asset System은 "의사결정에 필요한 것"과 "의사결정에서 나온 것"을 모두 자산으로 관리합니다. 두 영역으로 구성됩니다 — 5-Asset Model (자산 종류)과 Asset Operations (자산 운영).

📦5-Asset Model — 자산 5종

①
📋 Policy Pack — 정책 자산

의사결정의 기준이 되는 정책·규칙·절차·체크리스트. 표준화된 형태로 코드처럼 관리됩니다. 예: 지급 한도 정책, 감사 대응 절차.

②
🕸️ Ontology — 지식 자산 (Knowledge Map)

도메인 지식의 의미 모델. 9-타입 노드 (Fact · Dimension · Workflow · Event · Actor · Policy · Support · Domain · Decision)로 개념과 관계를 구조화합니다.

③
🧰 Skill Package — 스킬 자산

재사용 가능한 스킬·도구·에이전트의 패키지화. 한 번 만든 스킬을 여러 도메인에서 재활용할 수 있도록 표준화된 인터페이스로 배포합니다.

④
📜 Decision Log — 의사결정 자산

모든 의사결정의 근거·결과·피드백을 자동 기록. 완전한 추적성 (Audit Trail)을 제공하며, 자기진화 학습의 원천 데이터가 됩니다.

⑤
📚 System Knowledge — 암묵지 자산

벡터·문서·메타 지식 저장소. 인터뷰·내부 문서·메신저 등에서 추출한 암묵지를 명시적 자산으로 전환합니다. 검색·학습·재사용의 기반.

⚙️Asset Operations — 자산 운영

자산은 만들기만 해서는 의미가 없습니다. 찾기·재사용·관리·승계가 가능해야 합니다. 4가지 운영 모듈로 구성됩니다.

Catalog
📚 자산 카탈로그

메타데이터·태깅·검색. 전사 자산의 가시화와 Discoverability를 제공합니다. "누가 만들었고, 언제 사용했고, 어디에 적용 가능한가."

Lifecycle
🔁 라이프사이클

생성 → 검증 → 승인 → 아카이브. 자산 버전 관리와 Promotion 게이트 (개발 → 스테이징 → 운영). 변경 시 영향도 자동 추적.

Marketplace
🏪 마켓플레이스

자산 공유·재사용·외부 배포. Phase 4에서는 다도메인 자산 라이선싱으로 확장 — 한 조직에서 만든 자산을 다른 조직이 라이선스로 사용.

Governance
🛡️ 거버넌스

소유권·권한·보존정책·PII 마스킹. 조직별 + Cross-Org 4그룹 정책 (common · org · tacit · core_diff) 으로 정교한 접근 통제.

왜 자산화가 중요한가? 일반적인 회사에서 "이 결정 작년에 누가 어떻게 했더라?" 라는 질문에 정확히 답하기 어렵습니다. AI Foundry는 모든 의사결정이 Decision Log로 자동 기록되고, 정책·스킬·지식이 자산으로 누적되어 조직 지식의 영구 보관소가 됩니다. 직원이 떠나도 의사결정 노하우는 남습니다.

04Decision Flow — 의사결정 처리 흐름 (Runtime)

Decision Flow는 실제 의사결정이 처리되는 런타임입니다. 3개 Layer가 위로 쌓이며, 데이터는 아래에서 위로 흐릅니다 (Input → Reasoning → Execution).

1️⃣Layer 1 · Input Layer — 정보 수집 · 이해 · 맥락화

의사결정의 시작점입니다. 4개 모듈로 데이터를 수집·정형화·검색해서 Reasoning Layer로 전달합니다.

  • Ingestion: 문서·데이터·이벤트 수집. 내부 시스템, 외부 API, 파일, 스트림 등 모든 소스에서.
  • Structuring: 비정형 데이터를 구조화·정규화·분류. 메타데이터·태깅·엔티티 추출.
  • RAG & Retrieval: 검색·검색증강·맥락 구성. Vector DB · 키워드 · 하이브리드 방식.
  • Source Connectors: 내외부 시스템과의 표준 인터페이스. RDB · 도메인 시스템 · 외부 API.

🏞️ Lakehouse Medallion — 데이터의 물리 저장 계층

Input Layer 안에는 데이터가 적재되는 Medallion 아키텍처가 함께 있습니다. 원본을 잃지 않으면서 점진적으로 정제·가공해서 최종 사용자가 활용할 수 있는 형태로 만드는 3 zone 구조입니다.

🟫 Bronze · Raw Zone

원천 그대로 보존. 객체 잠금·보존정책 적용. CAD · MP4 · CSV · JSON · PDF 등 원본 포맷 그대로.

🟨 Silver · Curated Zone

정제·표준화·조인 가능. Apache Iceberg + Parquet 포맷으로 ACID·시간여행·스키마 진화 지원.

🟩 Gold · Mart / Feature Zone

분석/AI 사용 데이터셋. Feature Store · ML 학습/추론 데이터셋. BI · LLM · Agent가 직접 소비.

2️⃣Layer 2 · Reasoning Layer — 판단 · 추론 · 계획

실제 "AI 두뇌"가 작동하는 곳. 3개 핵심 컴포넌트가 협업합니다.

Orchestrator
🧠 Agent Brain

프로세스 오케스트레이션. 도구 호출·분기·병합. Multi-Agent 협업과 Tool Use 워크플로우. Plan → Act → Reflect 반복 패턴.

LLM Engine
🤖 Tiered Model

모델 선택·호출·응답 생성. 3-Tier 구조: Tier 1 Critical (SOTA) · Tier 2 Default (Enterprise) · Tier 3 Bulk (Local/Small). LLM Router가 작업에 맞게 자동 선택.

Policy Engine
📋 Rule Engine

정책·규칙·제약 적용. Asset System의 Policy Pack을 참조해서 비즈니스 로직 자동 적용. Rule · Constraint · Policy Pack 표준.

3️⃣Layer 3 · Execution Layer — 검증 · 실행 · 전달

판단을 실제 액션으로 옮기는 마지막 단계. 검증 게이트 → 실행 → 시스템 연계의 3단계.

Guard
🛡️ Governance Gate

정책 검증·위험 점검·승인 게이트. Policy Check · Risk Assessment · Compliance Check. Human Oversight 연결 지점.

Action Engine
🚀 액션 실행

액션 수행·패키징·배포 준비. Runbook 실행, 결과 패키징, 배포 단계. 결과는 Skill Package로 자산화.

Integration
🔌 시스템 연계

시스템 연계·배포·운영 전환. API · 외부 System · Runtime · To-Do 시스템과 연동. Decision Log 자동 기록 지점.

왜 역순(위→아래) 시각화인가? 사용자가 보는 결과(Execution)는 화면 위쪽, 데이터 깊이(Input)는 아래쪽. 데이터는 아래에서 위로 ↑ 흐릅니다. 마치 빙산처럼 — 사용자에게 보이는 결정은 작지만, 그 아래 거대한 데이터·맥락·정책이 받쳐주는 구조입니다.

05Control System — 자기진화 제어 시스템

Decision Flow가 한 번의 의사결정을 처리한다면, Control System은 모든 의사결정의 품질을 측정·개선합니다. 5개 모듈이 자기진화 폐쇄 루프를 형성합니다.

①
🔍 Quality Diagnostics

자산·정책 품질을 4대 영역으로 자동 진단: Missing (누락) · Duplicate (중복) · Overspec (과잉 명세) · Inconsistency (불일치). 정책 정합성 검사.

②
⭐ Quality Evaluation

다축 품질 평가 (5축 기준): 활용도 · 품질 · 생산성 · 매출 · 안전성. 정량 메트릭과 정성 평가를 결합. "이 자산·결정이 가치 있는가"를 측정.

③
👥 Human Oversight (HITL)

Human-in-the-Loop 검토. 자동화-검토의 균형 (예: 80-20-80 룰). Guard Gate와 Action Engine에 연결되어 위험도가 높은 결정은 사람이 최종 승인.

④
🔄 Continuous Learning Loop

학습·피드백·개선의 연속 루프. Decision Log를 학습 신호로 사용해서 스킬·프롬프트·온톨로지·정책이 자가 진화합니다.

⑤
🧪 Experiment & Versioning

실험·버전·변경관리. 모델·정책·프롬프트·데이터의 A/B 테스트와 회귀 추적. EvalOps와 MLOps 통합으로 변경의 영향을 정량적으로 측정합니다.

HITL은 왜 80-20-80인가? 자동화 비율이 너무 낮으면 (예: 30%) 인간 부담이 너무 큽니다. 너무 높으면 (예: 100%) 신뢰성을 잃습니다. 80% 자동화 · 20% 인간 검토 · 신뢰도 80% 보장이 현실적 균형점입니다. 위험도에 따라 검토 비율을 조정합니다 — 고위험 의사결정은 100% 인간 검토.

06Foundation Layer — 플랫폼 기반 + 물리 인프라

위의 모든 시스템은 견고한 기반 위에 작동합니다. 2개 줄로 구성됩니다 — 플랫폼 서비스(소프트웨어 기반)와 물리 인프라(하드웨어·네트워크).

⚙️플랫폼 서비스 — 8개 핵심 영역

☁️ Runtime

Container · K8s · Serverless · 엣지 · 멀티 리전. 분산 처리 기반.

💾 Data & Storage

DB · Object Store · Vector DB · Lakehouse (Iceberg · Parquet · ACID).

🔐 Identity & Access

SSO · RBAC · ABAC · 조직·역할·권한 · Cross-Org 4그룹 정책.

📊 Observability

Metrics · Logs · Traces · Monitoring · Alerting · 단일 통합 관제.

♾️ DevOps · CI/CD

Git · Pipeline · IaC · Build · Deploy · Test · GitOps · 자동화 표준.

🛡️ Security · Governance

Security · Privacy · Compliance · Audit · PII Mask · KMS · 망분리.

⚗️ MLOps · LLMOps

Model · Prompt · Data · LLMOps Pipeline · EvalOps 통합.

🧩 Plugin · Extensibility

Skill Plugin · MCP · External Integration · 25 표준 + 확장 API.

🏗️물리 인프라 — On-Premise / Hybrid Cloud

🖥️ Compute

Kubernetes (OpenShift / Rancher / vanilla) · 베어메탈 · VM · Container · Serverless.

🌐 Network

25/100 GbE · LB · WAF · 망분리 · VPC · Subnet · Cross-Region 라우팅.

💽 Storage

NVMe SSD · Object Store · 페타급 확장 · Tiered (Hot / Warm / Cold).

💾 Backup & DR

다중 사이트 백업 · BCDR · RPO/RTO 기반 복구 · Snapshot.

🏢 Deployment

On-Prem · Public · Hybrid · Multi-Cloud (AWS · Azure · GCP · KT Cloud · NCP).

왜 Hybrid가 중요한가? 금융·공공 도메인은 망분리 환경에서 작동해야 합니다. 그러면서도 외부 LLM 모델 (SOTA Tier 1) 활용이 필요합니다. Hybrid 배포 모델은 민감 데이터는 On-Prem 내부에서, 외부 모델 호출만 통제된 게이트웨이를 통하는 구조를 가능하게 합니다.

07흐름 시나리오 — 지급심사 의사결정 예시

실제 의사결정이 어떻게 처리되는지 구체적 예시로 살펴봅니다. 시나리오: "5억원 외주 용역 지급 요청서가 들어왔다. 승인할 것인가?"

1
Input · 요청서 수신지급 요청서 PDF가 시스템에 업로드됩니다. Ingestion 모듈이 PDF를 받아 Raw Zone (Bronze)에 원본 보존.
2
Input · 정형화 + 맥락 구성Structuring 모듈이 PDF에서 금액·계약처·기간·항목을 추출. 같은 계약처의 과거 지급 이력을 RAG로 검색 (Silver Zone).
3
Reasoning · 정책 적용Orchestrator가 LLM Engine (Tier 1)을 호출해서 "5억 이상 지급 정책"을 적용. Policy Engine은 Policy Pack에서 관련 정책 (예: "1억 초과 시 본부장 결재, 5억 초과 시 CFO 결재") 자동 조회.
4
Reasoning · 위험 평가Ontology의 지식 자산을 활용해 계약처의 재무 건전성·과거 분쟁 이력 확인. 정량+정성 평가로 위험도 산출.
5
Execution · Guard GateGuard 모듈이 정책 위반·위험 신호를 점검. 5억 초과이므로 자동 승인 불가 → Human Oversight (HITL)로 라우팅.
6
Control · HITL 검토담당자 화면에 자동 생성된 검토 요약 (근거·과거 이력·정책 인용·위험 점수)이 표시됨. 담당자는 승인 또는 추가 정보 요청.
7
Execution · Action 수행승인 시: Action Engine이 회계 시스템에 지급 등록. Integration & Delivery가 ERP에 자동 전달. 거부 시: 사유와 함께 신청자에게 반려.
8
Asset · Decision Log 자동 기록이 의사결정의 전 과정 (입력·근거·정책·검토자·결과)이 Decision Log에 자동 기록. 다음에 비슷한 요청이 들어오면 학습된 패턴이 자동 적용됨.
9
Control · Learning Loop매주 Quality Evaluation이 지급심사 결과를 평가. 잘못 분류된 케이스가 있으면 학습 신호로 LLM·정책에 반영. 다음 달에는 정확도가 향상됨.
핵심 차별점. 이 9단계가 매번 사람의 손을 거치지 않습니다. 단순 케이스 (예: 1000만원 일상 지급)는 1~7번이 모두 자동, HITL은 스킵. 복잡 케이스만 사람이 개입. "자동화 80% + 검토 20%"의 의미가 여기 있습니다.

08가치 명제 & 비즈니스 임팩트

기술 아키텍처가 비즈니스에 주는 실질 가치는 5가지입니다.

📋 정책 자산화 — 일관성

모든 정책이 Policy Pack으로 표준화되어 같은 조건에서는 같은 결정이 내려집니다. 담당자에 따라 결과가 달라지는 일이 사라집니다.

📜 완전한 추적성 — 감사 대응

Decision Log는 모든 결정의 근거·과정·결과를 보관합니다. 감독 대응 시간이 단축되고, "왜 그렇게 결정했나?"에 즉시 답할 수 있습니다.

🤖 자동화 80% — 생산성

반복 의사결정 업무가 80% 자동화됩니다. 담당자는 복잡·고위험 케이스에 집중할 수 있어 처리량과 품질이 동시에 향상됩니다.

📚 조직 지식 자산화 — 인력 의존도 감소

핵심 담당자의 암묵지가 System Knowledge로 명시화됩니다. 인력 이탈·인사 이동에도 의사결정 노하우가 조직에 남습니다.

🔁 자기진화 — 시간이 지날수록 정확해지는 시스템

Continuous Learning Loop가 매번의 결과를 학습합니다. 다른 어떤 시스템과도 다른 특징입니다 — 시간이 지나면 보통 시스템은 노후화되지만, AI Foundry는 더 정확해집니다. 1년 사용한 시스템보다 3년 사용한 시스템이 우월합니다.

09자주 묻는 질문 (FAQ)

Q1. 일반적인 RAG 시스템 / Copilot과 무엇이 다른가요?

RAG/Copilot은 "한 번의 추론"에 최적화됩니다 — 질문이 들어오면 검색해서 답합니다. AI Foundry는 의사결정의 전체 라이프사이클을 다룹니다: 정책 자산화 → 의사결정 실행 → 결과 기록 → 학습 → 다음 결정에 반영. RAG는 Decision Flow의 Input Layer (RAG & Retrieval)에 해당하는 일부 기능입니다.

Q2. 왜 5-Asset Model인가요? 자산을 더 늘리거나 줄일 수 있나요?

5종은 의사결정의 4가지 측면 + 1가지 토대를 커버합니다: 무엇을 따를지(Policy), 무엇을 알고 있는지(Ontology), 어떻게 행동할지(Skill), 무엇을 결정했는지(Decision Log), 토대 지식(System Knowledge). 도메인에 따라 하위 카테고리는 늘릴 수 있지만, 이 5축은 어떤 기업·어떤 도메인에서도 유효합니다.

Q3. Lakehouse를 왜 Asset System이 아닌 Input Layer에 두었나요?

Lakehouse는 데이터의 물리 저장 계층입니다 — Asset System의 5개 자산은 Lakehouse 위에 만들어진 의미적 추상입니다. 데이터는 Input Layer를 통해 들어와서 Bronze → Silver → Gold로 정제됩니다. Asset System은 Gold Zone 위에서 Policy Pack·Ontology·Skill 등의 자산을 정의합니다. 분리하는 이유는 데이터 엔지니어(Lakehouse 운영)와 도메인 전문가(Asset 정의)의 역할이 다르기 때문입니다.

Q4. Decision Flow가 왜 역순(위에 Execution, 아래에 Input)으로 그려져 있나요?

사용자가 보는 결과(Execution)가 화면 위쪽, 데이터의 깊이(Input)가 아래쪽입니다. 빙산 모델과 같은 의도 — 사용자에게 보이는 결정은 작지만 그 아래 거대한 데이터·맥락·정책이 받쳐주는 구조를 시각적으로 강조합니다. 데이터 흐름 화살표는 ↑ 위로 향합니다.

Q5. HITL의 "80-20-80"은 정확히 무엇을 의미하나요?

80% 자동화 + 20% 인간 검토 + 신뢰도 80% 보장의 균형점입니다. 모든 의사결정에 사람을 넣으면 처리량이 낮아지고, 모두 자동화하면 신뢰성을 잃습니다. 위험도 분류에 따라 자동/검토 비율을 조정하고, 검토를 거친 결정은 80% 이상의 정확도를 보장한다는 운영 기준입니다. 도메인·조직 정책에 따라 70-30, 90-10 등으로 조정 가능합니다.

Q6. 자기진화는 정말 가능한가요? 학습이 어떻게 일어나나요?

Decision Log의 결과 데이터가 학습 신호 역할을 합니다. 예: 자동 승인된 결정 중 사후 문제가 발생한 케이스 → 정책 조정. 자주 사용되는 LLM 프롬프트 패턴 → 프롬프트 라이브러리 업데이트. 새로 추가된 도메인 지식 → Ontology 확장. 이 학습이 매번의 추론 안에서가 아니라 운영 주기 (일/주/월)로 일어납니다. Continuous Learning Loop가 이 주기를 자동화합니다.

Q7. 망분리 환경에서도 외부 LLM (Tier 1 SOTA) 활용이 가능한가요?

가능합니다. Hybrid 배포 모델을 사용합니다 — 민감 데이터는 On-Prem 내부에서 처리하되, 일반화된 추론은 외부 LLM 호출 (통제된 게이트웨이 경유). PII·기밀 정보는 호출 전 자동 마스킹되고, Audit Bus가 모든 외부 호출을 기록합니다. 도메인에 따라 Tier 3 Local LLM만 사용하는 옵션도 가능합니다.

Q8. 도입 첫날부터 가치를 낼 수 있나요? 자산이 쌓이는 데 시간이 필요하지 않나요?

두 단계입니다. Day 1 가치: 기존 정책 문서를 Policy Pack으로 정형화하면 즉시 일관성 + 자동 적용 가능. RAG로 기존 문서 검색 + Q&A 즉시 작동. Day 90+ 가치: Decision Log가 쌓이면서 학습 루프가 작동, 자동화율 60% → 80%로 상승. Day 365+ 가치: 도메인 특화 자산이 누적되어 신규 케이스 처리 정확도가 도입 초기 대비 20-30% 향상. 시간이 지날수록 가치가 커지는 구조입니다.
