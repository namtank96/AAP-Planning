Proactive Pitch · Agentic AI Platform SI Offering

AAP는 Data Platform 이후, 기업 업무를 실제로 수행하게 하는 Agentic Execution Layer입니다.

고객의 데이터·문서·시스템을 업무 의미로 구조화하고, Agent가 판단·도구호출·검토·시스템 반영을 수행하며, 운영 결과를 지식과 스킬로 축적하는 구축형 플랫폼 오퍼링입니다.

선제안의 목적대규모 구축을 바로 요구하는 자료가 아니라, 고객이 아직 막연히 느끼는 “AI 운영 전환” 문제를 명확히 하고 다음 논의를 만드는 자료입니다.

고객은 이미 데이터 플랫폼과 AI PoC를 경험했습니다. AAP는 그 다음 단계의 질문에 답합니다.

02

기존 AI PoC가 운영으로 확장되지 못한 이유

AAP는 생성형 AI PoC의 한계를 플랫폼 구조로 해결하는 접근입니다.

Context

업무 맥락 부족

문서 검색은 되지만 정책·프로세스·권한·의사결정 기준을 업무 의미로 연결하지 못합니다.

Action

실행 연계 부족

AI 답변이 실제 ERP, CRM, 그룹웨어, 업무 시스템의 처리로 이어지지 않습니다.

Control

검토·통제 부족

위험 업무에서 사람 검토, 승인, 감사, 설명 가능성 체계가 약합니다.

Learning

운영 자산화 부족

결과·피드백·예외 처리 경험이 지식, 스킬, 판단 기준으로 누적되지 않습니다.

03

고객 유형별 적용 패턴

공통 Offering Map은 유지하되, 고객 유형별 진입 메시지는 다르게 가져갑니다.

Type A

대기업 / 자체 AI 조직 보유

이미 데이터 플랫폼과 AI 조직이 있으나, 운영 전환과 실제 업무 실행이 병목입니다.

제안 메시지

기존 플랫폼 위에 업무 실행형 Agentic Layer를 얹어 속도와 안정성을 높입니다.

Type B

산업 특화 중견

AI 조직은 크지 않지만, 반복 업무·문서·승인·예외처리 니즈가 명확합니다.

제안 메시지

업무와 시스템에 맞춘 구축형 AAP로 작은 영역부터 빠르게 시작합니다.

Type C

공공·금융·국방 등 규제/폐쇄망

보안, 감사, 망분리, 승인 체계가 도입 가능 여부를 결정합니다.

제안 메시지

폐쇄망·권한·감사·HITL을 포함한 통제형 AAP로 접근합니다.

06

Business Semantic Modeling — 왜 단순 RAG가 아닌가

AAP의 핵심은 문서를 검색하는 것이 아니라, 고객 업무의 의미·정책·권한·판단 기준을 Agent가 사용할 수 있는 구조로 만드는 것입니다.

01

Domain Ontology

업무 개념과 관계를 정의해 Agent가 고객 업무 용어를 이해하게 합니다.

02

Process Map

업무 흐름, 단계, 검토/승인 지점을 구조화합니다.

03

Decision Rule Repository

정책, 조건, 예외, 판단 기준을 재사용 가능한 규칙으로 관리합니다.

04

Skill Catalog

반복 가능한 Agent 업무 능력을 스킬 단위로 축적합니다.

05

System Map

업무가 어떤 시스템과 연결되는지 인터페이스와 권한을 정의합니다.

06

Authority Matrix

누가 무엇을 자동 실행하고, 무엇은 검토·승인해야 하는지 정의합니다.

Past / Present / Future Capability Continuum

갑자기 Agentic AI를 하겠다는 것이 아니라, 기존 수행 경험 위에 현재 기술을 조합하고 미래 Agentic 기술을 단계적으로 흡수합니다.

Past

이미 해온 수행 경험과 운영 노하우

  • KMS / AICC 운영 경험
  • 기간계·그룹웨어 시스템 연동
  • 폐쇄망·보안 환경 구축
  • ITO / 운영 경험
→

Present

지금 조합 가능한 솔루션과 기술

  • LLM / RAG / Copilot
  • Workflow / API / RPA 연동
  • 감사·로그·모니터링
  • 업무별 Agent Flow 설계
→

Future

Agentic AI 발전에 맞춰 내재화할 기술

  • 자기통제 / Guardrails 고도화
  • EvalOps / 품질 루프
  • Tool Registry 진화
  • 자율 실행 통제·운영 자산화

※ Past 항목은 실제 수행 레퍼런스 확인 후 고객/산업별로 조정하는 것을 전제로 합니다.

08

선제안에서 강조할 Top 4 차별화

Multi-LLM 등은 기본 기능으로 보고, 고객 의사결정에 더 직접적인 4가지를 전면에 둡니다.

1

Business Semantic Modeling

업무 의미, 정책, 의사결정 기준을 구조화해 단순 검색형 AI를 넘어섭니다.

2

Enterprise System Writeback

AI 답변에서 멈추지 않고 API/RPA/Workflow로 업무 시스템에 반영합니다.

3

Closed / Private Deployment

공공·금융·국방·제조의 폐쇄망/사내망/망분리 환경을 고려합니다.

4

HITL + Decision Control

위험 등급별 자동 처리, 사람 검토, 승인, 예외 처리를 통제합니다.

09

Why SI, Why Us

프론티어 모델이나 범용 AI 포털이 아니라, 고객 업무와 시스템에 들어가는 구축형 오퍼링이 필요한 이유입니다.

Accountability

구축·운영 책임 인수

단순 API 제공이 아니라 고객 환경에 맞는 설계, 연동, 운영 전환까지 책임집니다.

Embedded Knowledge

고객 업무 내재화

고객 업무, 정책, 예외, 권한 체계를 함께 구조화해 현업에 맞는 Agent 흐름을 만듭니다.

Deep Integration

깊은 시스템 통합

ERP, SCM, CRM, 그룹웨어, 레거시 시스템과 실제 실행·반영까지 연결합니다.

10

산업별 적용 예시

상세 시나리오는 고객별 Discovery에서 구체화하되, 선제안에서는 적용 가능성을 빠르게 보여줍니다.

제조

작업표준·설비 매뉴얼 Copilot, 품질/불량 원인 대응, OT망 보안 고려

유통

매장 운영 매뉴얼, 프로모션 정책, 반품/환불 예외 기준 지원

물류

배송 예외·클레임 처리, 보상 기준, 고객 안내 및 정산 연계

공공

민원·행정 지식 KMS, 법령/지침 근거 제시, 담당자 검토 화면

국방

폐쇄망 정비/운영 매뉴얼 Copilot, 교범 검색, 감사/권한 통제

교육

학사·교무·상담 지식 AI KMS, 규정/FAQ 기반 응대 지원

11

가능한 시작점 — Discovery Workshop

첫 단계는 대규모 구축이 아니라, AAP 적용 가능성이 높은 업무를 찾는 낮은 부담의 진단입니다.

Step 1

업무 후보 발굴

문서·정책·시스템·예외 처리가 많은 업무 1~2개를 선정합니다.

Step 2

데이터·문서·시스템 확인

사용 가능한 문서, 데이터 플랫폼, API/RPA 가능성, 보안 제약을 확인합니다.

Step 3

Pilot 범위 정의

8~12주 내 데모 가능한 Copilot, Tool Use, Writeback 범위를 정의합니다.
