## 0. 결론

v240은 방향성과 골격은 유지할 만하다. 다만 리뷰 Agent 기준으로 보면 아직 고객용 최종본으로 쓰기에는 다음 문제가 남아 있다.

1. 고객용 문서와 내부 검토용 문서의 경계가 일부 흐림
2. Hero와 01장 일부 표현이 Agent의 직접 판단·실행처럼 읽힐 수 있음
3. 04-1 Architecture Overview가 아직 “아키텍처 한판”으로 강하지 않음
4. 06장 국방/보안 영역이 미완성이고 산업별 핵심 AAP 영역 연결이 약함
5. 07장 Type A/B/C가 엄밀히 MECE하지 않음
6. HTML 문법·테이블 정합성·버전 표기 등 산출물 완성도 이슈가 있음

따라서 v250은 전면 재작성보다 **부분 리팩토링**이 적절하다.

### 1.3 조정해서 반영할 것

외부 전문가 의견 중 “별첨 A 삭제 + 08장 후속 논의 방향 신설”은 리뷰 Agent 원칙상 타당하지만, 현재 작업 방향에서는 **별첨 A 유지**를 전제로 한다. 따라서 별첨 A를 삭제하지 않고 다음 방식으로 조정한다.

- 별첨 A 제목을 `내부 검토용 · 사업화 확장 항목`으로 명확화
- 별첨 상단에 외부 공유 시 분리 운용 권장 문구 삽입
- 고객용 본문과 내부 검토용 내용을 시각적으로 분리
- 향후 외부 공유본에서는 별첨 A를 제거하고, 별도 08장 또는 별도 미팅 아젠다로 전환 가능

1. 사건 단위 정리
2. 정책·권한·예외 기준 반영
3. 처리 경로 제안
4. Workflow/API/RPA/Writeback 연결
5. HITL·감사·이력
6. 운영 학습 환류

## 6. 04장 Architecture Overview 재설계

### 6.1 현재 문제

현재 4-1은 세로 스택 구조라 이해는 쉽지만, 다음 한계가 있다.

1. Customer Environment가 아래 기반처럼 보이며, 입력·실행 대상이라는 의미가 약함
2. Trust · Security · Governance가 하단 박스 하나처럼 보임
3. Service Experience와 Service Outcome이 분리되지 않음
4. Review/Approval의 시각적 비중이 작음
5. Hero에서 “04장 통합 아키텍처가 핵심”이라고 했지만, 4-1의 한판 임팩트가 부족함

### 6.2 v250 권장 구조

```text
┌─────────────────────────────────────────────────────────────┐
│ Trust · Security · Governance                               │
│ 공통 기반 · 전 영역 적용                                      │
│ IAM · 권한 · 감사 · HITL · Guardrails · Private/Closed       │
│                                                             │
│  [Service Experience & Outcome]                             │
│  AI Agent Portal · Business Agents · Review/Approval         │
│  Workflow/Writeback · Ops Report                             │
│                                                             │
│  [AAP Execution Loop]                                       │
│  Enterprise Data & Inputs                                   │
│  → Business Semantic Modeling ★                             │
│  → Agentic Reasoning                                        │
│  → Enterprise Integration & Action                          │
│  → Operational Learning ★                                   │
│       ↺ Learning feedback to Semantic / Agentic             │
│                                                             │
│  [Customer Environment]                                     │
│  Data Platform · ERP/CRM/SCM · Groupware/Approval            │
│  Document/KMS · Internal API                                │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 4-1 설명 문구

```text
AAP는 고객 환경을 대체하지 않고, 기존 데이터·문서·업무 시스템을 실행 구조에 연결합니다.
현업 사용자는 Service Experience에서 업무를 요청하고 검토·승인하며,
AAP Execution Loop는 고객 자산을 입력으로 받아 업무 의미화, Agent 판단·계획, 시스템 반영, 운영 학습으로 이어집니다.
Trust · Security · Governance는 별도 기능이 아니라 전 영역에 적용되는 공통 운영 기준입니다.
```

### 6.4 핵심 시각 표현 원칙

- Trust는 하단 박스가 아니라 전체 아키텍처를 감싸는 외곽 프레임으로 표현
- Service는 `Service Experience & Outcome`으로 확장
- Customer Environment는 아래 기반이 아니라 “연결 대상이자 실행 대상”으로 보이게 표현
- Learning은 Semantic/Agentic으로 환류되는 루프를 표시
- `Review/Approval`과 `Workflow/Writeback`은 Service 또는 Outcome 영역에서 가시화

## 7. 04-2 Capability Map 고도화

### 7.1 유지 원칙

`Layer × 대표 Capability × 확장 컴포넌트` 방식은 유지한다.  
사용자가 중시한 “확장 컴포넌트가 잘 드러나는 구조”를 살려야 하기 때문이다.

### 7.2 chip 정리 원칙

- 각 Capability 카드당 chip은 5~6개 내외 권장
- 고객용에서는 “대표 확장 컴포넌트”만 표시
- 기술적으로 너무 세부적인 항목은 내부 전략 문서에서 상세화
- 그러나 확장성이 보이지 않을 정도로 과도하게 줄이지는 않음

### 7.3 범례 보완

현재 범례는 좋다.

```text
설계·통합 주도 영역
검증 자산·파트너 조합 영역
차별화 축적 영역
```

단, 아래 설명을 추가한다.

```text
AAP는 모든 기술 요소를 자체 개발하는 방식이 아니라,
고객 업무 기준에 맞게 검증된 AI·자동화 자산을 조합하고,
업무 의미화·시스템 연계·통제·운영 구조를 설계·통합하는 접근입니다.
```

### 7.4 카드·chip 보완

#### [5] Agentic Reasoning

| Capability | 추천 chip |
|---|---|
| Planning & Orchestration | Planner, Supervisor, Task Decomposition, State Memory, Work Context |

| HITL & Action Policy | Risk Scoring, Confidence Threshold, Escalation, Guardrail Hook, Human Review Rule |

#### [6] Enterprise Integration & Action

| Capability | 추천 chip |
|---|---|
