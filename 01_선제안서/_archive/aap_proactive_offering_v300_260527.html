Enterprise Agentic AI Platform

Agentic AI Platform은 고객의 데이터·문서·시스템을 업무 맥락으로 연결하고, Agent가 업무 기준에 따라 판단 후보를 만들며,
필요한 지점에서 HITL과 시스템 반영으로 이어지도록 만드는 구축형 플랫폼입니다.

01

Enterprise AI의 현 주소

기업들은 이미 Data Platform · LLM · RAG · Copilot 등 다양한 데이터·AI 기반을 확보하고, 다수의 PoC를 통해 활용 가능성을 확인했습니다.
하지만 많은 시도가 여전히 개인/일부 부서 단위의 파일럿에 머물며, 전사 조직 차원의 업무 운영으로 확장되는 데에는 구조적 병목이 남아 있습니다.

① 전사 조직 차원 표준 부재

개인별/부서별 단편적인 AI 도입 및 활용

용어·정책·권한·예외 기준을 회사 차원에서 일관되게 해석할 기반이 없습니다. 유사 PoC와 중복 투자가 반복됩니다.

② 시스템 실행 연계 부족

AI 결과와 실제 처리 흐름의 단절

RAG·Copilot이 답변·초안을 만들어도 ERP·CRM·결재·그룹웨어로 이어지지 않아, 사람이 결과를 복사·재입력해야 업무가 완료됩니다.

③ 신뢰·책임 경계 불명확

검토·승인 기준이 모호

AI 답변이 맞아 보여도 다시 확인해야 하는 부담이 남고, 자동/수동 경계·승인·책임 소재가 정리되지 않아 자동화 확대가 보류됩니다.

④ 운영 학습·재사용 부족

PoC 경험이 자산화되지 않음

프롬프트·검색 품질은 개선되지만 판단 근거·반려 사유·예외 대응이 조직 자산으로 남지 않아 매번 새로 시작합니다.

Enterprise AI의 다음 단계는 업무 기준 · 실행 경로 · 통제 책임 · 운영 학습을 하나의 운영 구조로 묶어, AI가 실제 업무 성과를 만들어내는 운영 중심 AI 구조로의 전환이어야 합니다.
02

'운영 중심' Enterprise AI로의 전환

01장의 4가지 구조적 병목은 AI 기능을 더 추가하는 방식만으로는 풀리지 않습니다. 운영 중심 AI로의 전환은 단계별 활용 흐름이 만들어낸 공통의 빈 영역을 인식하고, 데이터 활용 방식 자체의 패러다임 변화를 수용하며, 그것을 작동시키는 실행 구조를 함께 설계할 때 가능합니다.

2-1. Data Platform 의 단계적 확장: AI 활용 중심

Data Platform 위의 AI 활용은 분석 · 예측 AI → RAG · AI 검색 → Copilot UI 순으로 단계적으로 확장되어 왔으며, 단계마다 데이터 접근성과 단위 생산성이 향상되어 왔습니다. 그러나 판단 · 실행 · 통제 · 학습이라는 동일한 영역이 매 단계 빈 채로 남는 패턴이 반복됩니다.

2-2. Human Analytics에서 Agentic Operation으로

기존 Data Platform은 사람이 데이터를 보고, 리포트를 해석하고, 판단한 뒤 시스템에 반영하는 Human Analytics 구조를 전제로 발전해 왔습니다. AI Agent가 새로운 업무 수행 주체로 합류하면, 같은 데이터·문서를 Agent가 사람과 같은 의미·정책·권한으로 해석할 수 있는 구조와, Agent의 판단이 실행·검토·통제·학습으로 이어지는 운영 방식이 함께 필요합니다.

이 전환은 두 가지 축이 함께 진화한 결과입니다 — 자원 구조(데이터가 어떤 형태인가)와 수행 주체(누가 일하는가). 두 축을 각각 풀어보면 다음과 같습니다.

축 ① 자원 구조의 변화 왜 진화해야 했나 — Agent가 사람과 같은 의미·정책·권한으로 데이터를 해석할 수 있어야 함

축 ② 수행 주체의 변화 왜 진화해야 했나 — AI Agent가 새로운 업무 수행자로 합류하게 됨

2-3. Agentic Operation의 5가지 조건

Agentic Operation은 AI가 모든 판단을 자동으로 수행하는 방식이 아닙니다. 업무 의미화 · 판단 후보 · 실행 연계 · 사람 검토 · 운영 학습이 하나의 운영 구조로 함께 설계될 때 비로소 작동합니다.

업무 의미화
(Meaning) 데이터·문서·정책을 업무 용어, 권한, 예외 기준으로 재구성
판단 후보 생성
(Reasoning) 근거, 처리 경로, 위험도, 다음 단계 후보를 생성
실행 연계
(Action) Workflow, API, RPA, 업무 시스템 반영 경로 확보
HITL · 통제
(Control) 권한·위험도·승인 기준에 따라 자동화와 검토 경계 설정
운영 학습
(Learning) 판단·실행·예외·피드백을 재사용 지식과 Skill로 축적
Agentic Operation은 운영 중심 AI가 작동하는 방식입니다. 이를 기업 환경에서 반복 가능하고, 통제 가능하며, 확장 가능한 플랫폼 구조로 구현한 것이 03장에서 정의할 AAP (Agentic AI Platform)입니다.
03

Agentic AI Platform은 무엇인가

운영 중심 AI로의 전환은 목표를 해석하고, 필요한 데이터와 도구를 선택하며, 결과를 검토하는 Agentic AI가 전제되어야 하고,
그것을 기업 업무 안에서 안전하고 반복 가능하게 운영할 수 있게 하는 Platform 형태의 시스템이 필요합니다.

3-1. Agentic AI가 바꾼 AI의 역할: Reactive to Proactive

Agentic AI는 목표를 해석하고 데이터·문서·도구를 선택해 여러 단계의 업무 흐름을 계획·수행·검토하는 AI 패러다임입니다.
이제 Enterprise AI의 역할은 단발적 답변 생성이 아니라, 검토 가능한 실행 후보와 운영 이력을 만드는 목표 기반 업무 수행으로 이동합니다.

3-3. Agentic AI Platform

정의 (1) 고객의 데이터·문서·시스템을 업무 맥락으로 연결하고,
(2) Agent가 업무 기준에 따라 판단 후보 생성·실행 보조·사람 검토 연계를 수행하며,
(3) 그 결과가 시스템 반영과 운영 이력으로 안전하게 이어지도록 만드는(Agentic Operation) 기업용 플랫폼
① 업무 의미화

Agent 해석 기반 정제

Data Platform·KMS·문서·업무 시스템에 흩어진 업무 용어·정책·권한·예외 기준을 Agent가 동일한 의미로 해석할 수 있는 구조로 정리합니다.

② 시스템 반영

업무 실행 연결

Agent의 답변·추천이 화면에서 끝나지 않고, Workflow·API/RPA·ERP·CRM·결재 등 실제 업무 시스템의 처리 흐름으로 이어지게 합니다.

③ 통제 · 권한

안전한 실행 경계

Agent의 도구 호출과 실행 분기를 권한·승인·위험도·감사 요건에 따라 통제하고, 자동/사람 검토/보류의 경계를 Platform 수준에서 관리합니다.

④ 운영 학습

지속 개선 구조

판단 근거·도구 호출·사람 검토·실행 결과·예외 이력을 축적해 업무 지식·정책 기준·재사용 Skill 개선으로 환류시킵니다.

Agentic AI Platform은 업무 의미화 · 시스템 반영 · 통제 · 운영 학습을 하나의 구조로 묶어 공통 기반 위에서 Agentic AI가 안전하고 반복 가능하게 작동하도록 만드는 기업용 플랫폼입니다.
04

Agentic AI Platform 통합 아키텍처

AAP는 고객 자산(Data Platform, 문서·KMS, 업무 시스템, 보안·감사 체계 등) 위에 Agentic Operation Layer를 더해,
업무 의미화 → 판단 후보 생성 → 실행 연계 → 사람 검토와 통제 → 운영 학습을 하나의 운영 루프로 실행합니다.

AAP의 초기 적용은 정책·권한·예외 처리·시스템 반영·감사 이력이 중요한 산업과 업무에서 시작하는 것이 적합합니다.

[타깃 1] 금융·보험·자산운용

판단 기준과 감사 이력이 성과와 리스크를 좌우
고객 상황
규정·승인·감사 요구가 높고, 건별 판단 근거를 일관되게 남겨야 하는 업무가 많음
적용 업무
손해사정 심사, 내부통제 점검·보고 보조, 투자 리서치·자문 보조, 리스크 검토
적용 방식
약관·규정·계약·거래 이력을 사건 단위로 묶고, 업무 기준에 따라 판단 근거와 처리 경로 후보를 검토자에게 제시
Pilot 후보
손해사정 심사 보조, 내부통제 검토, 투자 리서치 보조

[타깃 2] 공공·협회·산업단체

도메인 관계 모델링이 정책 대응 속도를 좌우
고객 상황
정책·규제·회원사·산업 데이터가 분산되어 변화 발생 시 즉각 대응이 어려움
적용 업무
정책 영향도 분석, 회원사 차등 통보, R&D 정책 매칭, 규제 변화 리포트
적용 방식
산업 개념·품목·회원사·정책 관계를 구조화하고, 변화 이벤트에 따라 영향 대상과 대응 메시지 후보를 도출
Pilot 후보
통상규제 영향도 분석, R&D 정책 매칭, 회원사 통보 보조

[타깃 3] 제조·유통

적용 업무

적용 방식
규정·계약·생산·처리 이력을 업무 건 단위로 묶고, 처리 경로와 후속 조치 후보를 담당자 검토 후 업무 시스템에 반영
Pilot 후보

07

고객 유형별 진입 방식

AAP 도입은 고객의 AI 성숙도만으로 결정되지 않습니다. 기존 데이터·AI 자산, 반복 업무의 명확성, 보안·감사 요구 수준에 따라 진입 메시지와 Pilot 범위를 다르게 설계해야 합니다.

Type A

기존 Data/AI 자산 보유 고객

대상대형·중견 · 운영 계층이 비어 있는 고객
NEEDS 기존 투자(Data Platform·AI)를 유지하면서 분석·답변을 실행 흐름으로 잇고, 판단·승인 이력이 운영 자산으로 쌓이는 구조.
As-Is · 페인포인트

Data Platform·BI·RAG·Copilot 결과가 업무 처리로 이어지지 않아, 현업은 여전히 리포트 재해석 → 승인자 탐색 → 시스템 재입력을 반복. 운영 학습은 사람 머릿속에만 남음.

→
AAP 적용 (To-Be)
기존Data Platform + BI + RAG + Copilot
+ AAPAgentic Operation Layer (업무 의미화 · 판단 후보 · HITL · Writeback · 운영 학습)
기대효과

분석 → 판단 → 승인 → 시스템 반영이 단일 흐름으로 연결, Decision Log·Skill이 운영 자산으로 축적.

산업별 적용 예시
제조품질 이상 알림 → 조치 후보 → MES 반영
금융리스크 모니터링 → 대응 후보 → 정책 갱신
통신망 알람 → 대응 시나리오 → NMS·티켓 반영
Type B

산업 특화 중견 고객

대상반복 업무 구조화가 먼저인 고객
NEEDS 대형 플랫폼 선구축 없이 업무 단위로 시작하고, 성공 처리 패턴을 다른 업무로 확장할 수 있는 가벼운 구조.
As-Is · 페인포인트

반복 업무(문서 처리·승인·예외 판단·정산·품질 점검)는 명확하나 처리 기준이 담당자 경험에 의존해 품질·속도 편차가 큼. 매뉴얼·규정은 있어도 검색이 비효율적, 신규 직원 onboarding이 느림.

→
AAP 적용 (To-Be)
기존ERP·CRM 등 업무 시스템 + 사내 매뉴얼·규정·KMS
+ AAPRAG·Knowledge Pipeline + Business Semantic + Business Agent + 승인 Workspace
기대효과

반복 판단 품질이 담당자에 덜 의존, 성공 처리 패턴이 Skill·Template으로 축적되어 인접 업무로 확장.

산업별 적용 예시
유통반품 사유 분류 → 처리 가이드 → 승인
제조작업지시서 검토 → 예외·누락 식별
보험청구 사유 분류 → 약관 검토 → 처리 가이드
Type C

공공·금융·국방 고객

대상통제 요건이 먼저인 고객
NEEDS 자동 실행보다 검토 가능한 판단 보조부터 시작하고, 모든 처리가 감사·재현 가능해야 함.
As-Is · 페인포인트

AI 기술 가능성은 확인돼도 근거·권한·승인 이력·감사·망분리가 확보되지 않으면 운영 적용 보류. 외부 클라우드 LLM 제약, 모든 판단에 설명 가능성과 인용 근거 필요.

→
AAP 적용 (To-Be)
기존폐쇄망 업무 시스템 + 규정·내규·교범 + 결재 체계
+ AAPPrivate/폐쇄망 배포 + 근거 인용 RAG + Decision Logging · Audit Trail · HITL Workspace
기대효과

모든 판단에 근거·인용·승인 이력이 남아 규정 위반 위험 감소, 폐쇄망에서도 통제 가능한 AI 운영.

산업별 적용 예시
공공민원 분류 → 법령·내규 검색 → 회신 초안
금융여신·심사 시 규정 검색 → 근거 제시
국방작전·교범 검색 → 규정 위반 검토

후속 논의 방향

대상 업무를 정하고, 기존 데이터·문서·시스템 자산을 확인한 뒤, 업무 의미·정책·권한 기준과 Pilot 범위를 구체화하는 순서로 진행

7-1. 후속 논의 5단계

각 단계의 산출물이 다음 단계의 입력으로 이어집니다. 구현 방식은 고객 환경에 따라 직접 구축, 자사·상용 솔루션 활용, 외부 검증 자산 조합으로 구체화합니다.

단계핵심 질문산출물
1. Data Foundation 진단어떤 데이터 플랫폼·BI·문서·업무 시스템을 보유하고 있는가?Customer Environment Map
2. Semantic Foundation 진단어떤 업무 개념·정책·권한·판단 기준이 필요한가?Business Semantic 후보
3. Build · Buy · Borrow 매핑어떤 capability를 직접 구축하고, 어떤 자산을 활용·차용할 것인가?Capability 확보 전략 (별첨 A 참조)
4. Agentic Pilot 설계어떤 업무를 첫 Pilot으로 검증할 것인가?Pilot Scope · HITL · Action Flow
5. Operational Learning 설계실행 결과와 피드백을 어떻게 축적·개선할 것인가?Decision Log · Feedback Loop · Now-Near-Next Roadmap

각 질문은 5단계 진단의 시작점이 됩니다.

① 기반 자산

Data Platform 범위

현재 Data Platform, BI, Catalog, Lakehouse, 문서·KMS, 업무 시스템은 어떤 범위로 구축되어 있는가?

② 업무 흐름

수작업 잔여 영역

분석 결과 이후 사람이 수작업으로 판단·승인·시스템 반영하는 업무는 무엇인가?

③ Agent·HITL 경계

판단·검토 경계

AI Agent가 판단 후보를 만들 수 있는 업무와 반드시 사람이 검토해야 하는 업무는 어떻게 나뉘는가?

④ 시스템 반영

Writeback 가능 영역

시스템 Writeback이 가능한 업무와 검토·승인까지만 가능한 업무는 무엇인가?

⑤ 학습 환류

로그·환류 구조

운영 결과를 정책·룰·Agent Skill 개선으로 환류할 수 있는 로그 구조가 있는가?

⑥ 기술 확보 전략

Build · Buy · Borrow

어떤 capability는 내부 구축이 필요하고, 어떤 capability는 자사·상용·외부 검증 자산 활용이 적합한가?

A

별첨 · 사업화 추진 및 확보 전략

본 별첨은 AAP 사업화를 위해 무엇을 직접 주도하고, 무엇을 보유 자산과 통합하며, 무엇을 외부 전문 역량으로 빠르게 보완할지를 정리한 추진 전략입니다. 고객 업무 의미화·시스템 통합·운영 통제는 KT DS가 주도하되, 기술 변화가 빠른 LLM·Agent Framework·EvalOps·Guardrail 등은 외부 후보를 병렬 검증해 Time-to-Market을 단축합니다.

원칙 설명

Accelerate · 병렬 검증과 단계 확장 외부 의존 영역은 단일 업체에 고정하지 않고, 후보군 Pool 구성 → 기술 미팅/샌드박스 검증 → 표준 조합안 확보 → 고객별 적용 순서로 빠르게 가속화합니다.
