## 표지 (Hero)

### 제목
기업의 실제 업무를 E2E로 수행하는 Enterprise Agentic AI Platform

### 부제
AAP는 고객의 데이터·문서·시스템을 업무 의미로 구조화하고, Agent가 판단·실행·검토하며, 운영 결과를 지식과 스킬로 축적하는 구축형 Agentic AI Platform입니다.

### 목적
공통 고객 대상으로 Data Platform 이후의 다음 과제로 AAP를 인식시키고 후속 논의를 만드는 자료입니다.

## 01 · 기업 AI 활용의 현재 위치

**리드 문장**
많은 기업이 생성형 AI PoC, 챗봇, 문서 요약, 검색 자동화까지는 경험했습니다. 이제 다음 과제는 **AI를 실제 업무 흐름 안에서 실행 가능한 운영 체계로 전환하는 것**입니다.

### 카드 1 — 가능성은 확인했지만, 운영 전환은 제한적
생성형 AI PoC는 데모와 파일럿 단계에서 충분한 가능성을 보여주었습니다. 그러나 실제 업무 프로세스에 연결되어 지속적으로 사용되는 운영형 사례는 아직 제한적입니다.

### 카드 2 — 챗봇·요약·검색 등 점 단위 활용에 머무름 (배지: ② 단일 업무 자동화)
FAQ 챗봇, 문서 요약, 사내 검색, RPA 보조 등은 개별 업무의 생산성을 높이고 있습니다. 다만 업무 흐름 전체를 이해하고 다음 단계까지 이어서 처리하는 구조로는 확장되지 못하고 있습니다.

### 카드 3 — 판단·승인·실행은 여전히 사람이 담당 (배지: ③ 인적 개입의 지속)
AI가 답변과 초안을 만들더라도 정책 판단, 예외 처리, 승인, 시스템 입력은 여전히 현업과 IT가 수행합니다. 결국 AI와 업무 시스템 사이의 마지막 공백을 사람이 메우는 구조가 남아 있습니다.

## 02 · AI의 운영 전환을 막는 4가지 공백

**리드 문장**
AI PoC가 운영으로 확장되지 못한 이유는 기술 가능성이 부족해서만은 아닙니다. 실제 업무에 적용되기 위해 필요한 **맥락·실행·통제·학습 구조**가 함께 갖춰지지 않았기 때문입니다.

### 카드 1 · ① 맥락 — 업무 맥락 부족
문서 검색과 답변은 가능하지만, 고객사의 정책, 프로세스, 권한, 예외 기준까지 이해하지는 못합니다.

### 카드 2 · ② 실행 — 시스템 실행 연계 부족
AI의 답변이나 추천이 ERP, CRM, SCM, 그룹웨어, 결재 시스템의 실제 업무 처리로 이어지지 못합니다.

### 카드 3 · ③ 통제 — 신뢰·통제 체계 부족
어떤 근거로 판단했는지, 언제 사람이 검토해야 하는지, 어떤 실행을 제한해야 하는지에 대한 기준이 약합니다.

### 카드 4 · ④ 학습 — 운영 학습 구조 부족
질문, 답변, 실패, 예외 처리 이력이 다시 업무 지식과 Agent 스킬로 축적되는 구조가 부족합니다.

## 03 · 왜 지금 AAP가 필요한가

**리드 문장**
AAP는 기존 데이터 플랫폼이나 AI PoC를 대체하는 개념이 아닙니다. 이미 구축된 데이터·AI 자산 위에 **업무 맥락, 실행, 통제, 학습을 연결하는 Agentic Layer**를 더하는 접근입니다.

### 카드 1 · Data Platform 이후 — 데이터 기반은 마련됐지만, 업무 실행 계층은 비어 있음
데이터 플랫폼은 수집, 분석, 시각화 기반을 만들었습니다. 그러나 정책 판단, 승인, 예외 처리, 시스템 반영은 여전히 별도의 업무 절차와 사람의 개입에 의존하고 있습니다.

### 카드 2 · PoC 이후 — AI 실험을 운영 가능한 구조로 전환해야 함
생성형 AI PoC를 통해 가능성은 확인되었습니다. 이제는 단일 기능 검증을 넘어, 업무 맥락과 시스템 실행, 검토 체계, 운영 학습이 결합된 플랫폼 구조가 필요합니다.

### 카드 3 · Agentic AI 흐름 — 답변형 AI에서 업무 수행형 AI로 이동
프론티어 모델과 Agent 프레임워크의 성숙으로 AI의 역할은 답변 생성에서 계획, 도구 호출, 실행 보조로 확장되고 있습니다. 기업 환경에서는 이를 안전하게 운영할 수 있는 AAP 구조가 필요합니다.

### 비교표 — Data Platform vs 개인 AI Tool vs AAP

| 구분 | Data Platform | 개인 AI Tool (GPT, Claude, Copilot 등) | AAP |
|---|---|---|---|
| **목적** | 데이터 수집·저장·분석 | 질의응답·요약·초안 생성 | **업무 이해·판단·실행·검토** |
| **핵심 단위** | 데이터셋, 파이프라인, 대시보드 | 프롬프트, 대화, 문서 | **업무 프로세스, 정책, 도구, 결정, 실행 로그** |
| **결과물** | BI, 리포트, 데이터 서비스 | 답변, 문서 초안, 요약 | **Copilot, Workflow, API/RPA 실행, 시스템 반영** |
| **02의 4가지 공백을 어떻게 메우나** | 맥락·실행·통제·학습 모두 사람이 수행 | 맥락·실행·통제·학습 연계 약함 | **① 업무 온톨로지로 맥락 확보 / ② Tool·Workflow로 실행 연결 / ③ 감사·승인·HITL로 통제 / ④ 실행 로그·피드백으로 학습** |

### AAP는 무엇이며, 어디로 향하는가 (AAP 정의 박스)

**태그** · AGENTIC AI PLATFORM · 정의
**부제** · 개별 AI 기술의 묶음이 아닌, 기업 업무를 처음부터 끝까지 수행하는 닫힌 루프 구조

**본문**
업무 의미를 이해해 판단·실행하고, 그 결과를 학습 자산으로 축적하는 **기업용 Agentic AI Platform**. Data Platform·PoC 자산 위에 얹혀 **운영 가능한 업무 실행 계층**을 더합니다.

### AAP 닫힌 루프 도식

- **상단 띠** · Service Experience · 고객이 사용하는 AI 업무 면
- **01 Semantic** — 의미 이해
- **02 Agentic** — 판단·계획
- **03 Action** — 시스템 실행
- **04 Learning** — 운영 학습
- **루프 캡션** · 운영 학습이 다음 의미·판단에 반영 — 시간이 갈수록 더 정확해지는 닫힌 루프
- **하단 띠** · Trust · Foundation · 보안·감사·HITL·폐쇄망

### 왜 단순 RAG가 아닌가 · 진흥회 회원사 통보 시나리오

**단순 RAG**
"미국이 X 품목에 25% 관세를 부과합니다." — 문서를 검색해 요약만 합니다.

**AAP**
1. **회원사 매핑** · Knowledge Graph로 이 회원사가 X 품목을 어느 시장에 수출 중인지 확인
2. **영향도 판단** · 관세·환율·규제 룰로 회원사별 매출 영향을 계산
3. **차등 통보** · 회원사 등급·노출도에 따라 통보 양식·우선순위·대응 가이드를 다르게 적용

**캡션** · RAG는 *문서 검색의 기법*이고, AAP는 **업무 의미·정책·권한을 Agent가 사용할 수 있는 구조로 전환**합니다. 이 구조는 06장 진흥회 케이스에서 이미 검증된 영역입니다.

### 어디로 향하는가 · 운영 고도화 3대 축

- **학습 누적** · 실행 이력과 검증된 스킬이 다음 업무에 재사용 — 시간이 갈수록 정확도·재사용도 증가
- **자동 분기 · 예외 운영** · 위험도·권한 기준의 자동/검토 분기, 실패 건은 사람 검토와 재실행으로 회수
- **품질 · 비용 통제** · 품질 회귀 테스트 정착 + 업무 중요도에 맞춘 모델 사용량·쿼터 관리

## 04 · AAP Offering Map & Capability Detail

### 헤더
- **타이틀** · 7개 영역 기반 AAP 오퍼링 맵
- **설명** · 외부 제언의 자산화·통제·학습 관점은 어려운 아키텍처 용어 대신, 업무 실행 이력·품질 점검·스킬 재사용·운영 리포트로 풀어 넣었습니다.
- **범례** · ■ 기본 구성 / 즉시 오퍼링 · □(점선) 확장·고도화 후보

### Customer Environment (AAP가 새로 대체하지 않고 연결하는 고객 기존 자산)
Data Platform · ERP / CRM / SCM · Groupware / KMS · Document Repository · External API · Security Zone

### 영역 1 — Service Experience (현업 사용자가 접하는 업무 서비스)

- **AI Agent Portal** — Agent 검색, 실행, 이력 확인을 제공하는 업무 진입점
  - 칩: Agent Search · Execution · History · Permissions
- **Horizontal Agents** — 부서 공통 업무를 빠르게 적용하는 표준 Agent
  - 칩: HR · Document · CS · Approval
- **Vertical Agents** — 산업·직무별 판단 기준을 반영한 전문 Agent
  - 칩: 제조 · 금융 · 공공 · 의료
- **Review / Approval UI** — 사람 검토가 필요한 단계에서 승인과 근거 확인을 지원
  - 칩: Approval UI · HITL Console · Audit View
- **(확장) 업무 스킬 카탈로그** — 검증된 Agent 스킬을 조직 안에서 찾아 재사용하는 구조
  - 칩: Internal Catalog · Skill Pack · Versioning

### 영역 2 — Enterprise Data & Inputs (고객 데이터·문서·시스템 연결)

- **System Connectors** — 기간계·업무 시스템을 Agent 실행 흐름에 연결
  - 칩: API · DB · File · Legacy
- **Data Platform Integration** — 기존 데이터 플랫폼과 분석 자산을 AAP 입력으로 활용
  - 칩: Lakehouse · Warehouse · Catalog · Iceberg
- **Document / Knowledge Integration** — 문서·규정·FAQ를 검색 가능한 업무 지식으로 전환
  - 칩: RAG · OCR · Parser · Hybrid Search
- **Metadata & Lineage** — 데이터 출처, 스키마, 변경 이력을 관리해 신뢰성을 확보
  - 칩: Schema Registry · Lineage · Tagging
- **(확장) Event / External Interface** — 외부 데이터와 이벤트를 업무 판단 흐름에 반영
  - 칩: Public API · Subscription · Stream

### 영역 3 — Business Semantic Modeling (업무 의미·정책·판단 기준 구조화) [Core]

- **Domain Ontology** — 업무 개념과 관계를 표준화해 Agent가 같은 의미로 이해
  - 칩: OWL · RDF · SKOS · Taxonomy
- **Knowledge Graph** — 업무 지식, 규정, 시스템 관계를 그래프로 연결
  - 칩: Property Graph · SPARQL · Cypher · Neo4j
- **Process / Decision Map** — 업무 절차와 판단 규칙을 실행 가능한 기준으로 정리
  - 칩: BPMN · DMN · Rule Repository
- **Authority Matrix** — 업무 권한, 승인 기준, 예외 처리 조건을 구조화
  - 칩: RBAC · ABAC · Approval Rule
- **(확장) Semantic Quality Check** — 누락·중복·불일치 지식을 찾아 정비 후보를 제시
  - 칩: Missing · Duplicate · Inconsistency

### 영역 4 — Agentic Reasoning & Execution (Agent 판단·계획·도구 호출)

- **Planning & Orchestration** — 복잡한 업무를 단계로 나누고 Agent 실행 흐름을 제어
  - 칩: LangGraph · AutoGen · Semantic Kernel
- **Work Context Memory** — 업무 대화, 중간 결과, 참조 정보를 작업 맥락으로 유지
  - 칩: Session · Vector Store · Long-term Memory
- **Tool Registry / Runtime** — Agent가 호출 가능한 도구와 API를 등록·통제
  - 칩: MCP · OpenAPI · Function Calling
- **HITL & Action Policy** — 위험도와 확신도에 따라 자동 실행과 사람 검토를 분기
  - 칩: Confidence · Escalation · OPA · Guardrails
- **(확장) Agent Interop** — 조직·시스템별 Agent가 업무를 넘겨받아 협업하는 구조
  - 칩: A2A · Handoff · Multi-agent
- **(확장) Cost-Aware Routing** — 업무 중요도, 응답속도, 비용에 따라 모델과 실행 경로를 선택
  - 칩: Model Router · Priority · Cost Policy

### 영역 5 — Enterprise Integration & Action (업무 실행·시스템 반영)

- **Workflow Engine** — 승인, 예외, 재처리 흐름을 업무 프로세스로 실행
  - 칩: Camunda · Temporal · n8n
- **API / RPA Connector** — API가 없는 시스템까지 실행 자동화 범위를 확장
  - 칩: Kong · Apigee · UiPath · Power Automate
- **Writeback Connector** — Agent 결과를 실제 업무 시스템에 반영
  - 칩: ERP · CRM · Groupware · Approval
- **Report / Delivery** — 결과를 보고서, 대시보드, To-Do 형태로 전달
  - 칩: Dashboard · Email · To-Do
- **(확장) Exception Queue** — 실패·예외·저신뢰 실행 건을 사람 검토 대기열로 전환
  - 칩: Retry · Manual Review · SLA
- **(확장) Transaction Guard** — 실행 전 확인, 재시도, 취소 조건을 업무 정책으로 관리
  - 칩: Pre-check · Rollback Rule · Confirm
- **(확장) Runbook Packaging** — 반복 실행 가능한 업무 절차를 운영 패키지로 정리
  - 칩: Runbook · Deploy · Operations

### 영역 6 — Operational Learning (실행 결과의 축적과 개선) [Learning]

- **Decision / Action Log** — 판단 근거, 도구 호출, 실행 결과를 추적 가능한 이력으로 남김
  - 칩: Reasoning Trace · Tool Trace · I/O Capture
- **Feedback Store** — 사용자 피드백과 업무 결과를 개선 데이터로 축적
  - 칩: User Feedback · Outcome Data · Survey
- **Evaluation Pipeline** — 답변·실행 품질을 정기적으로 검증하고 회귀를 확인
  - 칩: Eval Set · Regression · A/B Test · MLflow
- **Skill Library** — 검증된 프롬프트, 도구 사용법, 업무 절차를 재사용 자산으로 관리
  - 칩: Validated Skill · Versioning · Reuse
- **(확장) Skill Approval Workflow** — 새 업무 스킬을 검증·승인한 뒤 운영 환경에 배포
  - 칩: Review · Approve · Publish
- **(확장) Improvement Candidate** — 운영 이력에서 정책·스킬·지식 개선 후보를 제안
  - 칩: Policy · Skill · Knowledge
- **(확장) Experiment & Versioning** — 모델, 프롬프트, 데이터 변경을 비교하고 안전하게 배포
  - 칩: Prompt · Model · Data Version

### 영역 7 — Trust · Security · Governance (보안·감사·운영 통제)

- **IAM / SSO** — 사용자, 조직, 역할별 접근 권한을 통합 관리
  - 칩: OIDC · SAML · MFA · RBAC
- **Data Security** — 민감정보와 업무 데이터를 보호하고 사용 범위를 제한
  - 칩: Encryption · Masking · Tokenization
- **Audit / Explainability** — 근거, 출처, 실행 이력을 감사 가능한 형태로 제공
  - 칩: Evidence · Citation · Action Lineage
- **Guardrails / Policy** — 금지 작업, 민감정보, 위험 실행을 정책으로 통제
  - 칩: Input Filter · Output Filter · Tool Policy · OPA
- **Closed / Private Deployment** — 공공·금융·국방 환경에 맞춘 폐쇄형 배포 옵션
  - 칩: On-premise · VPC · Air-gapped
- **(확장) Usage & Cost Control** — 조직·업무별 모델 사용량, 비용, 쿼터를 운영 정책으로 관리
  - 칩: Quota · Budget · Rate Limit
- **(확장) Agent Observability** — Agent 실행 단계, 지연, 오류, 도구 호출을 운영 관점에서 관측
  - 칩: OpenTelemetry · Trace · Metrics
- **(확장) Quality Review** — 품질 점수와 리스크를 운영 리포트로 관리
  - 칩: Quality Score · Risk Check · Ops Report

## 05 · 적용 레퍼런스 — 검증된 것과 사업 확장 방향

**리드 문장**
검증된 두 케이스를 봅니다. 각각이 Offering Map의 **어떤 부분을 입증**했고, KT DS가 **어떻게 사업화·확장**해 갈 것인지 로드맵과 전략 액션으로 풀어봅니다.

### Case 1 · AI 투자 메이트

- **배지** · Reference · 내부 프로젝트
- **도메인** · 자산운용 · 투자 의사결정
- **개요** · 매크로·뉴스·블룸버그 데이터를 통합해 투자 자문을 생성하는 LangGraph 기반 멀티에이전트. 데이터 수집 → 멀티에이전트 분석 → 자문 생성까지 E2E 검증 완료.

**✓ 검증된 영역**
- **Inputs** · 외부 데이터(블룸버그·뉴스·매크로) 통합
- **Agentic** · LangGraph 멀티에이전트 분석·자문 생성
- **Service** · Vertical Agent 형태의 자문 서비스화

**+ AAP로 보완**
- **Semantic** · 투자 정책·룰·권한 구조화
- **Action** · 주문·기록 시스템 Writeback / Workflow
- **Learning · Trust** · Decision Log · HITL · 감사·설명성

#### 사업 확장 로드맵 · 단일 자문 Agent에서 Vertical Decision Pack으로
자산운용 1차 진입 → 자문형 의사결정 영역(법무·세무·M&A) 패턴 복제 → Vertical Decision Pack 자산화.

| 시점 | 활동 |
|---|---|
| 현재 · FY26 1H | 내부 E2E 검증 완료, **외부 영업 자료화** 진행 |
| 단기 · FY26 2H | KT그룹 금융 채널 1차 PoC, **외부 자산운용사 1-2곳 영업 진입** |
| 중기 · FY27 | 자문형 의사결정(법무·세무·M&A)으로 **패턴 복제** |
| 장기 | Vertical Decision Agent **표준 Pack** 자산화 · 산업별 복제 |

**PLAY 1 · 금융 컴플라이언스 패키징**
운용 한도·감사 추적·HITL 통제를 AAP의 **Semantic·Foundation**에 매핑한 "금융 자문 AAP" 패키지 제작 → 금융권 영업 시 즉시 제시 가능

**PLAY 2 · 파트너 결합형 통합 SI 카드**
비투엔(RAG) · 인핸스(온톨로지) 결합 통합 SI 제안서로 **자체 모델 한계를 파트너 자산으로 보완** → 수주 경쟁력 확보

### Case 2 · 한국기계산업진흥회 온톨로지

- **배지** · POC 선제안 · 1차 완료
- **도메인** · 공공·협회 · 기계산업
- **개요** · 국제 정세·통상규제·환율 변화를 기계산업 도메인 온톨로지로 구조화해 회원사별 영향도를 분석·통보하는 의사결정 자동화. 1차 사업 완료, 기술 세미나 진행 중.

**✓ 검증된 영역**
- **Semantic** · 기계산업 도메인 온톨로지 · Knowledge Graph
- **Semantic** · 회원사 영향도 분석 룰 설계
- **방법론** · 도메인 SME 협업 · 규제·정책 모델링

**+ AAP로 보완**
- **Inputs** · 정세·환율·규제 외부 데이터 자동 수집
- **Agentic** · 자율 추론·도구 호출 자동화
- **Action** · 회원사 통보·리포팅 Workflow
- **Trust** · 폐쇄망·감사 (공공 요건)

#### 사업 확장 로드맵 · 1개 단체 PoC에서 Public Decision Pack으로
진흥회 본 사업 → 유사 산업단체로 수평 복제 → 진흥기관·공공 정책분석 카테고리화.

| 시점 | 활동 |
|---|---|
| 현재 · FY26 1H | 1차 사업 완료, **기술 세미나 → 본 사업 RFP 대응** |
| 단기 · FY26 2H | 본 사업 수주, **회원사 통보 자동화 본 구축** |
| 중기 · FY27 | 자동차·반도체·무역·섬유 협회로 **수평 복제** |
| 장기 | 공공 진흥기관 정책분석 **Public Decision Pack** 자산화 |

**PLAY 1 · 산업단체 온톨로지 표준 템플릿화**
1차 사업의 기계산업 온톨로지를 재사용 가능한 **"산업단체 도메인 템플릿"**으로 자산화 → 신규 단체 진입 리드타임 단축

**PLAY 2 · 공공 폐쇄망 결합 카드**
KT그룹 보안·망분리·감사 자산 + 보안 파트너(디피니트 등) 결합으로 **공공 통제형 AAP 패키지** 제작 → 공공 시장 진입 카드

### 두 케이스 종합 (Note)
**두 케이스는 서로 다른 축을 입증·요구합니다.** AI 투자 메이트는 *Inputs · Agentic · Service*를, 진흥회 온톨로지는 *Semantic*을 검증했고, 각자가 비워둔 영역은 거꾸로 상대 케이스의 검증 영역과 맞물립니다. AAP는 이 둘을 통합·일반화해 산업·고객에 재적용 가능한 형태로 결합합니다.

## 06 · 산업별 적용 예시

**리드 문장**
AAP는 특정 산업 전용 솔루션이 아니라, 문서·정책·시스템·검토 흐름이 많은 업무에 우선 적용할 수 있습니다.

| 산업 | 시작 Use Case | AAP 적합 이유 |
|---|---|---|
| **제조** | 작업표준·품질·설비 매뉴얼 Agent | 문서·설비·품질 이력이 많고 현장 대응 니즈가 큼 |
| **유통** | 매장 운영·프로모션 정책 Agent | 정책 변경과 CS 대응이 잦음 |
| **물류** | 배송 예외·클레임 처리 Agent | 예외 처리와 시스템 연계가 중요 |
| **공공** | 민원·행정 지식 AI KMS | 법령·지침·근거 제시 필요 |
| **국방** | 폐쇄망 정비·운영 매뉴얼 Agent | 보안·망분리·정비 지식 활용 필요 |
| **교육** | 학사·교무·상담 AI KMS | 규정·FAQ·상담 지식 활용성이 높음 |
