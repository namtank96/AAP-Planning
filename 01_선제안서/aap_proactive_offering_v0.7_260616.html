01

Enterprise AI의 현 주소

Data Platform·LLM·RAG·Copilot 기반은 이미 확보, 전사 운영 확장에 구조적 병목 4가지 잔존

① 표준 부재

단편적 AI 도입

용어·정책·권한 기준 분산, 유사 PoC·중복 투자 반복

② 실행 연계 부족

AI 결과 ↔ 처리 흐름 단절

RAG·Copilot 결과가 ERP·CRM·결재로 미연결, 사람이 복사·재입력

③ 책임 경계 불명확

검토·승인 기준 모호

자동/수동 경계·책임 소재 미정, 자동화 확대 보류

④ 운영 학습 부족

PoC 자산화 미비

판단 근거·반려 사유·예외가 자산화 안 됨, 매번 새로 시작

02

'운영 중심' Enterprise AI로의 전환

4가지 병목은 AI 기능 추가로 미해결 — 공통 빈 영역 인식 + 패러다임 변화 수용 + 실행 구조 동시 설계 필요

AI 활용은 분석·예측, RAG, Copilot 순 확장, 판단·실행·통제·학습 영역은 매 단계 공백

2-2. Human Analytics에서 Agentic Operation으로

기존: Human Analytics(사람이 보고·판단·입력), Agent 합류 시: 사람과 같은 의미·정책·권한 해석 + 실행·검토·학습 연결 구조

두 축이 함께 진화 — 자원 구조(데이터 형태) + 수행 주체(누가 일하는가)

축 ① 자원 구조의 변화 Agent가 사람과 같은 의미·정책·권한으로 데이터를 해석할 수 있어야 함
정적 분석 데이터 Lakehouse · Warehouse · BI 리포트 Data Platform 단독
·
+ 의미·정책·권한 해석 Metadata · Ontology · Metric · Knowledge Asset + Data Intelligence 보강
·
의도·맥락 반응 동적 자원 요청 시점의 의도·권한·이력을 반영해 그때마다 구성 Agent 즉시 활용 가능
축 ② 수행 주체의 변화 AI Agent가 새로운 업무 수행자로 합류하게 됨
사람 단독 해석·판단·시스템 입력·실행 모두 사람 몫 Human Analytics
·
사람과 Agent의 의미 공유 동일한 용어·정책 위에서 Agent가 데이터를 해석 의미 기반 협업 진입
·
사람 + Agent 협업 운영 Agent: 판단 후보·실행 후보 / 사람: 검토·승인·예외 Agentic Operation

2-3. Agentic Operation의 5가지 조건

AI 자동 판단이 아니라 의미화·판단·실행·검토·학습이 하나의 운영 구조로 묶일 때 작동합니다.

업무 의미화 데이터, 문서, 정책을 업무 용어와 권한, 예외 기준으로 정리합니다.
판단 후보 생성 근거와 처리 경로, 위험도를 반영해 판단 후보를 생성합니다.
실행 연계 판단 결과를 Workflow, API, RPA, 업무 시스템 반영으로 연결합니다.
검토와 통제 권한과 위험도, 승인 기준에 따라 자동 처리와 사람 검토 경계를 설정합니다.
운영 학습 판단, 실행, 예외, 피드백을 재사용 지식과 Skill로 축적합니다.
AAP는 Agentic Operation을 반복 가능하고 통제 가능한 플랫폼 구조로 구현합니다.

3-3. Agentic AI Platform

정의 (1) 고객의 데이터·문서·시스템을 업무 맥락으로 연결하고,
(2) Agent가 업무 기준에 따라 판단 후보 생성·실행 보조·사람 검토 연계를 수행하며,
(3) 그 결과가 시스템 반영과 운영 이력으로 안전하게 이어지도록 만드는(Agentic Operation) 기업용 플랫폼
①

업무 의미화

용어, 정책, 권한, 예외 기준을 Agent가 사용할 수 있는 업무 맥락으로 정리합니다.

②

시스템 반영

Agent의 판단 후보와 실행 결과를 Workflow, API, RPA, ERP, CRM, 결재 흐름에 연결합니다.

③

실행 통제

권한, 위험도, 승인 기준에 따라 자동 처리, 사람 검토, 보류 경로를 관리합니다.

④

운영 학습

판단 근거, 실행 결과, 예외, 피드백을 정책, 지식, 재사용 Skill로 축적합니다.

4-2. AAP Reference Architecture

4-2는 4-1 Operating Loop를 가능하게 하는 8개 플랫폼 계층입니다. 아래 8계층은 손해사정 데모와 동일한 AAP 아키텍처(Version 0.3) 계층 구조를 따르며, 4-1의 6단계 Loop는 업무 실행 순서(Operating Loop View), 4-2의 8계층은 이를 구현하는 플랫폼 구조(Layer View)입니다. 각 계층은 접힘 상태에서 작동 맥락(입력→처리→출력)을, 펼침에서 상세 Capability를 보여줍니다.

[타깃 1] 금융·보험·자산운용

적용 업무

적용 방식
약관·규정·계약·거래 이력을 사건 단위로 묶고, 업무 기준에 따라 판단 근거와 처리 경로 후보를 검토자에게 제시
Pilot 후보
손해사정 심사 보조, 내부통제 검토, 투자 리서치 보조

[타깃 2] 공공·협회·산업단체

고객 상황

적용 업무

적용 방식
