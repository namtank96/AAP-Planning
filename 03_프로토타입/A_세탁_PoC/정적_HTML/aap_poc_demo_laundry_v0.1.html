02

Business Semantic · 온톨로지 (v1 시연용)

모든 Agent 추론의 공통 의미 기반입니다. 매장 직원·점주·본사가 같은 어휘로 판단할 수 있게 만드는 자산. ★ 표시는 시연 케이스에서 핵심으로 다루는 항목입니다.

Clothing

의류 카테고리

  • Coat / Down / Leather ★ 외투
  • Jacket / Suit Top
  • Shirt / Blouse / Knit
  • Bottom · Suit
  • Hanbok / Fur / Wedding ★ Special

Material

소재

  • Cotton · Polyester · Nylon
  • Wool / Cashmere ★ 시연
  • Silk / Leather / Fur ★ 위험 高
  • Blended (혼방)

Stain

얼룩 유형

  • Oil — 화장품·립스틱·기름
  • Protein — 혈액·계란·유제품
  • Pigment — 와인·커피·잉크 ★ 시연
  • Complex (복합)

Process

처리법

  • Dry Cleaning · Wet Cleaning
  • Hand Wash · Spot Treatment
  • Leather Care / Fur Care ★ 본사 전용
  • Pigment Solvent (전용 약품)

온톨로지 관계 (Relations) — Agent가 추론할 때 사용하는 5축

canBeProcessedBy소재 → 처리법 적합도
recommendedFor얼룩 → 처리법 추천
riskLevel소재 × 처리법 위험도
priceRule의류 × 처리법 가격
claimPolicy사유 × 원인 → 책임 비율
왜 온톨로지가 차별인가 — 챗봇·RAG는 *문서 검색*까지가 한계입니다. AAP는 이 온톨로지를 통해 "캐시미어 × Dry Cleaning + Wine 얼룩" 같은 조합 추론을 하고, riskLevel·claimPolicy 같은 정책 관계까지 결합해 사람 검토 가능한 후보를 만듭니다.
03

시나리오 ① 접수

Case · 캐시미어 코트 + 와인 얼룩 · 매장 직원이 접수 · 약 3분 시연

매장 접수 흐름

고객 최민수 · 의류 캐시미어 코트 · 얼룩 와인(가슴 부위) · 접수자 김서연(직원)
Persona · 매장 직원
예상 시간 · 약 3분

입력 User

사진
코트 정면 1장 · 라벨 클로즈업 1장 · 얼룩 부위 클로즈업 1장
메모
"가슴 부위 와인. 어제 저녁 묻음. 고가 의류"
고객
최민수 · 정기 회원 · 이전 처리 12건

도구 호출 Tool

vision.label_extract(photo) → 라벨 OCR
customer.fetch(id) → 회원 정보·과거 이력
vs RPA / 챗봇 RPA는 사진 1장을 받아 *문자열 기반 항목만 채움*. 챗봇은 텍스트만 처리. AAP는 사진·라벨·메모·과거 이력을 통합 입력으로 받아 다음 단계의 추론 기반을 만듭니다.

Agent 추론 Reason

classify_clothing → Coat (확신도 0.98)
identify_material(label_ocr) → 라벨 "100% Cashmere" → Cashmere (확신도 1.00)
identify_stain(stain_photo) → 색소 계열 / Wine (확신도 0.91)
ontology.lookup(material=Cashmere, stain=Wine) → 위험도 High

참조한 온톨로지

Material · Cashmere — 천연·고가·수축 위험
Stain · Wine (Pigment) — 잔류 가능성 있음
Relation · riskLevel(Cashmere × Wine) = High
vs Vision 분류기 단순 Vision 모델은 의류·얼룩을 분류하지만 거기까지. AAP는 온톨로지 관계(riskLevel)까지 끌어와 다음 단계 결정에 연결합니다.

처리법·가격·위험 후보 Candidates

처리법 · Dry Cleaning + Pigment Solvent (Wine 전용)
가격 · ₩18,000 (기본) + ₩5,000 (얼룩 가산) = ₩23,000
소요 · 4영업일 (Cashmere 케어 기준)
성공 가능성 · 약 70% (와인 색소 잔류 가능성)
완전 제거
70%
옅은 잔류
22%
변색·수축 위험
8%

도구 호출

process.recommend(cashmere, wine)
price.calc(coat, dry+pigment)
risk.score(material, process, stain)
차별 위험도 후보 분포를 산출해 *고객에게 무엇을 고지해야 하는지*까지 사람이 판단할 수 있게 합니다.

HITL · 직원 → 고객 동의 Human

분기 기준 · 위험도 High → 직원 검토 + 고객 동의 필수 (자동 처리 불가)

직원 화면 표시 ·
· "캐시미어 + 와인 = 잔류 가능성 30%. 고객에게 고지 후 동의 필요"
· 권장 멘트 예시 자동 생성 → 직원이 그대로 안내

고객 동의 · 처리 진행 / 옅은 잔류 가능성 수용 / 변색 시 책임 비율 사전 안내

HITL 산출물

· 고객 동의서 (디지털 서명)
· 사전 안내 이력 (감사 추적)
· 처리 조건 합의 (옅은 잔류 수용)
vs 일반 챗봇 챗봇은 "처리 가능합니다"로 끝남. AAP는 위험을 사전에 사람과 공유하는 단계를 강제 — 클레임 사전 예방.

시스템 반영 Action

POS 자동 입력 · 의류·소재·얼룩·처리법·가격·소요·동의 이력 모두 한 번에
처리 워크플로 배정 · 본점 Cashmere 처리실로 자동 배정
고객 SMS · 접수번호·예상 완료일·옅은 잔류 가능성 안내
본사 클레임팀 사전 인지 · 위험도 High 케이스 자동 공유

Decision Log 기록

· 모든 도구 호출·근거·확신도 저장
· 고객 동의 시점·내용 기록
· 추후 클레임 발생 시 즉시 참조
!
접수 1건이 끝난 게 아니라, 운영 자산이 1건 누적됩니다. 처리 결과·잔류 여부가 추후 *유사 케이스 추천*과 *온톨로지 보강*에 그대로 쓰입니다 — 시나리오 ④에서 확인.

시나리오 ② 클레임 ⭐ 결정적 시나리오

Case · 처리 후 코트 수축 분쟁 · 점주가 본사 클레임팀 이관 검토 · 약 5분 시연

클레임 처리 흐름 — 원인 추론 · 책임 비율 · HITL · 학습

고객 최민수 · 코트 수축 주장 · 점주 박철민 · 본사 클레임팀 이지영
Persona · 점주 + 본사
예상 시간 · 약 5분

입력 User

접수자
점주 박철민
고객
최민수 (접수번호 #20260512-031 의 처리 결과)
클레임
"코트가 한 사이즈 줄었다. 입을 수 없다"
증빙
처리 전후 사진 · 고객 측 측정값 (어깨 너비 -2cm)
기존 점주가 *수기로* 처리 이력·약관·유사 사례를 뒤져야 함. 평균 30분 ~ 2시간. AAP는 다음 단계에서 자동 로드.

이력 자동 로드 Tool

case.load(receipt_id) → 접수·처리 전체 이력 1번에 로드

접수 시점 · 캐시미어 + 와인 / 위험도 High / Pigment Solvent 처리
고객 동의 · "옅은 잔류 / 변색 시 책임 비율 사전 안내" 동의 완료 (디지털 서명)
처리 기록 · 본점 Cashmere 처리실 김선임 / 처리 온도 30℃ / 약품 사용량 정상 범위

핵심 발견

· 고객은 *수축*이 아닌 *잔류·변색* 만 동의함
· 처리 기록은 정상 범위 내
· 측정값 -2cm는 별도 검증 필요

원인 추론 · 4개 후보 + 확률 Reason

claim.diagnose(case, evidence)
ontology.lookup(claimPolicy, material=Cashmere, symptom=수축)

① 측정 오차 (고객 측 측정 방법 차이)
30%
② 고객 사용 중 자연 수축 (캐시미어 특성)
30%
③ 처리 중 오류 (약품·온도)
25%
④ 소재 결함 (제조사 책임)
15%

추론 근거

· 처리 기록상 약품·온도 정상 → ③ 확률 낮춤
· Cashmere는 자연 수축 -1~3cm 흔함 → ② 확률 높임
· 측정 방법 미검증 → ① 확률 높임
vs RPA · 챗봇 RPA·챗봇은 절대 불가능. 다중 후보 + 확률 + 근거를 한 화면에 제시하는 게 AAP의 핵심 차별.

유사 사례 검색 + 책임 비율 후보 Reason

case.similar(material=Cashmere, symptom=수축, last=12M) → 본사 케이스 5건

유사 사례 1 · 강남점 2025-08 · 측정 오차 → 매장 책임 0%
유사 사례 2 · 송파점 2025-11 · 처리 오류 일부 → 매장 20% / 본사 10%
유사 사례 3 · 강남점 2026-02 · 자연 수축 → 매장 0% / 본사 0%

본 건 권장 책임 비율 후보
매장 책임
30%
본사 책임
20%
고객 / 자연 수축
50%

권장 액션

① 매장에서 정밀 측정 (어깨·소매·기장 3축)
② 본사 클레임팀 검토 이관
③ 측정 결과에 따라 책임 비율 확정

HITL · 본사 클레임팀 최종 판단 Human

분기 기준 · 책임 비율 ≥ 10% → 본사 클레임팀 필수 검토 (Agent 자동 확정 불가)

본사 클레임팀 이지영의 결정 ·
· 정밀 측정 결과 검토 → 실제 -1.2cm (자연 수축 범위)
· Agent 후보 ② 자연 수축 채택
· 최종 책임 비율 · 매장 0% / 본사 0% / 고객 50% (자연 수축)
· 고객 보상 · 50% 무료 재처리 (선의 보상)
· 결정 사유 · 감사 추적용 기록

HITL 산출물 · 감사 추적

· 최종 결정 + 사유
· Agent 후보 vs 본사 결정 차이
· 향후 유사 사례 학습 입력
차별 Agent가 단정하지 않고 후보 + 근거를 제시, 사람이 책임 있는 최종 결정을 내림. 감사 가능한 형태로 저장.

학습 환류 Learning

Decision Log 축적 · 본 건의 모든 후보·근거·최종 결정·사유
유사 패턴 강화 · "Cashmere + 수축 + 정밀측정 -1~2cm = 자연 수축 우세" 패턴 가중치 +
온톨로지 보강 · Cashmere 정상 수축 범위 갱신 후보로 등록
매장 점주 멘트 템플릿 · "캐시미어는 -1~2cm 자연 수축 가능" 사전 안내 가이드 자동 생성

다음 케이스에서 효과

· 다음 Cashmere 접수 시 사전 안내 자동 제시
· 유사 클레임 시 즉시 본 건 인용
· 본사 운영기획팀의 정책 개정 후보로 자동 등록
★
RPA·챗봇·Copilot으로는 절대 도달할 수 없는 영역 — 원인 추론 + 책임 비율 + 유사 사례 + HITL + 감사 + 학습 환류가 한 흐름으로 닫힙니다. 이게 AAP입니다.

시나리오 ③ 신규 점주 교육

Case · 신규 점주가 처음 받은 *모피 코트* 처리 문의 · 약 2분 시연

신규 점주 자문 · Knowledge Agent

신규 점주 · 모피 코트 첫 접수 · 어떻게 처리해야 할지 불확실
Persona · 신규 점주
예상 시간 · 약 2분

점주 질문 User

"고객이 천연 모피 코트를 맡기려고 합니다. 우리 매장에서 처리 가능한가요? 가격·기간·주의사항 알려주세요."
기존 매뉴얼 PDF 검색 → 본사 콜 → 대기 → 안내. 신규 점주는 자신 없음·시간 손실. OJT 수개월.

시연 클로징에서 한 화면으로 요약. AAP는 챗봇이 아니다를 가장 명확하게 보여주는 슬라이드.

시나리오 기존 POS RPA 개별 챗봇 / RAG AAP
① 접수캐시미어 + 와인 직원 수기 가격 계산만 의류 분류 추천만 분류 + 위험도 + HITL + 사전 안내 + 동의 + Log
② 클레임 ⭐수축 분쟁 점주 판단 ✕ 불가능 FAQ 검색 원인 추론 + 유사 사례 + 책임 비율 + HITL + 학습
③ 신규 점주 교육모피 처리 ✕ 수개월 OJT ✕ 불가능 매뉴얼 검색 온톨로지 추론 + 실제 사례 + 멘트 + 학습 추적
④ 본사 운영 학습온톨로지 갱신 ✕ 매뉴얼 개정 ✕ 불가능 ✕ 불가능 패턴 감지 + 본사 결재 + 전국 즉시 반영
핵심 메시지 · ②와 ④는 RPA·챗봇 어떤 조합으로도 불가능합니다. AAP의 차별은 *기능*이 아니라 운영 구조(Loop)이며, 이 구조가 작동하려면 Business Semantic·HITL·Operational Learning이 한 묶음으로 들어가야 합니다.
