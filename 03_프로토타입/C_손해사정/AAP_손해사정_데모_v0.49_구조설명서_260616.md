## 0. 한 장 요약

- **하나의 사건**(CL-2026-0417)을 **하나의 5단계 손해사정 프로세스**로 처리하는 과정을, **3개 탭**이 서로 다른 청중·깊이로 보여준다.
- 메시지는 *"AI 자동화"*가 아니라 **"책임 있는 분기 구조"** — 확실한 건 자동, 다툼·의심은 사람 검토로 **가른다**.
- 3탭은 **같은 케이스·같은 5단계·같은 색 의미체계**를 공유해, 시점만 다르고 내용은 한 사건을 추적한다.

|---|---|---|---|
| ① | 처리 방식별 Workflow | 직책자·의사결정자 | 기존 도구(챗봇·RAG·단일Agent)는 어디서 멈추고, AAP는 어떻게 가르는가 |
| ② | AAP 구조 지도 | 실무 리더·IT·AI 검토자 | 이 사건이 8계층·6단계 Loop의 **어떤 구성요소**를 지나는가 |
| ③ | AAP 작동 시뮬레이션 | SME·손해사정사 | 실제 업무 콘솔에서 단계별로 판단하고, AAP 작동을 순차 실행으로 확인 |

## 2. 탭① — 처리 방식별 Workflow

### 2-1. 목적·청중
- **청중**: 직책자·의사결정자.
- **목적**: "기존 방식 대비 AAP가 왜 다른가"를 **단계별 장면**으로 보여준다. 표가 아니라 흐름 도식으로 "단일은 덮고, AAP는 가른다"를 목격시킴.

### 2-2. 화면 구성(위→아래·좌→우 순서)
1. **색상 범례 바** — 5색 의미.
2. **기준선 밴드(`t1Base`)** — *이 단계의 실제 수기 처리* = 출발점.
   - 수기 처리 단계 칩(예: STEP1 = 신분증·계약 대조 → 운전자 한정 확인 → 청구이력 수기 조회 → 담당자 판단)
   - **「수기의 한계」** 한 줄(`MANUAL_PAIN`): 예) "본인·운전자·사기를 사람이 각각 조회 — 의심 신호를 놓치면 부당지급 위험."
   - 라벨에 *"비교 대상이 아니라 출발점"* 명시 → 수기를 폄하하지 않고 기준점으로 둠.
3. **VS 2분할**
   - **좌(`t1Tools`) = 도구 3종**: 챗봇 / RAG / 단일 Agent. 각각 **「돕는 곳」 / 「멈추는 곳」** 2줄(`METHOD_CMP`).
     - 예(STEP4): 단일 Agent — "과실비율 하나를 제시하지만, 20:80과 30:70처럼 갈릴 때 차이를 검증 대상으로 만들기 어렵다."
   - **우(`t1Aap`) = AAP**: "같은 업무를 구조로 바꾼다" — **단계마다 다른 흐름 도식**.

### 2-3. 단계별 AAP 도식 형상(`t1Aap(i)`) — 흐름이 단계마다 달라짐

| STEP | 도식 형상 | 분기 |
|---|---|---|
| 1 지급심사 | **3개 병렬 검증**(본인·운전자·사기) → 자격 판정 허브 | PASS / REVIEW / DENY |
| 2 견적산정 | **순차 산출**(사진분석 → 부품·공임 → 규칙적용) → 견적 판정 | 확정 / 전손·과다 검토 |
| 3 대인피해 | **2판정 합류**(상해등급 + 과잉진료) → 합의 판정 | 확정 / 적정성 검토 |
| 4 과실산정 | **A2A 비교**(블랙박스 과실 ⇄ 판례 과실) → 판정 비교 허브(차이 10%p) | 일치 확정 / 불일치 사람 검토 |
| 5 처리방법 | **시나리오 분기**(4안 생성 → 손해사정사 선택) | 자동 90만 / 검토 171만 |

> 하단에 **「이 단계에서 AAP가 처리하는 것·해석」**(`STEP_BASIS.plain` + 근거 3줄)이 붙어 도식을 업무 언어로 풀어준다.

### 2-4. 흐름·작동원리
- 상단 **STEP 네비/재생**으로 단계 전환 → `renderWorkflow(i)`가 기준선·도구·AAP 3영역을 동시 갱신.
- AAP 도식은 SVG(`buildDiagram`): 노드(`agent`/`hub`/`auto`/`review`/`risk`/`mint`) + 엣지(teal=주경로 / amber=검토분기 / faint=비선택) + 라벨.

## 3. 탭② — AAP 구조 지도

### 3-1. 목적·청중
- **청중**: 실무 리더·IT·AI 검토자.
- **목적**: 이 사건이 **AAP 아키텍처(캐논 Version 0.3)의 어떤 구성요소**를 어떤 순서로 지나는지 보여준다. 8계층은 항상 노출하되, **이 STEP이 지나는 셀만 강조·나머지 dim**.

### 3-2. 화면 구성(포스터 = 위→아래)
1. **AAP 추론 루프 바(`ARCH_LOOP`)** — 6단계, 단계별 관여 계층 태그:

   | Loop 단계 | 관여 계층 |
   |---|---|
   | Data | L5 |
   | Semantic | L4 |
   | Reasoning | **L3 · L6** |
   | Decision | **L7** |
   | Action | L1 · L8 |
   | Learning | L7 |

   > (v0.49 정합 수정: 셀 점등 로직 `ARCH_CAP2LOOP`와 바 표기를 일치시킴 — L6 평가는 추론 보조이므로 Reasoning, 거버넌스 L7은 Decision.)
2. **Customer Environment 밴드**(`ARCH_CUST`) — Data Platform · Business Systems · Knowledge Assets. *"고객 자산 · 대체 아님 · 입력 기반"*.
3. **8계층 스택(`ARCH_LAYERS`)** — 아래 3-3.
4. **Trust · Security 밴드**(`ARCH_TRUST`) — IAM·SSO·RBAC / 감사·근거·이력 / PII·DLP·Prompt Injection / 폐쇄망·On-prem. *"전 영역 횡단 통제"*.

### 3-3. 8계층 구조(`ARCH_LAYERS`)

| 계층 | 이름 | 강조 | 대표 구성요소(셀) |
|---|---|---|---|
| L1 | 경험 / 접근 | | 에이전트 포털 · 코파일럿/챗 UI · 관리자 콘솔 · API Gateway/SDK |
| L2 | 설계 / 개발 | | 에이전트 설계 · 프롬프트·워크플로우 설계 · Tool Registry · **Agent Registry**(자사 검증 Agent) |
| **L3** | **에이전트 실행환경** | **★ 핵심(core)** | 접근 · 코어·실행엔진 · **오케스트레이션·멀티에이전트(A2A)** · **HITL 런타임 게이트** |
| **L4** | **지식 / 시멘틱** | **★ ds 차별화** | 온톨로지·시멘틱 모델 · 도메인 지식그래프 · 하이브리드 지식스토어 · 컨텍스트 조합·근거 |
| L5 | 데이터 / 연동 | | 연결·수집(**AI:ON-U OCR**) · 데이터 처리·파이프라인 · 데이터 관리 |
| L6 | AI 품질 / 평가 | | 실행추적(Decision Log) · 품질·근거 평가(**Antbot**) · RAI·안정성 · 성능·비용(**BEAST**) · 사용자 피드백 |
| **L7** | **컨트롤 / 거버넌스** | **★ ds 차별화** | 정책 관리·통제 · 에이전트 권한 · HITL 승인 기준 · **Self-Improving(운영학습)** · 사용량·비용 가드레일 |
| L8 | 인프라 | | 컴퓨팅·GPU · 컨테이너·K8s · 네트워크·보안 · 스토리지·DB · 온프레/클라우드 |

### 3-5. 흐름·작동원리
- STEP 전환 시 `updateArchStep(i)`: ① 모든 셀 dim → ② 경유 셀만 **순서대로 점등**(번호 배지) → ③ 해당 Loop 단계·고객자산·Trust 항목 동반 점등.
- **셀 클릭 = 업무 의미 4행(`ARCH_MEAN`)**: 업무 개체 / 연결 관계 / AAP 역할 / 결과 영향(색).
  - 예) L7 정책: 업무개체 "자동/검토 분기·보류·SIU" / 연결 "정책·임계값·권한 기준" / 역할 "자동/검토/특별조사 경로 분기(ds 차별화)" / 결과영향 "처리 경로 결정(분기)".
- 우측 패널(`archPanel`)에 경유 셀 리스트(`seq-flow`) + **「이 단계를 어떻게 읽나·해석」**(`STEP_BASIS.interp` + 근거).

## 4. 탭③ — AAP 작동 시뮬레이션

### 4-1. 목적·청중
- **청중**: SME·손해사정사.
- **목적**: 실제 제품 콘솔에서 **단계별로 직접 판단**하면서, 우측에서 그 판단을 만든 **AAP 작동을 순차 실행**으로 확인 → "작동의 실제 증거".

### 4-2. 좌측 = 손해사정사 업무 콘솔(`renderPC`)
- **다크 상단바**: AI 손해사정 시스템 · 포털 경로 · *김현수 손해사정사*.
- **좌 네비**: 5단계(완료 시 ✓ 배지) + 사건 요약 카드.
- **본문(단계별 카드)**: 검증/산출 카드 — 각 카드에 **verdict 배지**(✅ PASS / ⏸ REVIEW / 확인완료), **규칙 태그(R##)**, **위험 플래그(⚠)**.
- **판단 영역(`decision`)**: 질문 + **직접 판단 버튼** → 선택 시 **결과 문구(`outcome`)** 표시.
- **AAP 한 줄**: 그 단계에서 AAP가 한 일(예: STEP1 "차등 HITL — 정상 자격은 자동 PASS, 사기 의심만 사람 검토·SIU로 분리").

#### 단계별 콘솔 내용(요지)

| STEP | 검증/산출 카드 | 판단 버튼 | AAP 역할 |
|---|---|---|---|
| 1 | 본인 확인(PASS) · 운전자 적합성(PASS) · **보험사기 패턴(REVIEW, ⚠78점·6개월 2회 [R20])** | PASS / REVIEW / **SIU 의뢰** | 차등 HITL |
| 2 | 차량 특정 · 파손부위(AI 92% · 연쇄손상 [R29]·ADAS) · 견적([R03] 인증부품 −25%, [R02] 전손 없음, 480만) | 견적 확정 / 미수선 현금정산 | 의미화(L4) |
| 3 | 피해자/상해(14급·14일) · **과잉진료(통상10일 대비 +40%)** · **AI 합의(벤치 220만 vs 요구 300만)** | 합의 확정 / 적정성 검토 | 검증(L6) |
| 4 | 증거(경찰>블랙박스>CCTV>목격) · **AI 과실(블박20:80 ⇄ 판례30:70, 불일치 10%p)** · 협상 3단계 | 과실 확정 / 사람 검토 | 충돌 표면화(A2A) |
| 5 | **4개 시나리오**(①최적 미수선현금정산★ ②공격 재협상+구상권 ③안전 제휴센터 ④전략 특수경로) · 결재라인 | 시나리오 선택 | 자동/검토 분리 |

> 결재 라인(STEP5): ~100만 팀장 전결 / ~500만 센터장 / ~2,000만 본부장+심사팀.

### 4-3. 우측 = AAP 작동 순서(`aapxSkeleton` / `runAapxAnim`)
- `STEP_TRACE`의 경유 셀을 **순차 실행 애니메이션**(스피너 → 완료 ✓ + 진행바)으로 재생 → 좌측 결과가 "어떻게 만들어졌는지".
- 완료 시 **「이 작동이 만든 결과·해석」**(`STEP_BASIS`): 좌측 화면 결과 + "쉽게 말하면" + 근거 3줄.
  - 예(STEP1): "본인·운전자·사기 확인을 한 번에 돌려, 정상 청구는 통과시키고 사기 의심만 따로 뺀다 / 78점이 기준선 75점 초과 / 거절이 아니라 SIU로 사실 먼저 확인."

### 4-4. 흐름·작동원리
- 좌측 콘솔과 우측 작동 패널은 **같은 `STEP_TRACE`를 공유** → 화면 결과와 아키텍처 경로가 1:1.
- 탭 진입·STEP 전환·재생 시 자동 실행, 「다시 실행」 버튼 제공.

## 5. AAP 기획안 개념과의 매핑

데모는 AAP **4대 역량**을 단계 전반에 분산 시연한다.

| 4대 역량 | 데모에서 보이는 곳 |
|---|---|
| ① 업무 의미화 | L4 온톨로지·지식그래프 — 대물/대인 분리, 담보 매핑, KNIA·판례 연결 |
| ② 시스템 반영 | L5 수집·정규화(AI:ON-U OCR) · L1 산출물 표시 · L8 지급·결재·로그 저장 |
| ③ 통제·권한 | L7 정책·권한·HITL 승인·가드레일 · Trust 스파인(IAM/RBAC/감사/폐쇄망) |
| ④ 운영 학습 | L7 Self-Improving — 검토 결과·SME 피드백을 기준 개선으로 자산화(STEP5) |

핵심 차별화 장치: **차등 HITL**(STEP1) · **A2A 충돌 표면화**(STEP4) · **자동/검토 분리**(STEP5) · **도구 3종 대비**(탭1).

## 6. 정합성 자체 점검 & 외부 검토 요청 포인트

### 6-1. 손해사정 도메인 (검토자: 손해사정 SME)
- ✅ 참고 실제 시스템 5단계 워크플로우와 거의 1:1(검증 4단계·FDS 23종·R03/R02/R29·KNIA 401유형·증거 우선순위·14등급·결재라인·미수선 70~80%).
- ❓ **검토 요청**:
  1. STEP4 **"블랙박스 과실 vs 판례 과실"를 별도 Agent로 나란히 비교**하는 구성은 데모 고유 장치(원본은 KNIA 단일판정 + 수정요소). 실무적으로 타당한 프레이밍인가? 과실 산정에서 "기준 충돌"을 이렇게 표면화하는 게 자연스러운가?
  2. 임계·수치 가정값의 합리성: 사기 SIU 75점, 과실 자동확정 임계 10%p, 대인 통상 치료기간 10일·+40% 편차, 자동 90만/검토 171만의 산식.
  3. STEP 순서(견적→대인→과실)가 실무 감각에 맞는지(과실상계가 지급액에 미치는 영향 고려).

### 6-2. AAP 아키텍처 (검토자: AI 아키텍트)
- ✅ 8계층·6단계 Loop·Customer Env·Trust = 캐논 Version 0.3 / 선제안서 v0.7과 동일 구조. Loop↔Layer 매핑을 데모·선제안서 간 통일(v0.49).
- ❓ **검토 요청**:
  1. Loop↔Layer 매핑(특히 **L6를 Reasoning에 둔 점**, L8을 Action에 둔 점, L2·L8을 특정 단계에 고정하지 않은 점)이 캐논 의도와 맞는가?
  2. L2 설계/개발 계층은 런타임 케이스 경로엔 등장하지 않음(설계타임) — 포스터에 항상 노출하는 게 적절한가?

### 6-3. 논리 구조·형상화 (검토자: UX·커뮤니케이션)
- ✅ 청중 3계층 × 단일 케이스 × 5단계 스파인 · 색 의미체계 일관 · STEP 반응 점등 · SVG 도식 엔진.
- ❓ **검토 요청**:
  1. 탭1(최상위 청중인 의사결정자)이 3분할(기준선+도구3종+AAP)로 정보량이 가장 많음 — 청중-깊이 매칭이 적절한가?
  2. 탭2 포스터 밀도(8계층 × 4~5셀 ≈ 35셀)의 첫인상 부담 — STEP dim 외 추가 완화가 필요한가?
