## 0. 한 줄 정의

자동차 사고 손해사정이라는 **누구나 이해하는 분쟁 사례**를 통해, ① 기존 방식(단일 LLM·단일 에이전트·수기)의 한계와 ② AAP의 작동 구조(데이터→온톨로지→추론→결정→액션→학습)를 **같은 케이스로 나란히 대비**시켜, AAP가 "도메인 솔루션"이 아니라 **도메인을 갈아끼우는 운영 지능 플랫폼**임을 증명한다.

> 세탁의 **책임비율 산정** = 손해사정의 **과실비율·손해액 산정**. 구조가 동형이라 엔진은 그대로 두고 어휘만 바꾼다.

## 1. 배경 — 왜 손해사정인가, 두 PoC와 무엇이 다른가

| 구분 | 세탁 PoC | 내부통제 PoC | **손해사정 (본 데모)** |
|---|---|---|---|
| 핵심 판단 | 클레임 책임비율 | 통제 위반 가능성 | **과실비율 + 손해액 + 면책** |
| 성격 | 교보재(이해 쉬움) | 규제 무게감 | **둘 다 — 분쟁·금액·약관이 실재** |
| 신뢰 쟁점 | "왜 이 책임비율?" | "왜 위반?" | **"왜 이 과실/이 금액?"(소송·민원 직결)** |

손해사정은 **금액이 즉시 분쟁이 되고**(가입자 민원·소송), **약관·법령·기준표가 두껍고**(과실비율 인정기준, 약관 담보·면책), **사기 리스크**(나이롱·공모사고)까지 얹혀 있어, "AI가 왜 그렇게 판단했는지"의 **재현성·추적성**이 가장 절실한 도메인이다. → 두 PoC가 쌓은 결정론 엔진·오케스트레이션을 가장 설득력 있게 보여줄 무대.

**이 데모의 새 강조점:** 직전 작동구조 데모는 노드마다 *[기존 문제]→[AAP 해결]* 칩만 붙였다. 본 데모는 한 발 더 나아가 **"기존 방식 3종 vs AAP"를 같은 케이스로 실제 돌려 보여주는 모드 토글**을 전면에 세운다.

## 2. 센터피스 — 기존 방식 3종 vs AAP (같은 사고 1건)

같은 입력(아래 5장 CL-2026-0417)을 4가지 방식에 넣었을 때의 차이.

| 차원 | ① 단일 LLM 챗봇 / RAG | ② 단일 범용 에이전트 | ③ 현행 수기 손해사정 | ★ **AAP (멀티 에이전트)** |
|---|---|---|---|---|
| **처리 방식** | 약관+서류를 통째로 프롬프트에 넣고 한 번에 답 | 도구 몇 개 쥔 1개 에이전트가 순차 호출 | 사정사가 기준표·약관 수기 대조·계산 | **전문 에이전트들이 오케스트레이션되어 분업** |
| **과실비율 근거** | "아마 30:70" — 출처·계산식 불명 | 단일 추론 경로, 가정 추적 어려움 | 사정사 경험+기준표, 사람마다 편차 | **사고유형↔과실인정기준 결정론 매핑 + 근거 트레이스** |
| **손해액 계산** | 환각 위험(금액 지어냄) | 항목 누락 시 그대로 통과 | 수기 계산 누락·오류 가능 | **항목별(수리·대차·위자료·휴업) 결정론 산식** |
| **사기·이상 탐지** | 거의 못 잡음 | 별도로 안 함 | 사정사 감(感)에 의존 | **전담 에이전트 + 이상패턴(반복·과잉) 플래그** |
| **재현성** | ✗ 매번 답이 달라짐 | △ 프롬프트 따라 흔들림 | △ 사정사마다 다름 | **✓ 동일입력=동일결과, 분석ID 부여** |
| **HITL·책임소재** | 누가 책임지는지 불명확 | 불명확 | 전 과정을 사람이 — **병목** | **임계 초과 건만 사정사 결정, AI는 증강·제안** |
| **감사·추적** | ✗ 로그 없음 | △ 부분 | 수기 기록 | **전건 감사로그 + 단계별 근거 보존** |
| **확장(타 보종)** | 프롬프트 한계 | 에이전트 1개 한계 | 인력에 비례 | **어휘만 교체 → 화재·배상·실손으로 복제** |

> **고객 메시지:** "RAG+LLM 하나로는 *그럴듯한 답*은 나와도 *재현·추적·책임*이 안 됩니다. 손해사정은 그게 곧 분쟁이고요. AAP는 전문 에이전트가 분업하고 사람이 통제하는 **운영 구조**입니다."

### 왜 단일 에이전트가 아니라 멀티 에이전트인가 (3줄)
- **분업·검증:** 과실/손해액/약관/사기를 각 전문 에이전트가 독립 산출 → 서로 교차 검증(Agent↔Agent).
- **추적·책임:** 단계마다 "무엇을 조회→무엇을 찾음→어떤 규칙 적용"이 남아 분쟁 시 방어 가능.
- **통제·확장:** 임계 초과만 사람에게(HITL), 도메인은 어휘만 갈아끼우면 복제.

## 3. AAP 작동 구조 (캐논 6단계) — 손해사정 인스턴스

같은 6노드 Execution Loop에 자동차 손해사정 케이스를 흘린다. **노드마다 [기존 문제]→[AAP 해결]**.

```
 ① 데이터 수집      ② 온톨로지            ③ 에이전트 추론       ④ HITL 결정       ⑤ 액션          ⑥ 학습 루프
 ──────────────   ──────────────────   ──────────────────  ─────────────    ──────────      ──────────────
 사고접수·블랙박스   사고유형·당사자·차량    과실/손해액/약관/사기  손해사정사 검토     지급결의·통지     사정기준·과실DB
 진단서·수리견적  →  손해항목·약관조항   →  4개 전문에이전트   →  ·확정/수정/반려  →  구상권·감사로그 →  ·사기패턴 환류
 경찰확인원·약관     면책사유 ⟶ 관계그래프   결정론+유사사례+LLM   (임계 초과 시)     단계별 근거첨부    전 지점 즉시반영
 ──────────────   ──────────────────   ──────────────────  ─────────────    ──────────      ──────────────
 [문제] 서류가      [문제] 약관·기준표가    [문제] 산정 근거가    [문제] 전 건을     [문제] 수기지급·  [문제] 사정사
  채널마다 흩어져     사정사 머릿속 암묵지     불명확·매번 달라      사람이 처리—병목    통지 누락·편차    개인 노하우가
  취합에 시간소모     로만 존재               민원·소송에 취약       기준 들쭉날쭉                      조직에 안 쌓임
 [해결] 한 곳        [해결] 기준·약관·사례   [해결] 결정론 산출    [해결] 근거 자동   [해결] 결의·통지  [해결] 이번 사정이
  자동 집결·정규화    를 구조화된 지식자산化    + "왜?" 추적·재현     첨부 + 임계기반     자동화·전건       즉시 기준·과실DB
                                          + 항목별 산식         이관(저위험 자동)   감사로그           ·사기룰에 반영
```

| 단계 | 하는 일 | 신뢰 장치 |
|---|---|---|
| ① 데이터 | 사고접수·블랙박스·진단서·견적서·경찰확인원·약관 통합·정규화 | 출처 보존 |
| ② 온톨로지 | 사고유형·당사자·차량·손해항목·약관조항·면책사유 관계 모델링(지식그래프) | 의미 일관성 |
| ③ 추론 | 과실비율·손해액 **결정론 규칙엔진** + 유사사례 + LLM 설명문 | **재현성·추적성** |
| ④ 결정 | 임계 초과 시 손해사정사 확정/수정/반려 (HITL) | **인간 권한 유지** |
| ⑤ 액션 | 지급결의·고객통지·구상권 라우팅·감사로그 | 감사 추적 |
| ⑥ 학습 | 사정결과→기준·과실DB·사기룰 환류, 조직 전파 | 거버넌스 승인 |

**관통 원칙:** 좌측 = 고객 시스템 연동 / 우측 = Trust·Security·Governance / 중심 = **손해사정사 의사결정**.

## 4. 전문 에이전트 구성 (Agent OS / Marketplace 매핑)

내부통제 5개 에이전트 구조를 손해사정으로 치환. Planner가 작업 분해 → Orchestrator가 배분 → 결과를 위험·금액 종합 에이전트가 취합 → HITL.

| 에이전트 | 하는 일 | 주 입력 |
|---|---|---|
| **과실비율 산정** | 사고유형 ↔ 과실비율 인정기준 매핑, 가감요소(신호위반·과속·선진입) 적용 | 사고경위·블랙박스·경찰확인원 |
| **손해액 산정** | 대물(수리·대차·격락·전손판단) / 대인(치료비·위자료·휴업·상실수익) 항목별 산식 | 수리견적·진단서·소득자료 |
| **약관·보장 판정** | 담보 적용·면책사유(음주/무면허/고의)·보상한도·자기부담금 | 약관·계약사항 |
| **사기·이상 탐지** | 경미사고 대비 과다치료·반복청구·공모 패턴 플래그 | 청구이력·치료패턴 |
| **유사사례 검색** | 동형 사고의 과거 사정결과(과실·금액) 근거 제시 | 사정 사례 DB |

> ③ 노드 줌인 = 세탁 v0.7 추론 위젯의 "왜? 드릴다운 + What-if + 재현성"이 **과실·손해액 산정 위젯**으로 그대로 편입된다 (버려지지 않음).

## 5. 시연 케이스 — CL-2026-0417 (고정)

> **신호교차로 직진 vs 좌회전 충돌, 대물 + 경상 대인.** (시뮬레이션 값 — 화면에 "Simulation PoC" 명시)

| 항목 | 값 |
|---|---|
| 사고 | 신호기 있는 교차로, A차량(직진) ↔ B차량(좌회전) 충돌 |
| 대물 | B차량 범퍼·휀더 파손, 수리견적 480만원 / 차량가액 1,200만원 (전손 아님) |
| 대인 | B차량 운전자 경추염좌 진단 2주, 치료비 청구 + 합의금 요구 |
| 기본 과실 | 직진 30 : 좌회전 70 (과실비율 인정기준) |
| 자기부담금/면책 | 음주·무면허·고의 없음 → 담보 적용 |

**What-if (라이브 재계산 — 세탁의 -2cm→4cm 토글과 동형):**
- 블랙박스로 B차량 **신호위반(황→적 진입)** 확인 → 과실 **10 : 90** 전환
- A차량 **과속 정황** 가산 → A 과실 +10
- 대인 경추염좌 + **과거 6개월 내 유사청구 2건** → 사기·과잉진료 **플래그 점등**

→ 기본은 "쌍방 과실·통상 지급" 결론, What-if 조작 시 과실·금액·사기플래그가 **결정론적으로 재계산**되며 근거 트레이스가 함께 갱신됨.

## 6. 화면 설계 — 모드 토글 + 줌아웃 ↔ 줌인

핵심 인터랙션 셋: **(A) 모드 토글**로 같은 케이스를 기존방식↔AAP로 대비, **(B) 줌아웃 캔버스**로 전체 흐름, **(C) 노드 줌인**으로 안으로.

```
┌──────────────────────────────────────────────────────────────────────────┐
│ [상단] 모드 토글:  ⟨ 단일 LLM ⟩ ⟨ 단일 에이전트 ⟩ ⟨ 수기 ⟩  ←→  ★ AAP        │
│   · 좌측 모드 선택 → "한 덩어리 답 + 근거 없음/재현 안 됨" 결과 카드          │
│   · ★AAP 선택 → 아래 캔버스가 살아 움직임                                    │
├──────────────────────────────────────────────────────────────────────────┤
│ [Execution Loop 바] ①②③④⑤⑥ 가 좌→우로 빛나며 케이스가 흐름                  │
├──────────────────────────────────────────────────────────────────────────┤
│ [줌아웃 캔버스 — single pane of glass]                                       │
│   ①──→②──→③──→④──→⑤──→⑥─┐   ← ⑥에서 ②로 환류(루프)                       │
│   데이터 온톨로지 추론 결정 액션 학습│                                        │
│   · 각 노드 [문제→해결] 토글 칩                                              │
│   · "사고건 흘려보내기" → 점이 노드를 타고 이동(작동 애니메이션)              │
│   · ③ 추론에서 4개 전문에이전트가 동시 점등 → 종합(Agent↔Agent)              │
├──────────────────────────────────────────────────────────────────────────┤
│ [줌인 패널] 노드 클릭 시                                                     │
│   ② → 지식그래프(사고유형·당사자·손해항목·약관·면책 관계, Cytoscape)         │
│   ③ → 과실·손해액 산정 위젯 (왜?드릴다운 + What-if + 재현성ID) ← v0.7 편입    │
│   ④ → 손해사정사 결정 화면 (근거 자동첨부 + 임계 이관 + 확정/수정/반려)       │
└──────────────────────────────────────────────────────────────────────────┘
```

**설계 원칙**
- **대비가 핵심** — 모드 토글로 "단일 방식의 빈약한 답" vs "AAP의 흐르는 구조·근거"를 즉시 체감.
- **Single pane of glass** — 전체 흐름·이상·액션이 한 화면(RaceOS).
- **사람이 중심** — ④ HITL을 시각적으로 강조, "AI는 증강·제안, 확정은 사정사".
- **줌인은 기존 자산 재활용** — ③에 세탁 v0.7, ②에 지식그래프.
- 직관 우선 — 기술 용어 텍스트 최소화, 단 오케스트레이션 구조 자체는 제대로 담는다.
