## 0. 이 문서가 답하는 세 가지 질문

기획안 v0.1은 **"같은 케이스를 기존 3종 vs AAP로 대비한다"**는 *연출*까지 정했다. 이 문서는 그 안쪽,
**"AAP가 실제로 어떻게 돌아서 그 결과가 나오는가, 그리고 이미 만든 데모를 거기까지 어떻게 끌어올리는가"**를 설계한다.

1. **우리가 이미 만든 데모는 4분면 중 어디 있나?** → 정직한 자기 진단 (1장)
2. **거기서 AAP(멀티)로 가면 무엇이 실제로 바뀌나?** → 작동 메커니즘 7개 델타 (2~4장)
3. **그 작동을 어떻게 보여주나?** → "결과 대비"에서 "작동 대비"로 (6장)

> 핵심 통찰 한 줄: **"기존 3종 vs AAP"의 ②(단일 에이전트)가 바로 우리가 이미 만든 것이다.**
> 따라서 이 비교는 경쟁사를 깎는 추상 논증이 아니라, **우리 데모의 다음 버전 로드맵**이다 — 훨씬 정직하고 설득력 있다.

## 1. 현 위치 진단 — 우리 데모는 "②에 가깝다"

dmp-portal 데모는 이미 ① 단일 LLM/RAG는 한참 넘어섰다. 그러나 ★ AAP에는 아직 못 미친다.
**가진 것**과 **아직 없는 것**을 정직하게 가른다.

| 축 | ① 단일 LLM/RAG | **② 단일 에이전트 ＝ 현재 우리 데모** | ★ AAP (멀티 에이전트) |
|---|---|---|---|
| 처리 형태 | 프롬프트 한 방 | **고정 5단계 파이프라인**(지급심사→견적→대인→과실→처리) | 케이스별 동적 작업분해·분업 |
| 판단 근거 | 출처·식 불명 | **온톨로지 + 49규칙 결정론**(재현 O) | 결정론 + **독립 에이전트 교차검증** |
| 자기모순 | 못 잡음 | **못 드러냄**(한 체인이라 모순이 숨음) | **충돌을 표면화** + 종합/사람 조정 |
| 도구·권한 | 전권 | **사실상 전권**(거버넌스 구멍) | **에이전트별 도구·자격 스코핑** |
| 사람 검토 | 책임 불명 | **전 단계 일괄 또는 끝 한 번** | **위험·금액별 차등 이관(HITL 라우팅)** |
| 추적 | 로그 없음 | **선형 트레이스** | **DAG 트레이스**(누가·무엇근거·어떻게 조정) |
| 보종 확장 | 프롬프트 한계 | **파이프라인 재코딩** | **에이전트 재조합**(어휘만 교체) |

**우리 데모가 이미 가진 자산(②까지 온 것, 버리지 않음):** 온톨로지, 49 결정론 규칙(R01~R49), 재현성, 참조 DB 118,902건, 단계별 HITL, KNIA 401 사고유형.
**아직 없어서 AAP로 가려면 채워야 할 것:** ↑표의 굵은 우측 6칸 — **동적 분해 · 독립 교차검증 · 충돌 표면화 · 권한 스코핑 · 차등 HITL · DAG 추적**.

> 즉 AAP화는 "처음부터 다시"가 아니라 **49규칙·온톨로지를 멀티 에이전트가 공유·분업하도록 재배치**하는 일이다.

## 2. 단일 → 멀티, 실제로 바뀌는 7가지 (작동 메커니즘)

추상적 "멀티가 좋다"가 아니라, **손해사정에서 구체적으로 무엇이 달라지는가**.

| # | 델타 | 단일(현 데모) | AAP 멀티 | 손해사정에서 왜 중요한가 |
|---|---|---|---|---|
| D1 | **작업 분해** | 모든 건이 5단계 고정 통과 | **Planner가 건별로 필요한 작업만 동적 라우팅** | 대물단독/대인포함/사망/사기의심이 다른데 고정 파이프라인은 단순건 과잉·복잡건 누락 |
| D2 | **독립 판단** | 앞 단계 출력을 뒤가 물려받음(**오류 전파**) | 전문 에이전트가 **독립 산출 후 종합** | 견적 오류가 과실·구상까지 오염되는 걸 차단. 독립=교차검증 가능 |
| D3 | **충돌 표면화** | 한 체인이라 모순이 숨음 | 에이전트 간 **불일치를 명시적으로 띄움** | 과실 agent=90:10인데 손해액 agent는 30:70 가정으로 구상 계산 → **검출·재조정** |
| D4 | **권한 스코핑** | 한 에이전트가 모든 DB 접근 | **사기 agent만 청구이력 전체, 약관 agent는 약관 코퍼스만** | 사기탐지가 손해액 산정과 분리돼야 게이밍 방지·책임 명확 (BEAST Credential Scope) |
| D5 | **차등 HITL** | 전 단계 일괄/끝 한 번 | **위험·금액·에이전트별로 다른 사람에게 이관** | 대인 고액→의료자문, 사기 高→SIU, 과실 분쟁→상급 사정사, 저위험 대물→자동. **처리시간 절감의 핵심** |
| D6 | **DAG 추적** | 선형 체인 | **누가 무엇 근거로 판단, 종합이 어떻게 충돌 조정**을 그래프로 | 민원·소송 시 "**두 독립 에이전트가 같은 결론**"=한 체인보다 강한 방어 근거 |
| D7 | **학습 단위** | 규칙 보정 | **에이전트별 성능·드리프트 추적, 에이전트/스킬 단위 교체** | 보종 갈아끼우기=에이전트 재조합. 사기룰 한 에이전트만 갱신해 전 지점 반영 |

## 3. 정직한 경계 — 멀티가 과한 경우 vs 제값 하는 경우

**항상 멀티가 정답은 아니다.** 이 경계를 명시하는 것이 설계의 신뢰성을 만든다(아키텍처 과시 ✕).

| 케이스 성격 | 단일 경로 | 멀티 오케스트레이션 | 결론 |
|---|---|---|---|
| 대물단독 · 명확과실 · 저액 | 같은 결과를 더 빠르게 | 지연·비용만 추가 | **단일로 충분** |
| 분쟁 · 고액 · 대인포함 · 사기의심 | 그럴듯하지만 **놓침** | 단일이 못 잡은 충돌·과잉을 잡음 | **멀티가 제값** |

→ **설계 결론:** AAP는 "항상 멀티"가 아니라 **Planner가 케이스 복잡도에 따라 단일 경로/멀티 오케스트레이션을 선택**한다(D1).
이게 단일 에이전트보다 진짜 우월한 지점이다 — 단일은 *선택할 줄 모른다*.

> **데모 함의:** 케이스 2개로 보여준다. ⓐ 단순 대물건(단일=AAP, 같은 답) → "쉬운 건 굳이 안 키운다" / ⓑ CL-2026-0417 분쟁건(**AAP가 단일이 놓친 충돌을 잡아냄**) → 가치 입증.

## 4. AAP 작동 구조 설계 (손해사정 인스턴스)

### 4-1. 에이전트 구성 + 오케스트레이션 역할

기획안 v0.1의 5개 전문 에이전트는 그대로 두고, **위에 오케스트레이션 3역할을 얹는다**(이게 ②→★의 골격).

```
                 ┌─────────── Planner ───────────┐   케이스 접수 → 필요한 작업만 분해·라우팅(D1)
                 │            (작업 분해)          │
                 ▼                                ▼
        ┌──── Orchestrator (배분·A2A 조정) ────┐  ← 에이전트 간 의존성·병렬 관리(D2)
        ▼          ▼          ▼          ▼          ▼
   [과실비율]  [손해액]   [약관·보장]  [사기·이상]  [유사사례]   ← 5 전문 에이전트(각자 도구·권한 스코핑 D4)
        └──────────┴──────────┴──────────┴──────────┘
                              ▼
                    ┌── Reconciler (종합·충돌 조정) ──┐  ← 불일치 표면화·재조정(D3)
                    └────────────┬───────────────────┘
                                 ▼
                    HITL Gateway (위험·금액별 차등 이관 D5) → 손해사정사 / SIU / 의료자문 / 자동
                                 ▼
                    Action(지급결의·통지·구상) + DAG 트레이스(D6) + 학습 환류(D7)
```

| 역할 | 하는 일 | 캐논 매핑 / ds 자산 |
|---|---|---|
| **Planner** | 케이스 분류 → 필요한 전문 에이전트·순서 결정 | [5] Planning · Task Decomposition · Next Best Action |
| **Orchestrator** | 배분, A2A 의존성 해소, 병렬/직렬 제어 | [5] Multi-Agent Coordination · A2A Protocol / BEAST |
| **5 전문 에이전트** | 과실/손해액/약관/사기/유사사례 독립 산출 | [5] Case Reasoning + [4] 온톨로지·DMN 규칙 |
| **Reconciler** | 산출물 교차검증, 충돌 표면화·종합 | [5] Decision Candidate + [7] Decision Log |
| **HITL Gateway** | 임계 기반 차등 이관 | [5] Risk Scoring·Escalation / [8] HITL |

### 4-2. 케이스 한 건이 실제로 흐르는 법 — CL-2026-0417 (A2A + 충돌)

기획안 5장 케이스를 **멀티 에이전트로 실제 흘려보낸 시퀀스.** 단일 파이프라인과 갈리는 지점을 ★로 표시.

1. **Planner**: "대물 + 경상 대인 + 사기점수 미상" → 5개 에이전트 전부 소집, 사기 agent를 **선행 경량 스캔**으로 배치.
2. **병렬 산출**(Orchestrator):
   - 과실 agent: 블박 분석 → B 신호위반(황→적) → 기본 30:70에서 **10:90으로 전환**, A 과속 가산 +10 (R04).
   - 손해액 agent: 견적 480만/차량가 1,200만 → 전손 아님(R02), 인증부품 25% 인셉티브(R03)로 **대물 산정**.
   - 약관 agent: 음주·무면허·고의 없음 → 담보 적용, 자기부담금 확정.
   - 사기 agent: 경추염좌 2주 + **과거 6개월 유사청구 2건** → 과잉진료 플래그 점등(R20).
3. ★ **A2A 의존성**: 손해액 agent는 **구상액 계산에 과실비율이 필요** → 과실 agent에 요청. *(단일 파이프라인이라면 손해액 단계가 끝난 뒤 과실 단계라 이미 늦음)*
4. ★ **충돌 표면화(Reconciler)** — 단일이 절대 못 잡는 두 지점:
   - **충돌 A**: 손해액 agent가 **초기 30:70 가정**으로 구상 계산 ↔ 과실 agent 최종 **20:80** → 불일치 검출 → **손해액 재산정 트리거**.
   - **충돌 B**: 사기 agent의 과잉진료 플래그 ↔ 손해액(대인)의 합의금 추정 → **합의금 보류, SIU 검토 경로로 분기**.
5. **HITL Gateway**: 대물은 임계 이하 → 자동 후보 / 대인은 사기 플래그 → **SIU + 사정사 동시 이관**.
6. **Action + 학습**: 확정 건 지급결의·통지, 사기 패턴은 사기룰로 환류(D7). 전 과정 **DAG 트레이스** 보존.

> **이 시퀀스의 메시지:** 단일 파이프라인은 4번의 두 충돌을 *조용히 통과시켜 그럴듯한 최종 금액*을 낸다.
> AAP는 **충돌을 띄우고 재조정**한다 — 그게 곧 소송·민원 방어이고, "AI가 왜 그렇게 판단했나"의 답이다.

### 4-3. 추적은 체인이 아니라 DAG (D6)

```
[블박]──┐                          ┌─→ 과실 20:80  ─┐
[경찰원]─┼─→ (과실 agent) ─────────┘                ├─→ [Reconciler]
[견적]──┬─→ (손해액 agent) ─ 구상요청 ─→ 과실 agent ─┘    │  충돌A 해소: 손해액 재산정
[진단서]┘                                                │  충돌B 해소: 합의금 보류→SIU
[청구이력]─→ (사기 agent) ── 과잉진료 플래그 ───────────┘
                                                         ▼
                                            확정 후보 + 근거 DAG (분쟁 시 그대로 제출)
```

각 노드에 **무엇을 조회→무엇을 찾음→어떤 규칙(R##) 적용**이 남는다. 단일의 선형 로그보다 **방어력이 높은 근거 구조**.

### 4-4. HITL 라우팅 규칙 (D5)

| 조건 | 이관 대상 | 근거 |
|---|---|---|
| 대물 · 임계 이하 · 충돌 없음 | **자동 처리** | 저위험 |
| 과실 분쟁(에이전트 간 ±10%↑) | 상급 손해사정사 | 충돌 조정 필요 |
| 사기점수 ≥ 75 | **SIU + 지급 보류** | R20 |
| 대인 고액/사망 | 의료자문 + 본부장 | R25 |
| 보험금 2,000만↑ | 심사역 | 결의 라인 |

## 5. 캐논 정합 — 이 설계가 선제안서 아키텍처에 앉는 자리

이 작동설계가 `AAP_v0.1/` 캐논과 **용어·구조 1:1로 맞는지** 확인(데모=선제안서의 살아있는 증거).

| Execution Loop | 본 설계의 해당 부분 | ds 자산 |
|---|---|---|
| **Data** | 사고접수·블박·진단서·견적·경찰원 통합·정규화 | **AI:ON-U**(OCR·Parser·Embedding) |
| **Semantic** | 사고유형·당사자·손해항목·약관·면책 온톨로지 | [4] ds주도 차별화 |
| **Reasoning** | Planner+Orchestrator+5에이전트+Reconciler | **BEAST**(Tool/A2A 통제) + 49규칙 |
| **Integration** | 지급결의·통지·구상 라우팅 | **Antbot**(Workflow·RPA) |
| **Learning** | 사정결과→기준·과실DB·사기룰 환류 | [7] ds주도 차별화 |

> **검토 포인트:** ②→★ 전환의 신규 컴포넌트(Planner·Orchestrator·Reconciler·A2A)는 캐논 [5] Agentic Reasoning에 이미 항목이 있다 → 선제안서가 약속한 구조를 **이 도메인이 실제로 구현**해 보이는 셈.

## 6. 작동을 보여주는 법 — "결과 대비"에서 "작동 대비"로

기존 데모(dmp-portal)·기획안 v0.1은 **결과 카드**를 대비했다. 이 설계의 데모는 **작동 자체**를 가시화한다.

| 보여줄 장면 | 단일(②) 모드 | AAP(★) 모드 |
|---|---|---|
| 케이스 투입 | 5단계가 **순서대로 하나씩** 진행 | **Planner가 작업 분해** → 트리로 펼침 |
| 추론 | 한 줄로 흐름 | **4 에이전트 동시 점등 + A2A 화살표** |
| ★ 결정적 순간 | 충돌 없이 최종 금액 산출 | **충돌 팝업 → Reconciler가 재조정** |
| 사람 검토 | "전 건 검토" 한 덩어리 | **건별로 다른 사람에게 라우팅** |
| 근거 | 선형 리스트 | **DAG가 실시간으로 쌓임** |

**핵심 인터랙션:** 상단 **단일 ↔ AAP 토글**. 같은 CL-2026-0417을 단일로 돌리면 *그럴듯한 답*이 나오고,
AAP로 돌리면 **"잠깐 — 과실 가정과 구상 계산이 안 맞습니다 / 과잉진료 의심입니다"** 충돌이 떠서 잡힌다.
→ *"보세요. 같은 입력인데 단일은 통과시켰고 AAP는 잡았습니다."* 이 한 장면이 7개 델타를 다 설명한다.

> 줌인 ③ 추론 위젯은 세탁 v0.7의 "왜? 드릴다운 + What-if + 재현성"을 **그대로 편입**(기획안 v0.1 §6 승계), 단 여기에 **멀티 에이전트 점등·A2A·충돌 조정 레이어를 덧댄다**.
