# AAP 손해사정 데모 — 외부 전문가 검토 핸드오프

> 작성일 2026-06-08 · KT DS AX컨설팅팀
> 목적: 자동차 손해사정 도메인에 AAP(Agentic AI Platform)를 적용한 **작동 설명 데모와 설계**를, 외부 전문가(에이전틱 AI 아키텍트 / 손해사정 도메인 SME)에게 **구조·정합성·도메인 타당성** 검토를 받기 위한 자기완결형 핸드오프.
> 이 문서 하나로 검토 맥락이 서도록 작성했고, 깊은 검토가 필요하면 §10 참고 파일을 함께 본다.

## 0. 한 문단 요약 (먼저 읽을 것)

자동차 손해사정 1건(분쟁·대인·사기 의심)을 소재로, **① 기존 단일 Agent(고정 파이프라인) vs ② AAP 멀티 Agent(동적 오케스트레이션)**가 같은 입력에서 어떻게 다르게 작동하는지를, **실제 AAP 아키텍처(8계층 + Execution Loop) 위에서 구성요소가 점등되는 방식**으로 보여주는 단일 HTML 데모와 그 설계 문서다. 핵심 주장은 *"멀티 Agent라서 좋다"가 아니라 **충돌을 표면화하고, 위험 건만 사람에게 차등 이관하며, 근거를 학습 자산으로 남긴다**"*이다. **현 단계는 결정론 시뮬레이션 기반 "작동 설명 레이어"이며, 실제 Agent·LLM·온톨로지가 구동되는 시스템이 아니다**(§8 한계 참조).

## 1. 배경 & 목적

- AAP는 고객 기존 IT 자산을 대체하지 않고 그 위에 "Agent가 업무 의미·정책·권한으로 해석→판단 후보 생성→사람 검토→시스템 반영→학습 환류"하는 **공통 운영 기반**을 얹는 플랫폼. 손해사정은 그 **3번째 검증 도메인**(앞서 세탁 PoC, 금융 내부통제 PoC 진행).
- 손해사정을 택한 이유: **금액이 즉시 분쟁(소송·민원)**, **약관·과실기준이 두껍고**, **사기 리스크**까지 얹혀 "왜 그렇게 판단했는가"의 **재현성·추적성**이 가장 절실 → AAP의 결정론 엔진·오케스트레이션·HITL을 가장 설득력 있게 보여줄 무대.
- 이 데모의 목표 청중: **고객/임원/기술검토자**. 따라서 "직관적 설명"과 "구조적 깊이"를 동시에 요구.

## 3. 구현 항목 & 방식

### 3-1. 데모 v0.6 (현행본) — 3탭 구성
- **탭① 아키텍처 위 작동**: 최신 8계층 아키텍처 + 상단 **Execution Loop 바**(Data→Semantic→Reasoning→Decision→Action→Learning)를 HTML로 재현. `▶ 사건 흘려보내기`를 누르면 사건이 **9단계**로 흐르며 *(a) Loop 단계 (b) 활성 계층 (c) 실제 쓰이는 구성요소*가 순서대로 점등되고, 단계마다 캡션 + "이 사건 데이터"가 갱신된다.
- **탭② 구성요소 트레이스**: 9개 핵심 구성요소를 `[구성요소·Layer·ds자산] / [이 사건에서 한 일 → 산출] / [기존 단일 Agent는]` 3열로.
- **탭③ 심층 비교**: 기존 vs AAP **7개 차원** 매트릭스(작업분해·교차검증·충돌처리·권한거버넌스·차등HITL·추적재현·학습확장) + 결과 대비.

### 3-2. 설계 문서 (작동설계 v0.1) — 단일→멀티 7개 델타
D1 동적 작업분해 · D2 독립 판단 · D3 충돌 표면화 · D4 권한 스코핑 · D5 차등 HITL · D6 DAG 추적 · D7 학습 단위. 각 델타에 손해사정 구체 예시와 "단일로 충분한 경우 vs 멀티가 제값 하는 경우"의 정직한 경계를 명시.

### 3-3. 시연 케이스 CL-2026-0417 (고정·시뮬레이션)
- 사고: 차선변경 접촉 + 경상 대인(경추염좌 2주). 대물 수리 480만 / 차량가 1,200만(전손 아님). 합의금 요구 300만. 기본 과실 30:70. 6개월 내 유사청구 2건(사기 점수 78).
- 결과 대비: **기존 단일 = 2,610,000원 일괄 지급권고** / **AAP = 자동 900,000원 + 보류 1,710,000원(SIU·손해사정사)**.
- 충돌 2건: ⓐ 과실 가정 불일치(과실 Agent 20:80 ↔ 손해액 Agent 30:70 가정) ⓑ 합의금 ↔ 사기 과잉진료.

### 3-4. 기술 구현 방식 (불변 원칙)
- **단일 HTML · 외부 라이브러리/CDN 0 · 오프라인 실행**.
- **결정론 시뮬레이션**: 과실·금액·사기점수를 입력값에서 계산(하드코딩 아님), 화면에 "Simulation PoC" 명시.
- 점등/탭/단계 재생은 vanilla JS. 접근성(aria-selected/tabpanel) 적용.
- 원본 28MB 데모는 **수정 없이** 별도 참조(과거 iframe 임베드안은 "설명에 불필요"로 제거).

## 4. 핵심 설계 결정 & 근거

| 결정 | 근거 |
|---|---|
| **순서도 ✕ → 아키텍처 위 구성요소 점등 ✓** | "어떤 구성요소를 어떻게 활용해 저 결과가 나오는지"가 핵심. 일반 플로우는 그 설명력이 없음 |
| 비교 = 1줄 숫자 ✕ → **7차원 매트릭스 + 구성요소 트레이스** | 기존 vs AAP 차이를 단편적으로 보이면 안 됨 |
| **"항상 멀티" ✕ → Planner가 복잡도 보고 선택** | 단순 대물건은 단일 경로가 더 빠름. 멀티의 가치는 분쟁·고액·대인·사기 건에 집중 |
| 결과를 **자동/보류로 분리** | "AI가 전액 지급"이 아니라 "위험분만 사람에게" — HITL·책임소재가 핵심 메시지 |
| 모든 수치에 **Simulation 기준** 명시 | 확정 성과로 오인 방지 |

## 5. AAP 구성요소 ↔ 손해사정 작동 매핑 (검토 대상)

> 데모 탭①의 9단계. **이 매핑의 정확성·누락이 핵심 검토 포인트.**

| # | Execution Loop | 활성 계층 | 점등 구성요소 | 이 사건에서 한 일 |
|---|---|---|---|---|
| 1 | Data | L5,L8 | 연결·수집 / 처리 파이프라인 / 스토리지 | AI:ON-U 문서파싱 → 구조화 |
| 2 | Semantic | L4 | 온톨로지 / 지식그래프 / 하이브리드 스토어 | 사고유형↔과실기준 매핑 |
| 3 | Reasoning | L3,L2 | 실행엔진(Planning) / **Supervisor·Task Routing** / Agent Registry | 복잡도 판단 → 멀티 경로·5 Agent 배분 |
| 4 | Reasoning | L2,L3,L7 | **Tool Registry / Credential Scope** / 도구실행 / 가드레일 / 권한관리 | Agent별 권한 범위 도구만 호출 |
| 5 | Reasoning | L3,L4 | 행동실행 / **Agent Coordination(A2A)** / 멀티에이전트 협업 / 컨텍스트 | 독립 판단 → 충돌 2건 표면화 |
| 6 | Decision | L3,L6 | **HITL 런타임 검토 게이트** / 실행과정추적(Decision Log) / 평가 | 충돌·위험 표면화 + 근거 기록 |
| 7 | Decision | L7 | HITL 승인 / 정책 통제 / 권한 | 임계·위험·금액별 차등 이관 |
| 8 | Action | L3,L1 | 응답생성 / 관리자콘솔 / 포털 | Antbot 지급결의·보류 케이스 |
| 9 | Learning | L7,L6,L4 | Self-Improving / 피드백 / 온톨로지 | 사기룰·과실DB·온톨로지 환류 |

> ds 자산 매핑: **AI:ON-U**(OCR·Parser·Workflow), **BEAST**(Tool Registry·Credential Scope·API Gateway), **Antbot**(RPA·Workflow), 자사 검증 Agent. 차별화 영역 = [4]의미화·[7]운영학습.

## 6. 기존 단일 Agent vs AAP — 비교축

| 차원 | 기존 단일 Agent | ★ AAP |
|---|---|---|
| 작업 분해 | 고정 5단계(전건 동일) | Supervisor가 복잡도 보고 단일/멀티 선택 |
| 독립 교차검증 | 순차 누적(오류 전파) | 전문 Agent 독립 + A2A 상호 검증 |
| 충돌 처리 | 한 체인(자기모순 숨음) | Reconciler·런타임 게이트가 표면화 |
| 권한·거버넌스 | 한 Agent 전권 | Agent별 Credential Scope |
| 차등 HITL | 전건/끝 일괄(병목) | 위험·금액별 차등 이관 |
| 추적·재현 | 선형 로그 | Decision Log 근거사슬 |
| 학습·확장 | 개인 노하우 | 환류 + Agent 재조합 |

## 7. 검토 요청 포인트 (전문가에게 묻는 질문)

### 7-1. AAP / 멀티 에이전트 아키텍처 관점
1. **차별화의 진정성**: 7개 델타가 실제 운영 가치인가, 일부는 과설계(architecture theater)인가? 특히 원본 단일 데모가 이미 49규칙·FDS·온톨로지를 갖춘 점을 감안할 때, 멀티 Agent가 손해사정에서 **정말** 우월한 지점은 어디까지인가?
2. **구성요소 매핑 정합성**: §5의 단계↔구성요소 매핑에 잘못 매핑·누락·중복이 있는가? (예: A2A를 step5 충돌검증에, HITL 런타임 게이트를 step6에 둔 것이 적절한가?)
3. **충돌 조정의 설계**: Reconciler/런타임 게이트가 충돌을 표면화하는 모델이 비결정적 Agent 출력과 실제 트랜잭션 사이의 안전을 충분히 보장하는가? (Idempotency·Compensation·재산정 루프)
4. **거버넌스 단일 소재**: Credential Scope(Agent별 권한)와 HITL 경계가 Platform 수준에서 일관되게 관리되는가? 책임소재가 모호하지 않은가?

### 7-2. 손해사정 도메인 관점 (SME)
5. **수치·로직 현실성**: CL-2026-0417의 과실(30:70 기본), 전손판정(수리/차량가 40%), 합의금 300만, 사기 78점, **자동 90만/보류 171만 분기**가 실무적으로 타당한가?
6. **충돌 시나리오 타당성**: ⓐ 과실 가정 불일치, ⓑ 합의금↔과잉진료가 실제 손해사정 분쟁에서 흔히 발생하는 패턴인가? 더 설득력 있는 케이스가 있는가?
7. **HITL 라우팅 정합성**: 자동/손해사정사/SIU/의료자문 차등 이관이 실제 보험사 업무 분장·결재 라인과 맞는가?

### 7-3. 거버넌스 · 규제 관점
8. **규제 적합성**: 보험업법·손해사정 관련 규정상 "AI 판단 후보 + 사람 확정" 구조의 설명책임·감사 요구를 충족하는가? 자동 지급의 허용 범위는?
9. **분쟁·소송 방어**: Decision Log 근거사슬이 민원·소송 대응 증거로 실제 효력이 있는 수준인가?

## 8. 한계 & 전제 (검토 시 반드시 감안)

- **작동 설명 레이어이지 작동 시스템이 아님**: 실제 LLM·Agent·온톨로지·규칙엔진이 구동되지 않는다. 점등·결과는 **결정론 시뮬레이션 가정값**.
- **수치는 미검증**: 과실·손해액·사기점수·자동/보류 분기는 SME·계리 검증 전 illustrative 값.
- **원본 단일 데모는 별개 팀 산출물**: 49규칙·FDS를 갖춘 실제 React 앱. 본 데모의 "기존 단일 Agent" 묘사가 그것을 **과소평가하지 않았는지**가 비교 공정성의 쟁점.
- **단일 사건 1건**: 단순건(단일 경로)과의 대비, 사망·복합사고 등 다양한 케이스는 미구현.
- 아키텍처는 **260608 PNG를 HTML로 재현**한 것 — 원 아키텍처 자체의 타당성은 별도 문서(`AAP_v0.1/aap_architecture_detail_for_llm_review_260605.md`)에서 검토.
