## 1. 탭① 처리 방식별 Workflow

### 1-0b. 도구 3종 메타(METHODS_META — 공통 정의)
| 도구 | 부제 | 하는 일(doIt) | 한계(limit) | vs AAP |
|---|---|---|---|---|
| 🖐 수기 처리 | 담당자 직접 판단 | 담당자가 자료를 확인하고 직접 판단합니다. | 판단 차이가 검토 항목으로 남지 않습니다. | AAP는 판단을 분리해 비교·검토로 나눕니다. |

### 1-1. STEP 1 지급 심사
- **기준선(수기) 칩**: 신분증·계약 대조 → 운전자 한정 확인 → 청구이력 수기 조회 → 담당자 판단
- **수기의 한계**: 본인·운전자·사기를 사람이 각각 조회 — 의심 신호를 놓치면 부당지급 위험이 커집니다.
- **도구 3종(돕는 곳 ~지만, 멈추는 곳)**
  - 챗봇: 본인·계약 내용을 안내할 수는 있지만, 사기 패턴을 검증 경로로 분리하지는 못합니다.
  - RAG: 약관·한정특약 근거는 찾지만, 의심 신호가 겹칠 때 REVIEW·SIU로 라우팅하지 못합니다.
  - 단일 Agent: 사기점수 하나는 낼 수 있지만, 본인·운전자·사기를 나눠 어디서 막혔는지 보여주기 어렵습니다.
- **AAP 도식 노드**: ①본인 확인 ②운전자 적합성 ③사기 탐지(78점) → ④자격 판정(3개 병렬 검증) → [정상]지급 대상 PASS / [의심]사람 검토 REVIEW / [면책]면책 DENY
- **이 단계에서 AAP가 처리하는 것**(쉽게 말하면): 본인·운전자·사기 확인을 한 번에 돌려, 정상 청구는 바로 통과시키고 사기 의심만 따로 빼냅니다.
  - 본인·운전자 확인을 통과해 정상 청구로 판단됨
  - 사기 점수 78점이 기준선 75점을 넘음 — 6개월 내 2회 반복 청구가 주된 이유(가정)
  - 바로 거절하지 않고, 보류 후 특별조사(SIU=보험사기 전담 조사)로 사실을 먼저 확인

### 1-2. STEP 2 견적 산정
- **기준선(수기) 칩**: 차량 특정 → 파손부위 수기 확인 → 부품·공임 조회 → 견적 확정
- **수기의 한계**: 파손 정도·통상가를 경험으로 판단 — 산출 편차와 누락이 생깁니다.
- **도구 3종**
  - 챗봇: 견적 항목을 요약할 수는 있지만, 사진으로 파손 정도를 판정하지는 못합니다.
  - RAG: 부품·공임 기준은 찾지만, 인증부품·외산배율·전손 규칙을 산출에 적용하지 못합니다.
  - 단일 Agent: 총 수리비 하나는 낼 수 있지만, 전손·연쇄손상·과다 항목을 분리하기 어렵습니다.
- **AAP 도식 노드**: ①사진 분석 → ②부품·공임 산출 → ③규칙 적용(인증부품·전손) → ④견적 판정 → [명확]견적 확정 480만 원 / [전손·과다]전손·과다 검토
- **AAP 처리(쉽게 말하면)**: 사고 사진을 분석해 수리비를 1차로 뽑고, 너무 비싸거나 폐차가 나은 건만 사람이 확인합니다.
  - 사진으로 파손 부위·정도를 눈대중이 아닌 수치로 환산
  - 정품 환급·외산차 할증 같은 규칙을 매번 같은 기준으로 적용
  - 수리비가 차값의 일정 비율을 넘으면 전손(고쳐 쓰기보다 폐차가 나은 상태)으로 검토

### 1-3. STEP 3 대인 피해
- **기준선(수기) 칩**: 진단서 확인 → 상해등급 판정 → 합의금 수기 산정 → 합의서 작성
- **수기의 한계**: 상해등급·합의금을 수기 산정 — 과잉진료·과다 합의금을 잡기 어렵습니다.
- **도구 3종**
  - 챗봇: 진단서 내용을 정리할 수는 있지만, 상해등급·합의금을 약관 기준으로 산정하지 못합니다.
  - RAG: 합의금 벤치마크는 찾지만, 통상 치료기간 대비 과잉진료를 탐지하지 못합니다.
  - 단일 Agent: 합의금 하나는 제시하지만, 적정성·과잉진료를 검토 대상으로 분리하기 어렵습니다.
- **AAP 도식 노드**: ①상해등급 산정 ②과잉진료 탐지 → ③합의 판정 → [적정]합의 확정 / [과잉·과다]적정성 검토
- **AAP 처리(쉽게 말하면)**: 다친 정도에 맞는 등급으로 합의금을 계산하고, 치료가 과한 건만 골라 검토합니다.
  - 상해 등급은 약관 기준으로 자동 산정
  - 통상 치료기간보다 길면 과잉진료(필요 이상 치료) 신호로 봄
  - 같은 등급의 보통 합의금보다 높으면 적정성 검토 대상

### 1-4. STEP 4 과실 산정
- **기준선(수기) 칩**: 증거 검토 → 사고유형 분류 → 과실비율 수기 판단 → 협상
- **수기의 한계**: 증거가 엇갈려도 과실비율을 한 값으로 확정 — 분쟁·민원 위험이 커집니다.
- **도구 3종**
  - 챗봇: 사고 설명을 정리할 수는 있지만, 블랙박스 기준과 판례 기준의 차이를 비교하지 못합니다.
  - RAG: 판례·사고유형 기준은 찾지만, 판정이 충돌할 때 사람 검토로 분리하지 못합니다.
  - 단일 Agent: 과실비율 하나를 제시하지만, 20:80와 30:70처럼 갈릴 때 차이를 검증 대상으로 만들기 어렵습니다.
- **AAP 도식 노드**: ①블랙박스 과실(20:80) ②판례 과실(30:70) → ③A2A 판정 비교(차이 10%p) → [일치]일치 시 확정 / [불일치]불일치 사람 검토
- **AAP 처리(쉽게 말하면)**: 블랙박스가 본 과실과 판례가 본 과실이 다르면, 한쪽으로 정하지 않고 사람에게 넘깁니다.
  - 블랙박스 영상 기준과 판례(과거 비슷한 사고 판결) 기준의 과실을 나란히 비교
  - 두 기준의 차이 10%p는 자동 확정 대신 사람 검토 기준(가정)
  - 협상 전략을 미리 세워 분쟁·민원을 사전에 줄임

### 1-5. STEP 5 처리 방법
- **기준선(수기) 칩**: 결과 취합 → 지급액 수기 산정 → 보고서 작성 → 결재 상신
- **수기의 한계**: 결과를 손으로 취합해 보고서 작성 — 자동/검토 구분·근거 추적이 약합니다.
- **도구 3종**
  - 챗봇: 처리 의견을 설명할 수는 있지만, 자사 손실 관점의 처리 시나리오를 만들지 못합니다.
  - RAG: 문서 양식·기준은 찾지만, 어떤 항목을 자동 처리하고 어떤 항목을 사람 검토로 보낼지 정하지 못합니다.
  - 단일 Agent: 단일 지급액은 내지만, 4개 처리 시나리오 비교나 자동/검토 분리를 만들기 어렵습니다.
- **AAP 도식 노드**: ①시나리오 생성 → [①최적안·②공격안·③안전안·④전략안] → ②손해사정사 선택 → [확실]자동 90만 원 / [다툼·의심]검토 171만 원
- **AAP 처리(쉽게 말하면)**: 확실한 금액은 자동으로 정산하고, 다툼·의심이 있는 금액만 사람이 합의서를 확인합니다.
  - 4개 처리 시나리오를 자사 손실 관점에서 자동 생성
  - 확실분 90만 원은 자동 처리로 빠르게 정산
  - 검토분 171만 원은 사람 확인 후 합의서·보고서를 자동 작성

## 2. 탭② AAP 구조 지도

### 2-0. 추론 루프 바(ARCH_LOOP)
AAP 추론 루프 — 이 단계가 거치는 순서: **Data(L5) ▶ Semantic(L4) ▶ Reasoning(L3·L6) ▶ Decision(L7) ▶ Action(L1·L8) ▶ Learning(L7)**

### 2-0b. Customer Environment 밴드
- 헤더: Customer Environment — 고객 자산 · 대체 아님 · 입력 기반
- 항목: Data Platform(Lakehouse · BI) / Business Systems(ERP · CRM · 결재) / Knowledge Assets(문서 · KMS · 상담이력)

### 2-0c. Trust · Security 밴드
- 헤더: Trust · Security — 전 영역 횡단 통제
- 항목: IAM · SSO · RBAC(접근 권한) / 감사 · 근거 · 이력(Explainability) / PII · DLP · Prompt Injection(데이터 보호) / 폐쇄망 · On-prem(배포 경계)

### 2-1. 8계층 셀(ARCH_LAYERS) — 계층 / 셀(부제 · 배지)
- **L1 경험 / 접근**: 에이전트 포털 · 코파일럿 / 챗 UI · 관리자 콘솔 · API Gateway / SDK
- **L2 설계 / 개발**: 에이전트 설계(Low-Code · UX) · 프롬프트·워크플로우 설계 · Tool Registry(API·함수·플러그인) · Agent Registry(카탈로그·버전·배포 · 배지: **자사 검증 Agent**)
- **L3 에이전트 실행환경 [★ core]** (미니플로우: 입력 처리 → 계획 → 행동 실행 → 응답 생성): 접근(REST · GraphQL · WebSocket · gRPC · Event) · 코어·실행 엔진(메모리 · 모델 게이트웨이 · 도구 실행 · 가드레일 · 에이전트 관리) · 오케스트레이션·멀티에이전트(Supervisor · Task Routing · A2A · 협업) · HITL 런타임 게이트(Runtime Review / Approval)
- **L4 지식 / 시멘틱 [★ ds 차별화]**: 온톨로지·시멘틱 모델 · 도메인 지식그래프 · 하이브리드 지식 스토어 · 컨텍스트 조합·근거
- **L5 데이터 / 연동** (미니플로우: 수집 → 정제 → 정규화 → 변환): 연결·수집(DB · API · SaaS · 파일/이벤트 커넥터 · 배지: **AI:ON-U (OCR)**) · 데이터 처리·파이프라인(정제 · 정규화 · 변환) · 데이터 관리(스키마 · 품질)
- **L6 AI 품질 / 평가**: 실행 추적(Decision Log) · 품질·근거 평가(배지: **Antbot (workflow)**) · RAI · 안정성 · 성능·비용(배지: **BEAST (tool)**) · 사용자 피드백
- **L7 컨트롤 / 거버넌스 [★ ds 차별화]**: 정책 관리·통제 · 에이전트 권한 · HITL 승인 기준 · Self-Improving(운영 학습 자산화) · 사용량·비용 가드레일
- **L8 인프라**: 컴퓨팅·GPU · 컨테이너·K8s · 네트워크·보안 · 스토리지·DB · 온프레/클라우드

### 2-2. STEP별 경유 셀 순서·결과(STEP_TRACE) + 우측 해석(STEP_BASIS.interp)

**STEP 1 지급 심사** · 결과: 지급 대상 PASS + 사기 78점 REVIEW·SIU
1. L5 연결·수집 — 신분증·계약·청구이력 수집
2. L4 도메인 지식그래프 — 관계인·반복청구 연결
3. L3 코어·실행 엔진 — 자격·사기 탐지 Agent
4. L6 품질·근거 평가 — FDS 사기점수 검증
5. L7 정책 관리·통제 — PASS/DENY/REVIEW 분기
- 해석: 사기 의심 점수는 지급을 거절하는 근거가 아니라 "더 확인이 필요하다"는 신호입니다. 정상 청구는 자동으로 통과시키고 의심 건만 사람·특별조사로 보내, 부당지급과 과잉심사를 함께 줄입니다.

**STEP 2 견적 산정** · 결과: 대물 견적 480만 원 + 미수선 옵션
1. L5 연결·수집 — 사고사진·차량정보 수집
2. L4 온톨로지·시멘틱 모델 — 파손·부품·공임 의미화
3. L3 코어·실행 엔진 — 사진분석·견적 Agent

- STEP3: 진단서 기준으로 상해등급과 치료기간을 확인합니다. / 통상 치료기간과 비교해 과잉진료를 점검합니다. / 등급 벤치마크와 요구 합의금을 비교합니다.
- STEP4: 경찰·블랙박스·CCTV 순으로 증거를 검토합니다. / 기준별 과실을 비교합니다 — 판정이 갈립니다. / 타사 주장·중재안·증거로 협상 전략을 세웁니다.
- STEP5: AI 추천안입니다 — 미수선 현금정산으로 자사 비용이 가장 낮습니다. / 과실 재협상·구상권으로 타사 청구를 줄이는 안입니다. / 제휴센터 수리로 분쟁 리스크를 낮추는 안입니다. / 특수 경로를 활용하는 전략안입니다.

### 3-7. 우측 「AAP 작동 순서」 패널
- 헤더: AAP 작동 순서 / 부제: 단계를 차례로 거치며 좌측 콘솔 결과가 만들어집니다 / 버튼: 다시 실행
