## 1. 공통 변경 (페이지 하단·고지 정리)

> 탭1 피드백에 적혔지만 공유 요소이므로 전 탭에 일괄 적용.

1. **`#footnote` 완전 삭제** (HTML 요소 + `setPhase`/init에서 `footnote` 채우는 코드 제거).
2. **`.overclaim`(“본 화면의 금액·점수·분기 기준은 … 가정값입니다”) 삭제** — 페이지 하단·탭② `ndPanel`의 `nd-foot`·탭③ `sa`/콘솔 하단의 동일 취지 고지문(“※ 금액·점수·임계값은 설명용 가정값…”) 모두 **한 곳(검증 필요 항목 패널) 빼고 제거**. 헤더의 `badge-val`(“설명용 가정값·SME 검증 필요”) 1개만 남겨 중복 고지 종결.
3. **하단에는 `검증 필요 항목(SME)` = `.val-panel` 만 남김.** `.valwrap`에서 `overclaim` 컬럼 제거하고 `val-panel`을 단독 폭으로.
4. **`.legend-bar`(색상 의미) 재배치 + 축소:**
   - 위치: 페이지 최하단 → **각 탭 다이어그램 바로 위(또는 탭 본문 상단)** 의 눈에 띄는 위치로. (탭마다 쓰는 색이 다르므로 탭별로 두거나, 최소 탭①·②에 배치.)
   - 항목: 7개 → **화면에서 실제 쓰는 색 5개 이내**. 권장 세트 = `Agent·활성(teal)` / `사람 검토(amber)` / `위험·불일치·보류(red)` / `결과 후보(mint)` / `기록·기반(gray)`. (`입력 데이터(blue)`·`지식·문서(violet)`는 해당 탭에서 실제 노출되는 경우만 유지.)

## 2. 탭① 처리 방식별 Workflow

### 현재
3개 컬럼(기준선 | 도구 3종 | AAP)이 동등 폭으로 나란히. 상단에 안내 lede 1문단.

### 변경 지시
1. **lede 문구 삭제:** `<section data-tab="0">`의 `.tlede p`(“좌측은 … 우측은 … STEP n 기준.”) 제거. (제목 `h3` + `aud` 배지는 유지 가능, 단 “직책자/의사결정자용” 한 줄까지만.)
2. **레이아웃 재편 — 3등분 폐기, “기준선(상단 밴드) + 도구 vs AAP(하단 2분할)”:**
   - **상단 밴드(전폭): 기준선 = 이 STEP의 실제 수기 처리(출발점).** 회색/중립 톤으로 “비교 대상이 아니라 출발점”임을 시각적으로 표현. 내용 = `${icon} ${step} ${name}` + `wf.manual.nodes`(가로 칩) + `MANUAL_PAIN[i]`(수기의 한계 한 줄). 현재 `#t1Base` 콘텐츠를 가로 밴드로 변환.
   - **하단 2분할: 좌 = 기존 도구 3종(챗봇·RAG·단일 Agent) / 우 = AAP.** 가운데 **`vs` 또는 대비 강조선**을 둬서 “**도구 방식 ↔ AAP 방식**”이 이 탭의 핵심 대비임을 명확히.
     - 좌(도구): 현재 `#t1Tools`의 3 셀(돕는 곳/멈추는 곳) 유지하되, “여기까지 돕고 멈춘다”가 한 덩어리로 읽히게 묶음. 중립/회색 톤.
     - 우(AAP): 현재 `#t1Aap`(분리→비교→사람 검토→기록 + `t1Aap()` 다이어그램 + 작동/분기 기준/업무 결과 + 자동/검토 분리 금액) 유지하되 **더 크고 teal 강조**. 좌보다 시각 위계 우위.
   - **수용 기준:** 한 화면에서 “수기(출발점) → 도구는 여기서 멈춤 / AAP는 구조로 바꿈”이 좌우 대비로 즉시 읽혀야 함. 스크롤 없이 도구 vs AAP가 한눈에.
3. **색상 의미:** 위 §1.4대로 이 탭 비교 블록 **바로 위**에 5개 이내 범례 배치.
4. **하단:** 이 탭 끝에는 검증 필요 항목(공유 `val-panel`)만. (footnote·overclaim 없음 — §1.)

## 3. 탭② AAP 구조 지도  ← **가장 큰 재작업**

### 현재 문제 (피드백)
- 키스톤 단일 그래프(8 클러스터를 세부 노드로 쪼개 연결)는 **너무 어렵고 와닿지 않음.** “지도”라면서 **연결 관계만** 보여주고, **AAP 8 구성요소(8계층)가 매핑되지 않음.**
- **STEP 1~6 버튼에 전혀 반응하지 않음**(phase 미연동).
- 동적 점등·흐름은 **이전 버전(v0.44 자기설명 지도)이 더 나았음.**

### 변경 지시 — v0.44식 “자기설명 지도(phase 점등)”로 복원·개선
1. **graphMap 클러스터 뷰를 탭②에서 내리고**, v0.44 `renderRuntimeMap` 계열의 **8계층 캐논 지도 + STEP 점등**으로 교체.
   - 좌측 = **고정 캐논 8계층(CUST + L1~L8) 지도.** 현재 STEP의 `runtimeRoute`에 해당하는 계층만 **점등**, 나머지는 dim. `MAP_POS`·`MAP_SKELETON`·`LAYER_ROLE` 재사용.
   - 활성 경로에 **연결선 라벨(`FLOW_OUT`: 계약·사고 자료 → 입력 데이터 → 담보·약관 기준 → 판단 후보 → 검증 결과 → 분기 결과 → 결과 산출)** + **경로 끝 결과 노드(`wf.aap.result`)** 표시.
   - 상단 **Execution Loop 띠(`.eloop`)** 로 이 STEP의 루프 위치 점등(`L2LOOP`/`LOOPB`/`activeStages`).
   - 구현: `..._데모_v0.44.html`의 `renderRuntimeMap(i)`를 이식하고 `#graphMap`(또는 새 `#runtimeMap`)에 렌더. **`setPhase(i)`에 `renderRuntimeMap(i)` 호출 추가** → STEP 1~6 클릭 시 좌측 지도가 갱신되게(현재 가장 큰 결함 해소).
2. **우측 패널 = “지금 무엇이 작동하는가”를 손해사정 업무 언어로 풀이.**
   - 단순 3관점 나열을 넘어, **이 STEP에서 점등된 각 계층별로 [무엇이 작동] + [손해사정 업무에서 의미하는 바]** 를 한 줄씩. 데이터 소스 = `runtimeRoute[].label` + `subflow[layer]` + `MAP_KW` + 업무 글로스. 예) `L4 지식·시멘틱 — 약관·담보·면책을 의미화` → “이 사건이 어떤 보장에 걸리는지 약관 기준으로 정리”.
   - 하단 1줄 = `routeMeaning`(이 단계 핵심) + 결과(`wf.aap.result`).
   - hover 연동(우측 계층 항목 ↔ 좌측 지도 강조) 유지 가능(v0.44 로직).
3. **노드 단위 의미 탐색이 필요하면** 기존 `ndPanel`/`NODE_INFO`를 “심화 보기(접힘)”로 강등하거나 제거. 단, **GRAPH/renderGraph 자체는 삭제 금지** — 탭③ 미니 서브그래프(`renderGraph('miniGraph',{scope:i+1})`)에서 계속 사용함.
4. **색상 의미:** §1.4대로 지도 위에 5개 이내 범례. 계층 역할색(blue/purple/teal/orange/mint/gray)이 많으면 **역할색은 지도 안 라벨로 설명**하고, 범례는 활성/검토/위험/결과/기록 상태색 위주로.
5. **고지문 삭제:** `ndPanel`의 `nd-foot`(“금액·점수·임계값은 설명용 가정값…”) 등 §1.2 일괄 적용.
6. **수용 기준:** STEP 버튼을 누르면 (a) Loop 띠 위치, (b) 점등 계층, (c) 연결 라벨, (d) 결과 노드, (e) 우측 업무 의미가 **모두 그 STEP으로 바뀐다.** “8계층 중 이번 단계가 어디를 쓰는지 + 그게 손해사정 업무로 무슨 뜻인지”가 읽혀야 함.

## 4. 탭③ AAP 작동 시뮬레이션

### 현재 문제 (피드백, 이미지2)
- 좌(콘솔)·우(작동 보조)의 **용도/디자인 구분이 약함.**
- 우측 작동 흐름이 **디자인에 과몰입**, 우측 전반(작동 흐름 + 미니 서브그래프 + 결과 3카드)에 **정보 과다.**
- 우측 **SME 검증이 하단 공유 SME와 겹침(중복).**

### 변경 지시 — 좌=현업 화면 / 우=AAP 작동, 톤으로 분리
1. **역할·디자인 분리 명확화:**
   - **좌 `#simConsole` = 손해사정사에게 보이는 업무 화면(콘솔).** 라이트/현업 톤 유지(흰 배경, 업무 카드). 현행 `renderConsole(i)` 구조(사건 카드·상태·검토 큐·액션·Decision Log·산출물·다음 단계) 유지.
   - **우 `#simAside` = AAP 내부 작동.** **명확히 구분되는 톤(다크/테크니컬)** 부여 — 어두운 배경 패널 + teal/모노 라벨로 “이건 화면 뒤 작동”임을 시각적으로 분리. (좌=고객 화면, 우=엔진. 한눈에 다른 영역으로 읽혀야 함.)
2. **우측 정보 다이어트:**
   - **SME 검증(`sim-sme`) 제거** — 하단 공유 `val-panel`과 중복(§1.3). 우측에는 두지 않음.
   - 3카드(①작동 흐름 ②미니 서브그래프 ③결과) → **핵심만.** 권장: **① 작동 흐름(cap 흐름: DATA→SEMANTIC→AGENT→…)을 메인**으로 크게, **③ 결과 한 줄** 유지. **② 미니 서브그래프는 보조로 축소하거나(작게/접힘) 제거** — “작동 흐름”과 의미가 겹쳐 과밀의 원인. (둘 중 흐름을 남기고 서브그래프를 내리는 쪽 권장.)
   - 작동 흐름 카드의 장식(애니메이션 등)을 줄여 **흐름 단계가 또렷이** 읽히게.
3. **좌측 콘솔 ↔ 우측 작동 연결**은 콘솔 섹션의 얇은 주석(`TWIN_ANNOT[i]`, “이 검토 큐 = … 게이트 결과”) 한 줄로 유지(중복 패널 금지).
4. **수용 기준:** 좌(밝은 업무 콘솔) / 우(어두운 AAP 작동)가 톤으로 즉시 구분되고, 우측은 “작동 흐름 + 결과”만 담아 가벼워야 함. SME 중복 없음.

## 5. 보존·주의 (회귀 방지)

- `CASE`(CL-2026-0417 단일 출처)·`PHASES`·`SIM`·`METHOD_CMP`·`MANUAL_PAIN`·`TWIN_ANNOT`·`RESULT_LINE` **데이터 보존.**
- `buildDiagram`/`anchor`/`spread`/`playSequence` 다이어그램 엔진 **유지.**
- **GRAPH/renderGraph 삭제 금지**(탭③ 미니 서브그래프에서 사용). 탭②에서 클러스터 뷰만 내리는 것.
- 탭 콘텐츠/요소 삭제 시 **해당 요소를 참조하는 JS(`getElementById`)도 함께 제거** — 안 그러면 `null.innerHTML`로 throw하여 이후 렌더가 멈춤(과거 반복된 버그).
- `setPhase(i)` 호출 목록에 탭② 렌더 추가 시, **init 순서**(`renderMapInit()` 또는 신규 지도 init → `setPhase` → `setTab`) 정합 확인.
