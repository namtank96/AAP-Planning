## 0. 포지셔닝 — "배치표"가 아니라 "인력 구성 사건"
인력관리 ≠ 엑셀 배치표 ✕ → **모호한 인력 수요 요청을 '인력 구성 사건'으로 의미화 → 수급 분석 → 구성안 → 사람 승인 → 배치 반영 → 운영 모니터링 → 학습**하는 업무 사건.
핵심 메시지(회의 Pack과 동일): *AAP는 Agent를 많이 띄우는 게 아니라, 업무를 이해하고 실행 구조를 구성하며 Agent·Module·기존 솔루션·Connector·Policy를 조합해 운영 가능한 업무 흐름을 만든다.*

**AAP 아키텍처 정합 보너스** — 인력 확보 4방식이 AAP 역량 확보 프레임과 동형:
`내부 재배치(Move) · 채용·육성(Build) · 외주·협력사(Borrow) · 기존 RMS·ERP 연동(Integrate)` = **Build·Buy·Borrow·Integrate**.

## 2. 8단계 캐논 흐름 (PLATFORM_CORE 스테이지 그대로 · 라벨만 교체)

| # | CORE 스테이지(중립) | actor | 회의 Pack | **인력관리 Pack** |
|---|---|---|---|---|
| 1 | intake (요청 접수) | human | 회의 요청 | **인력 수요 접수** |
| 2 | understand (업무 이해) | aap | 회의 유형·목적 | **수요 이해** (역량·등급·기간·예산) |
| 3 | compose (실행 구조 구성) | aap | 작업분해+구성요소 조합 | **수급 분석·구성** (공급 조회→매칭→갭→소싱→마진) |
| 4 | approve ★HITL① | hitl | 계획·시각 승인 | **구성안 승인** (소싱 방식·외주 단가·배치 확정) |
| 5 | execute (준비 실행) | aap | 회의 준비 | **배치 실행** (배치 반영·온보딩·채용 요청·외주 발주) |
| 6 | operate ★HITL② | hitl | 회의 진행(시작/종료) | **운영 모니터링** (가동·롤오프 추적, 사람이 모니터링 가동/이벤트 확인) |
| 7 | commit ★HITL③ | hitl | 최종 발송 승인 | **변경 반영 승인** (재배치·교체·추가투입 = 계약·원가 영향) |
| 8 | close (반영·기록·학습) | aap | 공유·기록·학습 | **반영·기록·학습** (가동률·마진 기록·추천 개선) |

→ **step id·actor·gate 구조 100% 동일.** WORK[].ops / explain / detail(odTable)만 인력관리 내용으로.

각 스테이지 ops 예시(우측 8계층에 detail로 표시):
- **③ compose**: 작업 분해(수요분석·공급조회·매칭·갭·소싱·마진 T1~T6 / L2) → 공급·근거 식별(인사·프로젝트·스킬 DB / L5) → 구성요소 조합(Agent6·Module3·솔루션3·Connector4·Policy3 / L3) → 누락·리스크 점검(롤오프 리스크·예산 초과 / L6 Antbot)
- **④ approve**: 매칭안·갭·**소싱 옵션 3안**·마진 시뮬 제시(L1) → 시뮬 산출(L3) → 게이트 보류(L3). `select`=소싱 옵션 선택.
- **⑤ execute**: 배치 반영(RMS/L3) ∥ 온보딩 트리거(계정·장비·교육/L8) ∥ 채용 요청 발행(부족분/L3) ∥ 외주 발주(협력사 포털/L5) ∥ 원가 반영(ERP/L8)
- **⑥ operate**: 가동률·롤오프 모니터링(L5) → 이상 신호 의미화(롤오프 임박·가동률 이탈/L4) → 알림·재구성 후보(L3). 신호 버튼=모니터링 시작 / 이벤트 확인.

## 3. 발산·수렴 리듬 (PDF2 차별 개념)
1차 수렴: 수요 요청 → 인력 구성 사건 ·
1차 **발산**: 가용 인력·스킬·단가·외주 후보 **병렬 탐색** ·
2차 수렴: 매칭 결과 + 갭 ·

재수렴: HITL 승인된 **단일 구성안**.
**HITL 수렴점**(책임 큰 곳만): 구성안 확정 · 외주 단가 승인 · 채용 승인 · 재배치 승인.

## 5. Component Registry (관리 뷰 · 상세 카드 일부)
> 각 카드: type 배지 · 계층(L#) · 설명 · 🧭언제(Master 선택 근거) · 🗂️사용 데이터 · ⚙️수행 방식. kt ds 자산(보라)=매칭 품질 평가(Antbot)·학습(Self-Improving).

- **공급 매칭 Agent**(L3) — 언제: 수요 직무·스킬에 맞는 가용 인력을 찾을 때 / 데이터: 인사·스킬 DB, 가용·롤오프, 단가 / 방식: 직무·스킬·등급·기간을 인력 풀과 매칭 → 적합도%·근거와 함께 후보 산출.
- **소싱 전략 Agent**(L3) — 언제: 내부 가용으로 수요를 못 채울 때 / 데이터: 갭, 채용 리드타임, 협력사 단가·역량 / 방식: 부족분을 Move·Build·Borrow로 분배해 옵션 생성.
- **마진 시뮬 Agent**(L3) — 언제: 소싱안의 수익성을 따질 때 / 데이터: 인력 단가, 고객 청구단가, 기간·인원 / 방식: 원가·매출·마진을 옵션별 계산·비교.
- **가용성·롤오프 모듈**(L6) — 언제: 운영 중 수급 위험을 감지할 때 / 데이터: 현 배정·롤오프 일자·가동률 / 방식: 롤오프 임박·과소가동을 규칙으로 탐지.
- **품질 검증 정책 / Antbot**(L6, ★) — 매칭 적합도·예산 위반·누락 직무 점검.
- **단가·승인 정책**(L7) — 단가 상한·마진 하한·외주 계약을 자동 점검, 위반 시 HITL 보류.
- **지식 자산화 Agent / Self-Improving**(L7, ★) — 배치 결과·PM 수정을 학습 → 다음 프로젝트 추천 개선.
- 기존 솔루션 카드: HR 시스템 · RMS · ERP(Integrate) / Connector: 인사·프로젝트·스킬 DB·외주 포털.

## 10. PLATFORM_CORE ↔ DOMAIN_PACK 경계 (표준화 핵심 산출)

**PACK(교체)**: `domain meta`(이름·아이콘·고객 화면 타이틀) · 온톨로지 · `WORK`(스테이지별 label/role/ops/explain/detail) · `COMPOSE` · `COMPONENTS` · `DLV` · 더미 데이터셋 · 고객 콘솔 렌더(head/workspace) · 선택 옵션(`SOURCING_OPTIONS` vs `TIMES`).
