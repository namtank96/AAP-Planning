## 1. 한 줄 요약

금융 내부통제 사건(예: "승인권한 초과 가능성") 한 건이 **AAP(Agentic AI Platform) 위에서 접수→해석→추론→사람 검토(HITL)→실행→감사·학습**으로 처리되는 과정을, **에이전트 오케스트레이션이 실제로 작동하는 모습**으로 보여주는 데모를 만들고 있다.

현재 가장 적절하다고 판단되는 기준선은 **v0.5 (Agent OS 워크플로우)** 이며, 이를 기준으로 고도화 방향을 검토받고자 한다.

---

## 2. PoC의 목적과 제약

### 2.1 목적
- AAP의 개념·아키텍처·차별점을 **고객/임원이 직관적으로 체감**하게 한다.
- "기존 내부통제 시스템에 AI를 붙이는 것"과 "**AAP 기반 의사결정 운영**"의 차이를 보여준다.
- 단순 챗봇/추천이 아니라, **여러 전문 에이전트가 오케스트레이션되어 판단을 만들고 사람이 통제하는** 구조를 시연한다.

### 2.2 제약 (의도적)
- 단일 HTML 파일, 브라우저에서 바로 실행. 외부 라이브러리·LLM·실제 시스템 연동 없음.
- **결정론 시뮬레이션 엔진**: 위험 점수·위험도·조치 후보는 **하드코딩하지 않고 입력값에서 계산**.
- 화면에 "Simulation PoC" 명시.

### 2.3 1차 시나리오 (고정)
> 정기 점검에서 **승인권한 초과 가능성** 이슈 발생 (여신심사본부, IC-2026-001).
> 승인금액 5,800만원 / 전결한도 5,000만원, 필요 승인권자=임원 / 실제 승인권자=팀장, 위임 없음, 증빙 완전성 45%, 유사사례 4건, 미완료 개선과제 1건.

## 5. 현재 기준선: v0.5 상세

### 5.1 구성
- **좌측 — Agent OS 실행 아키텍처 맵** (`agent 워크플로우 예시` 그대로 매핑)
  - User(Task↓/Solution↑), Guardrails, Response 개인화
  - Orchestration Layer → Planner(작업 분해) + Orchestrator(에이전트 배분) + Integration Layer(Agent↔Agent / Agent↔System / Agent↔Human·HITL) · State&Memory
  - 좌측 Enterprise System(내부통제 시스템·규정 DB·책무 시스템·점검이력 DB) ↔ 우측 Agent Marketplace(규정매핑·책무연결·유사사례·위험평가·조치제안 에이전트)
- **우측 — 에이전트 실행 스트림** (호르무즈 방식)
  - "워크플로우 실행" → 13단계 순차 수행(스피너→✓), 진행바, 단계별로 "무엇을 했는지" 업무 언어로 표시
  - 실행에 맞춰 좌측 아키텍처 맵의 관여 요소 점등
  - **위험평가 = Agent↔Agent 협업**(규정·책무·사례 결과 종합), **HITL 단계에서 멈춰 사람이 승인/수정/반려**, 이후 출력 가드레일 → Solution → State&Memory 학습

### 5.2 결정론 엔진 (하드코딩 아님)
- 입력 9개(승인금액, 전결한도, 필요/실제 승인권자, 위임 여부, 증빙 완전성, 반복 횟수, 미완료 개선과제, 유사사례 수)
- **Rule Score 60% + Similar Case Score 40% = 위반 가능성(%)** → 위험도(높음≥70 / 보통≥40 / 낮음) → 검토 수준 → 조치 후보
- 1차 시나리오 입력 → **위반 가능성 77%, 높음, 준법 검토**로 계산됨
- (v0.3에는 3개 수용 테스트가 코드에 내장되어 통과 확인됨)

## 6. 알려진 이슈 & 고도화 방안 (검토 포인트)

### 6.1 알려진 이슈
- **좌-우 템포 불일치**: 좌측 아키텍처 맵이 너무 빨리 다 점등되어 "이미 끝난 것"처럼 보이는 반면, 우측 스트림은 계속 진행됨.

### 6.2 고도화 방안 (제안)

| # | 방안 | 내용 | 우선순위(제안) |
|---|---|---|---|
| A | **좌-우 동기화 정교화** | 아키텍처 노드를 해당 단계 실행 중에만 펄스(active), 완료 시 done. Enterprise System→에이전트→Orchestrator로 **데이터 흐름 연결선/토큰 애니메이션**. 우측 스텝과 1:1 동기화 | P0 (필수 보정) |
| B | **Planner 계획 인라인 표시** | Planner 단계에서 5개 하위작업 체크리스트 생성 → 각 에이전트 완료 시 체크 (v0.5 레이아웃 유지) | P1 |
| C | **에이전트 추론 깊이** | 각 에이전트 스텝 클릭 시 펼쳐 "무엇을 조회→무엇을 찾음" 상세 + Agent↔Agent 결과 수신 메시지 시각화 (호르무즈 수준) | P1 |
| D | **지식그래프 탭** | 사건↔규정↔책무↔유사사례↔개선과제 온톨로지 별도 탭 시각화 | P2 |
| E | **다건 라우팅 + 학습 메모리** | 큐의 여러 사건을 Agent OS가 자동 분류(저위험 자동/고위험 HITL), State&Memory가 실행마다 Decision Log로 축적 | P2 |

## 7. 외부 전문가께 드리는 검토 질문

1. **방향성**: "Agent OS 아키텍처 맵 + 에이전트 실행 스트림"(v0.5) 구도가 AAP의 에이전트 오케스트레이션을 보여주는 데 적절한가? 더 나은 표현 방식이 있는가?
2. **에이전트 구성**: 5개 전문 에이전트(규정매핑/책무연결/유사사례/위험평가/조치제안) 분할이 내부통제 업무 관점에서 타당한가? Planner/Orchestrator/Integration Layer 매핑은 적절한가?
3. **추론의 신뢰성 표현**: 결정론 엔진(Rule 60%+Case 40%)으로 위험 점수를 산출하는 방식이, "AI가 왜 그렇게 판단했는지" 설득에 충분한가? 실제 제품에서는 어떤 근거 제시가 필요한가?
4. **HITL 위상**: 사람 검토 지점(승인/수정/반려)을 워크플로우 중간에 두는 현재 방식이 금융 내부통제 거버넌스 관점에서 적절한가?
5. **고도화 우선순위**: 6.2의 A~E 중 고객/임원 설득력 관점에서 우선순위는?
6. **기존 시스템과의 관계**: "기존 내부통제 시스템 위에 AAP 의사결정 계층을 얹는다"는 메시지가 명확히 전달되는가?
