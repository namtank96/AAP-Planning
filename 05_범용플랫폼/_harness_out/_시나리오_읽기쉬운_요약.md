# ⚖️ A vs B 비교

## 변수가 바꾼 것 (발산)
| 측면 | A사 (제조·SAP·온프레미스·보수) | B사 (IT·SaaS·클라우드·중도) |
|---|---|---|
| 슬롯 출처 | SAP·SharePoint·자체구축 | Notion·Google Drive·DocuSign |
| Connector | ERP 대조축(SAP read) | 전자서명·제재 수기보강 |
| 결재선 | 금액 4구간 위임전결 | 담당→대표 2단 |
| 위험 임계 | 보수(composite≥30·배상×2·위약>20%) | 중도(금액 5M/30M·배상×3·위약>10%) |
| gap 종류 | 템플릿·평판 | **제재 외부소스 부재** |
| 회색지대 | 적극 법무검토(R8·R9·R11) | 대표 결재로 흡수 |
| 결과 | fail(3회) | **pass(2회)** |

## 변하지 않은 것 (재사용 골격 = sameCore)
1. 아키타입: 접수→추출→결정론 룰 1차판정→HITL 게이트→**route 3분기**→결재선→대장 등록
2. decide 구조: 파생 boolean·금액·신뢰도 → 가중 위험점수/등급 → route 상위우선 평가
3. route 3대 종착: **자동승인 / 법무검토(HITL) / 반려**
4. gap 슬롯 = 자동승인 누출 차단(미충족 gap은 자동승인 도달 불가)
5. 0 vs null 명시 구분 규약(무금액·무제한·미확인 구분)
6. 제재·신용 결격 = route 최우선 가로채기

## 결론
> **"고객 변수에 따라 시나리오가 바뀌면서도 핵심 골격은 재사용된다"** — 입증. *무엇을 어디서 읽고 어떤 임계로 누구에게 보내는가*는 프로파일 데이터로 갈리지만, *어떻게 판정하고 어떤 종착으로 분기하는가*라는 실행 골격은 그대로.
> ※ 위 A/B critic 결과: 1차엔 A가 fail이었으나, **N1b에서 critic에 severity(blocking/minor)를 넣자 A·B 둘 다 무수정 pass** — 진짜 문제는 시나리오가 아니라 critic의 과엄격 이분법이었음([[aap_harness_poc_result_v0_1]] §9).

## ⚖️ 도메인 비교 (계약 · 구매 · 경비)

**변하지 않은 골격 (토큰 단위 공유)**
1. 6단계 파이프라인: 접수→슬롯추출→결정론 룰 1차판정→HITL 게이트→route 3분기→결재선→writeback
2. decide 2계층 (룰 1차판정 + 암묵지 사람 보완)
3. route 3분기 + 서열: 하드 가드레일 반려 → HITL 검토 승급 → 자동승인 fallback (위험 결격 최우선 가로채기)
4. gap↔게이트 1:1 / 임계 패밀리 5종(금액캡·결재선룩업·신뢰도컷 0.8·패턴탐지·소진초과) / 결재선 위임전결표 / writeback

> **재사용 레시피**: 새 도메인 = ①슬롯 매핑 ②knowledge 4종 ③route 사유표 ④writeback 대상만 교체. 골격(파이프라인·decide 2계층·route 서열·gap↔게이트·임계 패밀리)은 고정. → **같은 하니스에 knowledge·슬롯만 갈아끼우면 도메인이 무수정 1패스로 찍혀 나온다 = 범용 시나리오 공장.**
