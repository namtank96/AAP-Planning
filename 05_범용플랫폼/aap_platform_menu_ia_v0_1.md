# AAP 플랫폼 메뉴·IA 기획 v0.1 (260619)

> KEY MESSAGE에서 도출한 사용자 메뉴 구조 + 핵심 플로우. 구현 전 기획 확정본.
> 출처 결정: 유저 — 그룹형 메뉴(운영/스튜디오/관리), 자산은 스튜디오 아래, '실행' 메뉴 삭제.

## 0. 출발점 — KEY MESSAGE → 플랫폼의 두 루프
"업무를 **이해**하고 → 실행 **구조를 구성**하며 → Agent·모듈·기존솔루션·Connector·정책을 **조합**해 → 운영 가능한 업무 **흐름을 만들고 운영**한다."

```
[구성 루프]  새 업무/프로세스 → 분해 → 5타입 조합 → 워크플로우 편집·시각화 → HITL 배치 → 격상(=도메인 팩 등록)
                                        ↕ (유형 흐름 없으면 구성으로, 있으면 운영으로)  ← ★현재 빠진 연결
[운영 루프]  인박스(들어온 일) → 케이스 실행(그 유형 흐름) → HITL → 산출물 → 로그
```

## 1. 메뉴 IA (그룹형 · 확정)

```
운영 (Operate)
  └ 인박스(Work Inbox)            모든 유형 업무 유입 → 열면 그 유형 흐름으로 실행
스튜디오 (Build / Compose)
  ├ 워크플로우 · 도메인 팩         업무 유형별 AAP 흐름: 목록 · 편집 · 시각화(팔란티어식) · 신규 격상
  └ 자산(Assets)                 Agent · Module · 기존솔루션 · Connector · 정책 = 조합 재료 레지스트리
관리 (Govern)
  ├ 로그 · 트레이스               Run Trace · Decision Log · Writeback 감사
  └ 거버넌스 · 정책               권한 · 가드레일 · 평가
(하단) 시연                       가이드 투어 (랜딩 ✕, 별도 진입)
```
- **'실행' 메뉴 삭제**: 실행 = 인박스에서 케이스 열기. 별도 1급 메뉴 불필요.
- 랜딩 = 인박스. 시연은 하단 별도.

## 2. 핵심 개념 정리 (유저 질문 답)
- **도메인 팩 = 업무 격상 = 워크플로우** — 셋은 같은 것. '업무 격상' 파이프라인(분해→구성→HITL→등록)의 **산출물이 곧 도메인 팩(=운영 가능한 워크플로우)**. 그래서 둘을 **'워크플로우·도메인 팩' 스튜디오** 한 메뉴로 통합하고, '신규 격상'은 그 안의 생성 액션. (현재 상단 '업무 격상' 버튼 → 이 메뉴로 흡수.)
- **자산(5타입)** = 조합 '재료'. 스튜디오 아래 둠 — 워크플로우 편집 중 바로 끌어다 조합. (운영 자산 카탈로그·거버넌스 관점은 관리에서 참조만.)
- **팔란티어식 워크플로우 편집/시각화** = 스튜디오 '워크플로우' 안의 **시각 편집기** — 단계·5타입 블록·HITL 게이트·데이터 흐름 노드 그래프를 *편집 가능*하게. (현재 디자인 가이드에만 존재 → 실제 기능화.)
- **관리** = 로그·트레이스 + 거버넌스·정책. (자산은 스튜디오로 이동.)

## 3. 빠진 연결 — 인박스 On-Ramp (정형화·근거 애매 해소)
- **현 문제**: 인박스가 미리 만든 팩의 케이스만 보여줘 정형적. 새 업무를 즉시 쪼개 격상하는 과정이 없음. AAP 작동 근거가 맥락 없이 떠 애매.
- **해결**: 인박스 '새 업무'가 →
  - 해당 **유형 흐름(도메인 팩) 있으면 → 케이스 생성·실행**(운영 루프).
  - **없으면(신규/비정형) → 스튜디오 격상으로 자동 유도**(분해→구성→HITL→등록) 후 운영.
  - → 두 루프 연결, 인박스가 '격상의 입구'도 됨.
- **근거 노출**: 사용자가 스튜디오에서 직접 보고 구성한 워크플로우가 있으니, 실행 시 근거 레일이 "이 단계 = 이 블록들이 이 데이터로"로 **워크플로우 시각화와 연동** → 맥락 생김.

## 4. 구현 순서(제안)
1. **메뉴 IA 재편**(골격): gnav 그룹형(운영/스튜디오/관리)·'실행' 제거·상단 '업무 격상'→스튜디오로. 도메인 뷰→스튜디오 '워크플로우·도메인 팩'으로 개명·재배치.
2. **관리 실질화**: 로그·트레이스 + 거버넌스. (자산은 스튜디오로.)
3. **자산 메뉴**: 5타입 레지스트리(현 Component Registry 확장).
4. **워크플로우 편집기/시각화**(팔란티어식): 스튜디오 안 — 단계·블록·HITL·데이터 흐름 편집. (디자인 가이드의 노드 그래프 실제 기능화.)
5. **인박스 On-Ramp**: 새 업무 → 흐름 있으면 실행 / 없으면 격상 파이프라인. 격상 산출 = 도메인 팩.
6. 각 단계 골격(aap-platform) → 시각(aap-design). 검증·커밋 누적.
