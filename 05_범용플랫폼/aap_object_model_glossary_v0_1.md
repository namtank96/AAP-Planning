# AAP 객체 모델 · 용어 사전 (Object Model & Glossary) v0.1

> **목적**: 플랫폼의 핵심 객체(플랫폼/도메인 팩/프로젝트/케이스)와 자산·로그·거버넌스의 **스코프(전역 ↔ 케이스)**를 못 박는다.
> 발단 = "로그·트레이스가 지금은 특정 케이스 것만 보인다. 자산·로그 등을 (1) 전반 차원과 (2) 케이스별로 둘 다 관리해야 한다. 그 전에 용어부터 — 프로젝트? 케이스? 스튜디오?" (유저, 260622)
> **이 문서가 용어의 단일 출처(SSOT)**. 코드/스펙/핸드오프는 여기 용어를 따른다.

## 0. 유저 확정 (260622, AskUserQuestion)

| 결정 | 선택 |
|---|---|
| 업무 인스턴스 단위 | **프로젝트 + 케이스 (2계층)** |
| 스튜디오 범위 | **정의 + 케이스별 튜닝** |
| 관리(로그·자산·거버넌스) 뷰 기본 스코프 | **글로벌 기본 + 케이스 슬라이스** (상호 이동) |

## 1. 4계층 객체 모델

> 규칙 한 줄: **정의(definition)는 위층에, 실행 증거(evidence)는 아래층에.**

| 층 | 용어 | 정체 | 사는 곳(뷰) | 비고 |
|---|---|---|---|---|
| **L0** | **플랫폼** (Platform) | AAP 런타임 전체. 전역 레지스트리·전역 정책·전역 관측의 컨테이너 | 상단 셸 + 관리 뷰(글로벌) | 단수. "이 플랫폼(HTML) 자체가 고객이 보는 화면" |
| **L1** | **도메인 팩 = 워크플로우** (Domain Pack / Workflow) | "이 종류 업무는 이렇게 돈다"는 **재사용 정의**: surface + 단계(work) + 구성요소(components) + 조합(compose) + 정책(govern) | **스튜디오** | 채용·회의·VOC. 1 팩 = 1 워크플로우 정의 |
| **L2** | **프로젝트** (Project) | 한 도메인 팩의 **여러 케이스를 묶는 운영 컨테이너**. 기간·목표·소유자를 가짐 | 운영(인박스 그룹) | 예: "2026 상반기 채용". 기본적으로 1 팩에 귀속 |
| **L3** | **케이스** (Case) | 그 워크플로우를 **구체적 건 하나에 실행한 인스턴스** | 운영(인박스) → **Run** | 예: "백엔드 김OO". 실행 증거(trace)의 단위 |

```
L0 플랫폼
 └ L1 도메인 팩(워크플로우)  ← 스튜디오에서 '짓는다'
     └ L2 프로젝트            ← 케이스를 묶는 컨테이너
         └ L3 케이스          ← 인박스에서 생기고 Run에서 '돌아간다'
```

### 1.1 프로젝트 ↔ 케이스 관계 규칙
- **케이스는 항상 1개 도메인 팩에 귀속**(어떤 워크플로우로 도는지 결정). 현행 `case.packId`가 이 역할.
- **프로젝트는 0..1개** — 케이스는 프로젝트에 속할 수도(채용처럼 묶음 자연스러움), 단독일 수도(손해사정 1건, VOC 1건) 있다. → **프로젝트는 선택적 그룹 계층**(없으면 인박스에 케이스가 평면으로).
- **프로젝트는 기본 단일 팩** — "2026 상반기 채용" = 채용 팩의 케이스 묶음. (교차-팩 프로젝트는 v0.1 범위 밖, 후속.)
- 데이터: `case.projectId?`(옵션) · `project.packId`(귀속 팩) · `project.caseIds[]`(파생).

## 2. 운영 vs 스튜디오 — '돌린다' vs '짓는다'

| | **운영** (Operate) | **스튜디오** (Studio) |
|---|---|---|
| 다루는 객체 | 프로젝트·케이스 (L2/L3 인스턴스) | 도메인 팩·워크플로우 (L1 정의) + **케이스별 튜닝** |
| 동사 | 케이스를 **돌린다** | 워크플로우를 **짓는다 / 고친다** |
| 뷰 | 인박스 → Run | 스튜디오(워크플로우·도메인 팩, 자산) |

### 2.1 스튜디오 = 정의 + 케이스별 튜닝 (유저 확정)
스튜디오는 두 가지를 한다 — **경계가 흐려지지 않게 "오버라이드 레이어" 모델로 분리**:

1. **정의 저작(기본)** — 도메인 팩(워크플로우) 자체를 편집. 영향 = 그 팩의 **모든 케이스**. (현행 워크플로우 편집기 + `packOverrides`.)
2. **케이스별 튜닝(예외)** — 특정 케이스 1건에만 적용하는 **델타(override)**. 영향 = **그 케이스만**. 화면에 항상 "⚑ 이 케이스 한정 변경"으로 표식, 베이스 정의는 불변.

> **충돌 방지 원칙**: 케이스 튜닝은 **베이스 정의 위에 얹는 델타**일 뿐, 절대 베이스를 바꾸지 않는다. "이 변경을 워크플로우 전체에 반영" = 명시적 승격(델타 → 정의)으로만. (현행 `snapshotPack/restoreBase` → 케이스 스코프 override 로 확장.)

스코프 적용 우선순위: **플랫폼 기본 → 도메인 팩 정의 → 프로젝트 override(후속) → 케이스 override**.

## 3. 자산·로그·거버넌스 — 스코프 정의 (핵심)

> 유저 지적: 지금은 전부 "지금 열린 팩/케이스" 슬라이스만 보인다(전역 차원 부재).
> 규칙: **관리·스튜디오 메뉴 = 글로벌(정의/집계) 기본. 케이스를 열면 그 케이스 슬라이스가 Run 안에 붙는다. 둘은 상호 이동.**

### 3.1 자산 (Assets)
| 스코프 | 내용 | 위치 |
|---|---|---|
| 전역(플랫폼) | **자산 카탈로그** — 전 5타입(Agent·Module·기존솔루션·Connector·Policy) 재료 풀 + Master | 스튜디오 · 자산 (글로벌) |
| 도메인 팩 | 그 워크플로우가 **채택한** 자산 부분집합 | 스튜디오(팩 선택 시 필터) |
| 케이스 | 이 케이스 실행에서 **실제 발화한** 구성요소 | Run 우측 근거 레일 |
| (이동) | 케이스→"이 자산의 전역 정의 보기" / 전역→"이 자산을 쓰는 케이스" | 양방향 링크 |

> 현행 `자산` 뷰 = `PACK.components`(활성 팩) → **전역 카탈로그(전 팩 union)로 승격** + 팩/케이스 필터 추가가 과제.

### 3.2 로그·트레이스 (Logs · Trace)
| 스코프 | 내용 | 위치 |
|---|---|---|
| 전역(플랫폼) | **관측/집계** — 런 수·HITL율·가드레일 차단·도메인별·기간별 (**신규**) | 관리 · 로그·트레이스 (글로벌) |
| 프로젝트 | 그 프로젝트 케이스들의 집계 롤업 | 글로벌 뷰의 프로젝트 필터 |
| 케이스 | 이 케이스 한 건의 **Run Trace · Decision Log** (현행 `STATE.trace`) | Run + 글로벌에서 드릴다운 |
| (이동) | 글로벌 집계 → 케이스 드릴다운 / 케이스 → 글로벌 맥락 | 양방향 |

> 현행 `로그·트레이스` 뷰 = `STATE.trace`(활성 케이스) → **전역 집계가 기본, 케이스 trace는 드릴다운**으로 재배치가 과제.

### 3.3 거버넌스·정책 (Governance · Policy)
| 스코프 | 내용 | 위치 |
|---|---|---|
| 전역(플랫폼) | 전역 RBAC · 가드레일 · 정책 정의 | 관리 · 거버넌스 (글로벌) |
| 도메인 팩 | 그 워크플로우에 적용되는 정책(현행 `PACK.govern`) | 스튜디오/거버넌스(팩 필터) |
| 케이스 | 이 케이스에서 **평가·차단된** 정책(증거) | Run 근거 레일 |

## 4. 현행 코드 ↔ 모델 매핑 (구현 과제 식별용)

| 모델 개념 | 현행 코드 | 현 스코프 | 목표 스코프 |
|---|---|---|---|
| 도메인 팩 | `PACKS[id]` / `PACK` | L1 ✅ | 유지 |
| 케이스 | `APP.cases[]` / `case.packId` | L3 ✅ | 유지 |
| **프로젝트** | — | 없음 | **신규 L2**(`project`, `case.projectId?`) |
| 자산 | `PACK.components` → `renderAssets` | 팩 한정 | **전역 카탈로그 + 팩/케이스 필터** |
| 로그·트레이스 | `STATE.trace` → `renderTrace` | 케이스 한정 | **전역 집계 기본 + 케이스 드릴다운** |
| 거버넌스 | `PACK.govern` → `renderGovern` | 팩 한정 | **전역 정의 + 팩/케이스 슬라이스** |
| 케이스별 튜닝 | `packOverrides`(팩 단위) | 팩 전체 | **케이스 스코프 override 레이어 추가** |

## 5. 구현 로드맵 (제안 · 후속 합의)

- **P1 — 용어 정합**: 코드/UI 라벨을 본 사전에 맞춤(케이스/프로젝트/도메인 팩 용어 통일). 신규 hex 0.
- **P2 — 자산 전역화**: `renderAssets`를 전 팩 union 카탈로그 + 팩/케이스 필터로. Run 근거 레일과 "전역 정의 보기" 링크.
- **P3 — 로그 전역화**: 로그 뷰 = 전역 집계(런/HITL/차단 KPI + 케이스 목록) 기본, 케이스 trace는 드릴다운. `STATE.trace`는 케이스 단위로 보존(누적 store 추가).
- **P4 — 프로젝트 계층**: `project` 객체 + 인박스 프로젝트 그룹핑(선택적) + 케이스 묶음. 채용 팩에 먼저 적용.
- **P5 — 케이스별 튜닝**: `packOverrides` → 케이스 스코프 override 레이어("⚑ 이 케이스 한정 변경" + 정의 승격 경로).

> 각 단계 코어=도메인 무관 유지, file:// 동작, meeting/voc/recruiting·X1·영속 무회귀 원칙 준수.
