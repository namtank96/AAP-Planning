## 0. 플랫폼 한 장 요약

```
[Domain Pack: 회의 ▾]   [구성 | 실행 | 관리]   [▶ Run]      ← 플랫폼 셸(도메인 무관)
────────────────────────────────────────────────
구성(Design)  = AAP가 업무를 이해하고 "실행 구조"를 어떻게 짜는지   (설계 시점)
실행(Run)     = 그 구조가 자율 작동 — 좌 고객 서비스 화면 / 우 8계층 흐름  (런타임)
관리(Govern)  = 구성요소 레지스트리 + 거버넌스(Policy·Trace·Eval·Skill)  (운영)
```

플랫폼 = **{셸 + 런타임 문법 + 구성요소 타입 체계 + 8계층 + HITL 모델 + 거버넌스}** (도메인 무관 불변)
도메인 = **Domain Pack** (회의·손해사정·내부통제… 교체) 만 갈아끼움.

## ★ 평가 렌즈 3 + 통합 모델 (모든 고도화의 고정 기준)

> **통합 모델**: `AAP Platform = 〔표준 런타임 = AAP Operating Loop × 8계층 × 거버넌스〕 + 〔Domain Pack(회의)〕가 그 위에서 자율 운영.`
> 즉 **AAP 아키텍처가 곧 플랫폼 엔진이고, 회의는 그 위에 꽂힌 첫 Pack**(Reference Runtime Instance). "회의 8단계 시연"이 아니라 "엔진 + 첫 팩"으로 본다.

모든 고도화는 아래 3렌즈로 평가한다(외부 전문가 의견 채택):

| 렌즈 | 묻는 것 | 충족 근거(이 스펙) |
|---|---|---|
| **(1) 아키텍처 정합** | 표준 런타임 = AAP 아키텍처(Operating Loop·8계층·4대 역량·Trust)인가 | §4 8계층 + §5 런타임 문법(추론 루프 매핑) + 정합 뷰(§9-B) |
| **(2) 확장·표준화** | 런타임/팩 분리로 2번째 도메인이 같은 코어에 꽂히나 | §1 불변/팩 분리 + §8 Domain Pack 인터페이스 |
| **(3) 자율 운영** | 맥락분석→구조 구성→수행+HITL→거버넌스·학습을 자율 수행하나 | §3 구성요소 조합 + §5 자율 원칙 + §6 HITL + §7 거버넌스 |

> 4대 역량(업무 의미화·시스템 반영·통제·권한·운영 학습)은 (1) 정합의 라벨 축으로 명시한다.

## 2. 플랫폼 셸 — 3뷰

| 뷰 | 역할(렌즈) | 핵심 구성 | 불변/팩 |
|---|---|---|---|
| **구성 Design** | 구조(설계 시점) | Workload Analysis(업무 이해) → Execution Structure(구성요소 조합 hero) → Execution Plan(단계 표) → HITL 게이트 바 | 레이아웃=불변 / 내용=팩 |
| **실행 Run** | 작동(런타임) | 좌: 업무 순서(직전/현재/다음) + **고객이 보는 서비스 화면**(헤더+콘솔) / 우: **8계층 흐름**(단계별 op 점등·∥병렬·자산 보라) | 셸=불변 / surface·op=팩 |
| **관리 Govern** | 거버넌스·운영 | Platform Control strip(Policy·Run Trace·Evaluation·Skill Library) + Master Agent 카드 + **Component Registry** | strip=불변 / 컴포넌트=팩 |

상단바(불변): `Domain Pack 셀렉터` · `구성/실행/관리 토글` · `▶ Run` · `개발자 보기`.

> **위 3뷰는 도메인 팩 한 건의 작동 렌즈**(구성/실행/관리). 플랫폼 전체 운영 메뉴는 아래 §2.1 IA v0.2가 정본 — 3뷰는 스튜디오 '도메인 팩' 안에서 한 팩을 들여다보는 모드로 흡수된다.

### 2.1 운영 콘솔 메뉴 IA v0.2 (← `aap_platform_menu_ia_v0_1.md` 흡수 · 확정)

KEY MESSAGE("업무를 이해→실행 구조 구성→5타입 조합→운영 가능한 흐름을 만들고 운영")를 화면 구조로 증명하는 **그룹형 3그룹 + 시연**:

```
운영 (Operate)
  └ 인박스(Work Inbox)        배포된 도메인 팩의 업무 유입 → 열면 그 유형 흐름으로 실행(On-Ramp). 랜딩 = 인박스
스튜디오 (Build / Compose)
  ├ 자산(Assets)             5타입 블록(Agent·Module·기존솔루션·Connector·Policy) 타입별 세분 · 편집 · 배포(draft→active)
  ├ 워크플로우(Workflow)      흐름 빌더 — ①직접 비주얼(로우/노코드) ②AI 격상. 팔란티어식 노드 그래프 · 배포
  └ 도메인 팩(Domain Pack)    업무 유형 패키지(surface·데이터/온톨로지·산출물·연결 워크플로우·케이스) 구성요소별 관리 · 배포→인박스
관리 (Govern)
  ├ 로그·트레이스             Run Trace · Decision Log · Writeback 감사
  └ 거버넌스·정책             권한(RBAC) · 가드레일 · 평가(§7)
(하단) 시연 (Demo)            가이드 투어(랜딩 ✕, 별도 진입)
```

- **3층 위계(부분⊂전체)**: `자산(재료: 5타입 블록) → [조합] → 워크플로우(흐름: 단계·게이트·데이터흐름) → [패키징] → 도메인 팩(업무 유형 전체)`. **도메인 팩 ⊃ 워크플로우 ⊃ (자산 조합)**. v0.1에서 '워크플로우 = 도메인 팩'으로 합친 것은 부분과 전체를 뭉뚱그린 오류 → 정정.
- **'실행' 1급 메뉴 삭제**: 실행 = 인박스에서 케이스 열기.
- **인박스 On-Ramp(빠진 연결)**: 인박스 '새 업무' → 유형 흐름(도메인 팩) **있으면 케이스 생성·실행**(운영 루프) / **없으면(신규·비정형) 스튜디오 격상으로 자동 유도**(분해→구성→HITL→등록) 후 운영. 두 루프를 잇고, 인박스가 '격상의 입구'도 됨.
- **근거 노출**: 사용자가 스튜디오에서 직접 구성한 워크플로우가 있으므로, 실행 시 근거 레일이 "이 단계 = 이 블록들이 이 데이터로"로 워크플로우 시각화와 연동 → 맥락 생김.

### 2.2 배포 생애주기 (불변 spine · 전 층 적용)

자산·워크플로우·도메인 팩 모두 **`draft → deployed`** 생애주기를 가진다.

- **배포된 것만(deployed)** 운영(인박스)·조합·거버넌스 대상에 편입된다. draft는 스튜디오 안에서만 편집 가능.
- 배포·정책 권한 = 승인자·관리자(§7 RBAC와 연결).
- 이로써 "재료를 조합해 운영 가능한 업무 흐름을 만든다"의 '운영 가능한'이 상태로 강제된다 — 미배포 산출물은 운영에 끼지 못함.

## 3. 구성요소 타입 체계 (플랫폼 어휘) — "Agent만이 아니다"

플랫폼은 업무를 **5개 타입 구성요소 + 1 오케스트레이터**의 조합으로 만든다. (확보 방식 Build·Buy·Borrow와 매핑)

| 타입 | 의미 | 확보 방식 | 회의 Pack 예시 | 계층 |
|---|---|---|---|---|
| **Master Agent** (Supervisor) | 업무 맥락 분석→실행 구조 구성·지휘, HITL 창구 | Build | Master Agent | L3 |
| **Agent** | 전문 작업 단위 | Build | 참석자·자료수집(RAG)·일정·자료패키지·안건·회의록 | L3·L5 |
| **Module** | 재사용 결정론 기능 | Build/Buy | OCR·정규화·일정충돌검증·민감도분류·**AI:ON-U(AI 서빙)** | L5·L6·L3 |
| **기존 솔루션** | 보유·상용 자산 연동 | **Buy·Integrate** | 그룹웨어 캘린더·사내 KMS·문서함 | L5·L8 |
| **Connector** | 시스템 안전 연결 | Build/Buy | 메일·문서함·조직도 DB·캘린더 API | L5 |
| **Policy** | 통제·권한·게이트 | Build | 외부 공유 정책·PII·DLP·실행 승인(HITL)·품질·근거 검증 | L7·L6 |
| **kt ds 자산** | BEAST(Connector·게이트웨이)·Antbot(기존 솔루션·RPA)·AI:ON-U(Module·AI 서빙) | Buy/Build | AI·API 라우팅 / 시스템 반영·발송 자동화 / 모델 추론 | L5·L8·L3 |

> **메시지**: 모든 걸 Agent로 만들지 않는다 — 결정론은 Module, 보유 자산은 기존 솔루션, 연동은 Connector, 통제는 Policy. **선택·조합이 플랫폼의 일.**

## 4. 8계층 아키텍처 (도메인 무관)

`L1 경험·접근 · L2 설계·개발 · L3 에이전트 코어 · L4 지식·시맨틱 · L5 데이터·연동 · L6 AI 품질·평가 · L7 컨트롤·거버넌스 · L8 인프라`

- **kt ds 자산(보라)**: **BEAST**=AI·API 게이트웨이(모델·외부 API 라우팅) · **Antbot**=RPA(시스템 반영·발송 자동화) · **AI:ON-U**=AI 서빙(모델 추론). ⚠ 정정(v0.24): Antbot은 품질평가가 아니라 **RPA**, AI:ON-U는 OCR이 아니라 **AI 서빙**, Self-Improving은 명명 자산에서 제외. 범례 표기 필수. (자산명·역할은 갱신 가능)
- 실행 뷰 우측이 단계별로 어느 계층의 어느 구성요소가 점등되는지 보여줌(추론 루프 Data·Semantic·Reasoning·Decision·Action·Learning과 정합).

### 4.1 L4 지식·시맨틱 = 에이전트 접근 의미 레이어 (1급화 · ★차별화)

L4는 단순 '지식 저장소'가 아니라 **에이전트가 데이터에 접근하는 통제된 의미 레이어**다. KEY MESSAGE의 차별화 축이므로 1급으로 명시한다:

- **원칙**: LLM/에이전트는 **온톨로지(객체·관계·Action)를 통해서만** 데이터에 접근한다(통제된 reasoning, 블랙박스 직접 접근 ✕). 전시 캡션 ✕ — 구조로 읽히게.
- **구성**: 도메인 팩의 `ontology`(객체=노드 / 관계=엣지 / Action=객체 편집 배지[자동 green·사람확인 amber])가 L4 인스턴스. 코어가 그래프로 렌더(`renderOntology`). 스키마·렌더 표준 = **`aap_domain_pack_spec_v0_2.md` §4** + 디자인 가이드 그래프 시각 표준.
- **결정론/비결정론 경계가 여기서 갈림**: Agent(LLM, 비결정론)는 L4 의미 레이어를 거쳐 추론하고, Module·Connector·Policy(결정론)는 규칙·온톨로지 편집으로 작동(`blockKind` det/llm, 도메인팩 §3.1).
- **(Phase 3 연결)** 결정 레이어(`caseModel` 슬롯 = L4 온톨로지 슬롯)도 L4에 인스턴스화 — 단 구현 보류(도메인팩 §7).

## 5. 표준 런타임 문법 (8단계 캐논 + 운영 객체)

플랫폼의 실행 흐름. 단계 라벨·내용은 Pack이 채우되 **흐름 형태·actor·게이트·산출물 슬롯은 불변.**

| # | 단계(일반형) | actor | 산출 운영 객체(선제안서 4-3) | 게이트 |
|---|---|---|---|---|
| 1 | 요청 접수 | human | Work Event(요청 신호) | |
| 2 | 업무 이해 | aap | Work Event 확정 + 정책 플래그 | |
| 3 | **실행 구조 구성** | aap | Task Graph + 구성요소 조합(Execution Plan) | |
| 4 | 계획 승인 | **hitl ①** | Human Review Decision + Execution Package | ★ |
| 5 | 준비 실행 | aap (∥병렬) | Candidate→Action Result | |
| 6 | 진행 통제 | **hitl ②** | Raw Record→실시간 산출물 | ★ |
| 7 | 최종 승인 | **hitl ③** | 발송·기록 승인 | ★ |
| 8 | 공유·학습 | aap (∥병렬) | Action Result + **Skill Update** | |

- **각 단계 op 스키마**: `{g(병렬 그룹), feed(작업), out(중간 산출물), L(계층), comp(구성요소), micro[](하위 호출 예: calendar.create()), asset/badge(kt ds)}`.
- **자율 원칙**: 비-게이트 단계는 AAP 자율 실행, 사람은 HITL 3곳만(§6). 'AAP에게 맡기기' 같은 per-step 수동 버튼 ✕.
- **변경 흡수 슬롯**(옵션): 진행 중 요청 변경 시 영향 단계만 ∥재실행(데모 `change` 단계 — 플랫폼 일반 기능으로 승격 가능).

## 6. HITL 게이트 모델 (책임 지점만 멈춤)

| 게이트 | 트리거(불변) | 회의 Pack 적용 |
|---|---|---|
| **① 계획 승인** | 실행 전 자원·구성 결정 | 계획 승인 + 회의 시각 선택 |
| **② 진행 통제** | 실행 시점 통제 | 회의 시작/종료 신호 |
| **③ 최종 실행** | 외부·되돌리기 어려운 행동 직전 | 발송·KMS 기록 승인 |

원칙: **자동/확인/차단** 3분기 → 책임 걸린 것만 차단 후 게이트, 나머지 자율. 결정은 후속 단계로 전파(예: 외부 제외 → 발송 op '제외됨').

## 7. 거버넌스·운영 (Platform Control · 도메인 무관 상시)

| 항목 | 역할 |
|---|---|
| **Policy** | 외부 공유·접근 권한·실행 승인. 책임 지점 자동 차단 후 담당자 확인 |
| **Run Trace** | 입력→근거→Tool 호출→사람 확인→반영 전 구간 기록(감사·재현) |
| **Evaluation** | 근거 충족도·정책 위반·실행 성공률 평가 |
| **Skill Library** | 검증된 패턴·기준을 재사용 자산으로 축적(Self-Improving 피드백 루프) |

> 아래 §7.1~§7.4 = 관리 그룹(§2.1) '거버넌스·정책'·'로그·트레이스'를 **운영 콘솔**로 상세화. 대상 = **배포된(deployed) 자산/워크플로우/팩만**(§2.2 생애주기 연결). 디자인 표준 = 디자인 가이드 §6.6.4(Eval·RBAC·Agent Ops).

### 7.1 Policy = RBAC 권한 매트릭스 (역할 × 권한)

책임 지점 자동 차단 + **역할 기반 권한**을 매트릭스로 통제. 허용=체크 / 차단=dash.

| 권한 \ 역할 | 담당자 | 검토자 | 승인자 | 관리자 |
|---|:--:|:--:|:--:|:--:|
| 케이스 실행·HITL 확인 | ✓ | ✓ | ✓ | ✓ |
| 워크플로우 편집(draft) | — | ✓ | ✓ | ✓ |
| **배포(draft→deployed)** | — | — | ✓ | ✓ |
| 정책·가드레일 변경 | — | — | ✓ | ✓ |
| RBAC·계정 관리 | — | — | — | ✓ |

- 가드레일(외부 공유·PII·DLP·실행 승인)은 책임 지점에서 **자동 차단 후** 권한 보유자에게 HITL 게이트로 보류.
- 배포·정책 권한은 승인자·관리자만(§2.2 생애주기와 정합).

### 7.2 Evaluation = KPI 지표 + 계산식

근거·정책·성공·적시성을 정량 지표로 평가(종합점수 상태색: green≥85·amber≥60·red<60).

| KPI | 정의 | 계산식(개념) |
|---|---|---|
| **근거 충족도** | 판정이 근거(온톨로지·기준)에 기반한 비율 | 근거 부착 verdict / 전체 verdict |
| **정책 준수율** | 가드레일 위반 없이 처리된 비율 | (전체 − 위반) / 전체 |
| **실행 성공률** | HITL·반영까지 정상 종결 비율 | 성공 종결 케이스 / 전체 케이스 |
| **HITL 적시성** | 게이트가 SLA 내 처리된 비율 | SLA 내 결정 게이트 / 전체 게이트 |

### 7.3 Agent Ops = 성능·비용·품질 모니터링

배포된 구성요소(Agent·Module…)의 운영 지표 카드: **성능**(처리시간·처리량) · **비용**(호출·토큰) · **품질**(근거 충족·재현 일관성·편향 점검 통과). Datadog/Agent Ops 톤.

### 7.4 Run Trace 기록 항목 (감사·재현)

전 구간 기록: `입력 → 근거(온톨로지·기준 행) → Tool/Connector 호출 → blockKind(결정론/LLM) → 사람 확인(HITL 결정) → Writeback 반영`. (Phase 3 결정 레이어 시 verdict `{ruleId,inputs,output,basis}` 부착 → "왜 이 결론?"에 규칙+데이터로 답.) Decision Log = AAP 자동판단 / 담당자 액션 / 시스템 반영 / Replay 누적.

## 8. Domain Pack 인터페이스 (플랫폼화의 핵심)

> **정본 계약 = `aap_domain_pack_spec_v0_2.md`(SSOT)**. 아래는 요약 — 필드·타입·실태 매핑은 SSOT를 따른다.

새 도메인 = 아래 묶음만 제공하면 같은 셸·런타임·거버넌스로 작동:

```
DomainPack = {
  id, label,                         // 'meeting' / '회의'
  workload,                          // 요청·유형·목적·필요 산출물·확인 지점

```

회의 Pack은 완성 상태(검증된 Reference Instance). **claim/control 등 산업 Pack은 실제 착수 시 채움(가짜 placeholder ✕).**

## 9. 고도화 로드맵 (3렌즈 갭 기반 · 외부 2단계 제안을 플랫폼 관점에서 재배열)

**v0.21 갭 점검** (현행 실제 기준 — 외부 의견은 구버전 참고라 일부는 이미 반영됨):

| 렌즈 | v0.21 현재 | 실제 갭 |
|---|---|---|
| (1) 정합 | op에 L1~L8 태그·explain에 추론 루프 **이미 있음** | Operating Loop·4대 역량·Trust를 **검증 가능한 정합 뷰**로 명시 안 됨 |
| (2) 표준화 | §1·§8 분리 *정의*는 있으나 **코드는 회의 전용 inline** | `PLATFORM_CORE`/`DOMAIN_PACK` 코드 분리·스위처 없음 → 2번째 팩 못 꽂음 |
| (3) 자율 | compose·HITL 3·자율 run **있음** | 구성=사전정의·거버넌스=정적 카드 → **라이브 판단/Run Trace 증거** 부족 |

**권장 순서 — A를 B보다 먼저:**
- **A. 데이터 경계 분리 (저비용·먼저)** — 단일 HTML 안에서 `PLATFORM_CORE`(Loop·8계층·거버넌스·런타임 문법·HITL) ↔ `DOMAIN_PACK.meeting`(workload·compose·WORK·components·products)로 객체 분리. *멀티파일/스위처는 아직 ✕.* → 안 그러면 B 기능이 회의 결합 코드에 박혀 분리 시 재작업.
- **B. 정합 + 자율 라이브 (코어 엔진, pack 데이터를 읽음)** — ① **정합 뷰**: Loop 6단계·4대 역량·Trust 밴드(손해사정 v0.56 탭2 `renderArchMap` 차용, 현재 run이 그 위를 trace) ② **라이브 Run Trace/Decision Log**: 자율 판단·HITL·Writeback이 실행되며 누적 = 거버넌스 증거(정적 카드 → 라이브) ③ **동적 compose 연출**: 작업 분해 → **구성요소 5타입(Agent·Module·기존솔루션·Connector·Policy) 선택·조합**을 그 자리에서(키 메시지 first-class).
- **C. 확장 입증** — Domain Pack 스위처 작동 + 2번째 팩(손해사정 일부)로 **교체 1회 입증**. 단일 HTML 한계 시 localhost·멀티파일.
- **승격** — 검증된 셸 스펙·구성요소 타입·런타임 문법을 본 문서 다음 버전으로 확정.

> 외부 제안 Phase 1(B)·Phase 2(A·C)의 **가치는 전부 채택**하되 **A(데이터 경계)를 B 앞**으로 — premature 멀티파일 인프라는 C로 미뤄 over-engineering 회피. 작업은 D_회의, 확정 셸 스펙만 본 폴더(05)로 승격.
