# 채용 플로우 — 단계별 "보여줄 스토리" v0.1 (BD·고객·루프검증용)

> **왜 이 문서**: 데모의 목적 = *프로세스를 보여주는 것*(실사용 속도 ✕). 청중 = ①영업 BD 설득 ②고객 설명 ③AAP 자가생성 루프 검증. "요구사항 넣으니 바로 됨"은 매직처럼 보여 설득 실패 → **분해·의미이해·조합·추론 과정을 단계마다 드러내야** "매직 아님"이 증명된다. 빌드 전, 단계별 *무엇을 증명/보여줄지* 를 여기서 못박는다.

---

## 0. 가로지르는 원칙 (전 단계 공통)

- **워크스루 네비**: 자동재생만 ✕ → **이전 / 다음 / 처음부터**. BD가 한 단계씩 짚으며 설명. (사용자 실사용에선 빠르게 흐르지만, 데모는 멈춰서 설명 가능해야)
- **"이건 AAP가 생성한 워크플로우" 프레임**: 이 9단계는 *AAP가 "백엔드 2명 충원해줘" 한 줄에서 분해·구성한 결과*임을 명시 = 자가생성 루프의 시각화.
- **진짜 vs 연출 (정직 기준)**:
  - **진짜**: 결정 판정(`evaluate`)=결정론·재현 / 슬롯·룰 구조=실제 caseModel·knowledge / 요구사항→워크플로우 생성=**하니스(Claude Code 루프)가 4도메인 자가검증까지 실증**.
  - **연출**: 앱 안 "새 요청→즉시 생성"·추출 신뢰도 수치·학습 반영. (실제 생성기는 오프라인 하니스 — file:// 앱엔 라이브 미연결)
  - 화면에 "연출"이라 쓰진 않되, **BD 화법 = "결정·재현은 진짜, 생성 루프는 하니스로 검증됨"**. 금칙: "AI가 알아서 다 함"·"사람 볼 필요 없음".
- **단계마다 3답**: ①AAP가 한 일 ②왜(근거) ③사람이 볼/결정할 것. (매직이 아니라 *추론*으로 보이게)

## 1. 단계별 스토리 (9단계)

| # | 단계 | **BD·고객에게 증명할 것 (한 줄 메시지)** | 화면 — 보여줄 시각화 | 진짜/연출 |
|---|---|---|---|---|
| 1 | **요건 접수** | "AAP는 막연한 한 줄을 받아 *무엇이 정해졌고 무엇을 더 정해야 하는지* 안다" | 원문 요청 → 추출 신호(직무·인원·연차·스택) + **미정 구분**(우대·컷·패널·밴드) | 추출=연출 / 슬롯구조=진짜 |
| 2 | **요건 분석** ★ | "키워드 매칭이 아니라 *의미(온톨로지)*로 이해하고, 평가기준·통제를 함께 단다" | **★온톨로지 그래프** — 요청→표준 직무모델 매핑(백엔드 0.93 등) + 매칭 가중(50/30/20) 정의 + 차별금지·PII 정책 부착(L7) | 신뢰도=연출 / L4 시맨틱·L7 거버넌스 구조=진짜 |
| 3 | **스크리닝 설계** ★ | "AAP 본질 = Agent 많이 띄우기 ✕ → 업무를 *분해*하고 Agent·Module·기존ATS·Connector·Policy를 골라 *조합*" | **★구성요소 조합 도식** — 작업분해(T1파싱~T5취합) → 조합(Agent6·Module3·ATS2·Connector4·Policy3) + 편향·누락 점검(Antbot) | 카탈로그=연출 / "분해→선택→조합" 명제=진짜(form spec) |
| 4 | **숏리스트 기준 승인** ★ | "자동 결정·반려가 아니라 *책임 분기를 사람에게* 넘긴다. *사람 의도(컷·가중)가 결과를 바꾼다* — 정적 화면 ✕" | 후보 랭킹+컷라인 + **가중 steering(바꾸면 재계산)** + 통과 미리보기 / 모달=책임 지점 | **★evaluate 결정론·재계산=진짜** |
| 5 | **이력서 수집·스크리닝** | "사람이 정한 기준대로 AAP가 *실행*하고, *통제*(편향·PII)를 지키며 시스템에 반영" | 파싱·랭킹·통과/보류/탈락 분기 + 시스템 반영(스코어카드·ATS write) | screenVerdict=evaluate 경로(진짜) |
| 6 | **면접 조율** | "*외부(후보)로 나가는 행동* 전엔 사람이 확인한 뒤에만" | 캘린더 교차·충돌0 슬롯 + 일정·패널 확인 모달 | 일정=연출 / HITL 게이트=진짜 |
| 7 | **면접 평가 취합** | "면접관 평가를 모아 정합성 점검하고 오퍼안을 정리" | 스코어카드 취합 + 보상밴드·차별금지 사전 점검 | 연출 |
| 8 | **합격·오퍼 승인** | "후보에게 오퍼 나가는 *마지막 행동*, 사람이 최종 승인" | 스코어카드+오퍼 패키지 + 점검(밴드·헤드카운트·차별금지) 모달 | HITL 게이트=진짜 |
| 9 | **기록·학습** ★ | "끝내고 잊지 않는다 — *패턴을 학습해 다음을 개선*" (자가생성 루프와 연결) | 발송·기록 반영 + 학습 자산화 + "이 흐름은 AAP가 생성" 회수 | 학습=연출 / **하니스 자가검증 루프=진짜(별도 실증)** |

★ = 이번에 새로 시각화 빌드할 핵심 (2 온톨로지 · 3 조합 · 4 steering은 이미 있음 → 강화).

## 2. 빌드 (스토리 합의 후 — ✅ 완료 260624, 회의 데모 자산 재사용)
1. ✅ **워크스루 네비**(↺처음/이전/▶재생/다음) — runtop `#runNav`·`renderRunNav`/`walkNav`(코어, 도메인 무관). 이전/다음=자동재생 멈추고 그 단계 작업만 재생 후 정지. 검증: restart→1/9·next×2→3/9·prev→2/9.
2. ✅ **요건 분석 온톨로지 그래프**(②) — `opOntologyGraph`(인라인 SVG: 요청→백엔드0.93/플랫폼0.05/풀스택0.02, 엣지굵기=신뢰도)+`opOntologyAside`. opKind 'analyze' 분리. opMatchGraph 패턴 차용.
3. ✅ **스크리닝 설계 조합 도식**(③) — `opComposeMain`(업무분해 T1~T5 + 5타입 조합 Agent6·Module3·기존솔루션2·Connector4·Policy3, 타입색 토큰)+`opComposeAside`. opKind 'compose' 분리. 회의 데모 composeNode 차용.
4. ✅ **"AAP가 생성" 프레임**(④) — intake/record 배너(`op-genframe`): "이 9단계는 AAP가 한 줄 요청에서 분해·구성한 결과"/"학습 자산화".
- 검증(헤드리스): 4단계 시각화 캡처 OK + 전 도메인 무회귀(proc op·contract verdict·meeting/voc 렌더·nav 전역) + 채용 9단계 전부 op-console.
- 남은(후속): 4·5단계(게이트·스크리닝) 카피를 "추론으로 보이게" 다듬기 / 진짜 vs 연출 BD 화법 가이드 별도.

## 3. 합의가 필요한 것 (이 문서 확정 전)
- 단계별 한 줄 메시지가 BD·고객에게 맞나(특히 2·3·4·9).
- 온톨로지 그래프를 어디까지(직무 매핑만 vs 슬롯 추출까지).
- "생성됨" 프레임을 *얼마나 정직하게*(연출임을 BD에겐 구분, 화면엔 표시 ✕).
- 회의 데모(`aap_meeting_runtime_builder`)의 compose·온톨로지 시각화 자산을 재사용할지.
