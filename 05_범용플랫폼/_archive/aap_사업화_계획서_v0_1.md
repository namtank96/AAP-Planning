# AAP 사업화 계획서 v0.1

선제안서(개념)와 데모(작동) 위에 시장, 고객, 경쟁, 가격, 역량, 성장을 더한 사업 레이어입니다. 수치는 모두 (추정)이며 파일럿 실측 전까지 확정하지 않습니다. 관련 문서는 제품(`aap_runtime_expansion_strategy_v0_1.md`), 공학(`aap_서비스화_로드맵_v0_1.md`), 개념(`aap_proactive_offering_v0.7`)입니다.

---

## 1. 사업 정의

- **정의**: 검증에 머문 AI를 판단, 승인, 시스템 반영, 학습까지 잇는 운영형 플랫폼.
- **차별화**: Agent를 많이 띄우지 않고, 업무에 맞는 구성요소(Agent, 모듈, 기존 솔루션, Connector, 정책)를 필요한 만큼 조합하고 통제하고 기록.
- **kt ds 강점**: KT그룹 레퍼런스 기반, 폐쇄망·온프레미스 구축 경험, 자산(BEAST, Antbot, AI:ON-U) 보유, 결정론과 감사로 한국 규제 적합.

---

## 2. 시장

### 2.1 제품 정의
통제 가능한 업무 운영을 판매합니다. 분석과 대화에서 멈추던 AI를 판단, 승인, 반영, 학습까지 확장합니다. 비교 대안은 글로벌 운영형 AI 플랫폼, 국내 SI 플랫폼, 자체 구축, 수작업과 RPA입니다.

### 2.2 시장 규모 (추정)

| 구분 | 정의 | 추정 | 근거 |
|---|---|---|---|
| 거점 | KT그룹 계열과 1차 타깃 | 30~50개 계정 | 계열사, 규제 산업 대형 고객 |
| 유효 | 근거·승인·감사 요구가 높은 백오피스 보유 | 500~800개 계정 | 금융, 보험, 공공, 제조 상위군 |
| 전체 | 한국 기업 운영형 AI | 외부 자료 필요 | 미검증 |

절대 규모가 아니라 운영 전환에 막힌 백오피스 계정을 단계적으로 점유합니다. 1차 목표는 레퍼런스 운영 사례 3~5건입니다.

### 2.3 시점
- 2026년 화두는 검증을 넘어 운영 배포이며, 경쟁사 공통 방향입니다.
- 규제 강화(내부통제, AI 거버넌스 인증, 망분리)가 한국 적합성의 기회입니다.
- 모델 평준화로 경쟁축이 의미화, 통제, 운영으로 이동합니다.

## 3. 고객

### 3.1 고객 프로파일

| | A. 자체 AI 대기업 | B. 산업 특화 중견 | C. 규제·폐쇄망 |
|---|---|---|---|
| 산업 | 통신, 제조, 지주 | 유통, 보험, 제조 | 공공, 은행, 방산 |
| 페르소나 | 데이터·AI 추진 임원 | 업무 본부장 | 준법·감사·정보보안 책임자 |
| 통증 | 분석이 업무로 안 이어짐 | 사람 못 늘리는데 물량 증가 | 근거·감사·망분리 미확보 시 보류 |
| 진입 팩 | 회의, 보고, 승인 | 고객 응대, 품질, 청구 | 내부통제, 민원, 여신 심사 |
| 난이도 | 중간 | 낮음 | 높음, 계약 수명 길고 이탈 적음 |

### 3.2 거점 시장 선정
- 기준: 관계 레버리지 > 통증 적합 > 레퍼런스 가치.
- 이미 거래가 있는 계정부터 진입합니다. 고객 획득 비용이 낮고 성사율이 높습니다. 관계 없는 시장은 후속입니다.

| 순위 | 대상 | 레버리지 | 진입 팩 |
|---|---|---|---|
| 1 | KT그룹 계열사 | 데이터 접근·조달 장벽 최소, 6개 도메인 하니스 공급 준비 | 회의, 내부통제, 구매, 경비 |
| 1 | kt ds 기존 구축 고객 (중견 제조·유통·공공) | 신뢰·시스템 이해·접점 보유 | 구축 업무 인접 팩 |
| 2 | 금융 내부통제 (등대) | 1차 검증 과제 확정, 규제 적합 | 내부통제 콘솔 |
| 3 | 신규 산업 | 등대 레퍼런스 확보 후 | 후속 |

첫 과제는 kt ds·KT그룹 기존 계정 목록을 점수화하여 상위 3~5개를 1차 타깃으로 확정하는 것입니다.

## 5. 역량

### 5.1 소싱 원칙
- 확실한 자체 개발은 에이전트 코어(L3)입니다.
- 지식·시맨틱(L4)은 가장 중요하지만 현재 자체 역량이 부족합니다. 육성과 외부 제휴로 확보해야 할 최대 과제입니다.
- 나머지 계층은 부분 자체 개발과 도입, 제휴로 커버합니다.

### 5.2 chip별 소싱 (잠정)
> 아래 분류는 **잠정**입니다. 자사 자산 명세가 충분치 않아 소개서 검토로 확정해야 합니다. HTML에서는 각 칩을 눌러 소싱(자체 / 도입 / 제휴 / 미정)을 직접 지정하고 저장할 수 있게 만듭니다.

| 계층 | 주요 역량 (chip) | 소싱 (잠정) |
|---|---|---|
| L1 경험·접근 | Agent Portal, Copilot/Chat UI, HITL Console, Approval Queue, Audit View | 부분 자체 + 도입 |
| L2 설계·개발 | Agent Studio, Process Designer, Tool Registry, Agent Registry, Policy Designer | 부분 자체 |
| L3 에이전트 실행환경 | Supervisor, Task Routing, Agent Coordination(A2A), Model Gateway, Tool Calling, Memory/Context, HITL Runtime Gate, Runtime Guardrail | 자체 개발 (핵심), Model Gateway는 BEAST |
| L4 지식·시맨틱 | Domain Ontology, Policy/Rule Model, Knowledge Graph, Context Assembly, Evidence/Citation, RAG | 확보 과제 (약점), 육성·제휴 |
| L5 데이터·연동 | DB·API·SaaS Connector, OCR, Parser, Vector Index, Writeback | 도입·제휴, Writeback은 Antbot |
| L6 AI 품질·평가 | Decision Log, Execution Trace, Quality Evaluation, RAI/Safety, User Feedback | 도입 + 일부 자체 |
| L7 컨트롤·거버넌스 | Policy Management, RBAC/ABAC, HITL Approval Criteria, Approval Matrix, Self-Improving, Release Gate | 부분 자체 + 도입 |
| L8 인프라 | Compute/GPU, Object·Vector Storage, Network·Firewall, Secret Manager, Private Cloud, Closed Network | 제휴 (KT그룹), Model Serving은 AI:ON-U |

확실한 자체 개발은 L3입니다. L4는 확보 과제이고, 나머지는 부분 자체와 도입·제휴로 채웁니다.

## 6. 시장 진입

### 6.1 진입과 확장
1팩 검증 과제로 진입한 뒤 인접 팩을 더하고(한계비용 하락) 플랫폼 구독으로 전환합니다. 코어는 한 번 구축하고 팩을 늘립니다.

### 6.2 검증 과제 제안 (예시 · 보강 필요)
> 아래 수치는 **더미 예시**입니다. 선제안서의 오퍼링 패키지(적용 진단, 통제된 파일럿, 구축, 운영과 개선)를 기준으로 실데이터로 보강해야 합니다.

| 항목 | 내용 (더미) |
|---|---|
| 기간 | 8~12주 |
| 산출물 | 운영 가능한 팩 1개, 의사결정 로그, 처리 시간·자동화율 측정 |
| 진입 조건 | 데이터 접근 1종, 업무 전문가 1명, 책임 승인자 1명 |
| 가격 | 미확정. 그룹 내부 원가 기준, 외부 구축 단가 기준 |

서비스 임계선은 영속 저장, 실 Connector 1종, 권한 관리 셋입니다(최소 실서비스 조건).

### 6.3 채널
1차 KT그룹 직접 판매, 2차 kt ds 기존 고객 구축 영업, 3차 산업 파트너. 셀프서비스 판매는 후속입니다.

### 6.4 수행 모델
선제안서의 Assess~Operate 5단계를 따릅니다. 검증 과제는 이 다섯 단계를 8~12주로 압축한 것입니다.

| 단계 | 하는 일 | 주요 산출물 |
|---|---|---|
| Assess (적용 진단) | 업무·시스템·보안 조건으로 적용 후보·우선순위 선별 | 적용 후보 명세서, ROI 가설서, 도입 제약 진단서 |
| Define (업무 기준 정의) | 업무 의미·정책·권한·예외를 실행 구조로 정리 | 업무 의미 모델, 정책·권한 매트릭스, HITL 분기 기준서, KPI 정의서 |
| Compose (자산 조합) | 검증된 자산 조합, 화면·통제·보안 결합 | 에이전트 구성도, 역량 확보 명세, 검증 시나리오 |
| Connect (고객 환경 연계) | 고객 자산·시스템을 실행 흐름에 연결 | 시스템 연계 설계서, IAM·SSO 통합 설계, 연계 검증 보고 |
| Operate (운영 전환·학습) | 운영 기준·개선 흐름 설계, 재사용 자산 축적 | 운영 전환 기준서, 의사결정 로그 구조, 재사용 스킬 카탈로그 |

**R&R (SI 기준)** — 단계별 책임 주체를 분리합니다.

| 단계 | 고객 | kt ds (주관) | 파트너 |
|---|---|---|---|
| Assess | 업무·제약 정보 제공 | 진단, 우선순위 선정 | — |
| Define | 업무 지식 확인, 기준 승인 | 의미·정책·HITL 설계 | — |
| Compose | 검증 참여 | 자산 조합, 구성 설계 | 특화 자산 제공 |
| Connect | 시스템 접근, IAM 협조 | 연계 설계, 통합 | Connector 구현 |
| Operate | 운영 주체, 승인 | 운영 기준, 개선 지원 | 유지보수 |

상용은 적용 진단, 통제된 파일럿, 구축, 운영과 개선의 네 묶음으로 패키지화합니다.

kt ds 팀 기능의 하나로 **메타 AAP** 도구를 운용합니다. 고객 업무 설명을 넣으면 AAP 적용 시나리오를 자동 생성하여, Assess와 Compose를 가속하고 고객에게 미리 보여 줍니다. 6개 도메인 하니스가 그 엔진이며, 자동 생성물은 출발점이고 정확성은 전문가 검수가 필요합니다.

| 지식·시맨틱(L4) 약점 | 가장 중요한 계층의 자체 역량 부족 | 육성 범위 확정, 외부 제휴, 단계적 확보 |
| 단위 경제 미확정 | 매출·구축비·기간 가정 | 파일럿 3건 실측 후 고정 |
| 조직·인력 | 영업·구축·컨설팅 인력 계획 부재 | 2단계 진입 전 조직 설계 |

핵심 가정(검증 대상): KT그룹이 1차 레퍼런스로 열림, 운영 전환 통증이 구매로 이어짐, 팩 한계비용 급감, 규제 적합성이 실질 우위.
