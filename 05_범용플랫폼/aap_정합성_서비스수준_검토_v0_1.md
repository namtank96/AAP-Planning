# AAP 전체 정합성 · 서비스 수준 · 목적 부합성 검토 v0.1

> **무엇**: 결정 런타임·하니스 트랙과 app/(스튜디오·결정층 통합) 전체의 **정합성**, **서비스화 수준**, **목적 부합성**을 검토. 유저 요청(260624): "도메인팩·워크플로우·자산도 app writing 충돌? 전체 정합성 검토 + 서비스 차원 수준 올리기 위해 필요한 점 + 의도한 목적과 부합 여부."
> **방법**: 병렬 read-only 검토 2건(app/ 스튜디오+결정층 통합 / 문서 세트 정합성). **app/ 미수정.**

---

## 0. 충돌 범위 (질문 답)

| 작업 | app writing 충돌? |
|---|---|
| 도메인팩·워크플로우·자산 화면 **수정/구현** | ✅ 충돌 (app/core.js·wfeditor.js = 다른 세션 단일 writer) |
| 화면·결정층 **읽고 정합성 검토** | ✕ 무관 |
| 검토 결과를 **설계·로드맵 문서**로 | ✕ 무관 (본 검토가 그것) |
| 게이트 통과본 등 **데이터 산출물 공급** | ✕ 무관 |

→ **검토·설계·데이터 공급은 다 가능.** 화면 구현만 핸드오프.

## 1. 정합성 검토

### 1.1 잘 맞는 것 ✅
- **핸드오프 작동**: 다른 세션이 N3을 받아 **결정층을 app에 랜딩** — `app/core/evaluate.js`(window.AAP_EVALUATE) + `contract_a.js`(caseModel 23슬롯·knowledge.route 10룰·io.editable·seeds) + `evalCase()` 런타임 호출(케이스 open·임계 변경 시 재평가). = 다리 2(evaluate) 실작동.
- **용어 95% 일치**: 4계층(플랫폼/도메인팩/프로젝트/케이스)·결정층·8계층·5타입·route 3분기 전 문서 일관.
- **아키텍처 무모순**: 결정런타임(B)·Pack v2(flow/io)·N3·하니스가 충돌 없이 맞물림(검토 4/5).

## 2. 서비스 수준 갭 (PoC/데모 → 실서비스)

| 갭 | 심각도 | 영향 |
|---|---|---|
| **영속 저장**(케이스·trace store) | 🔴 HIGH | file:// 세션 한정 → 감사·재현·집계 불가 |
| **실 Connector**(OCR/STT/ERP/DB) | 🔴 HIGH | 전부 mock → 실데이터 유입·writeback 없음 |
| **멀티테넌트 격리** | 🔴 HIGH | 단일 테넌트 → "범용 플랫폼"(다고객) 미달 |
| **RBAC enforcement** | 🔴 HIGH | 정적 매트릭스만, 실권한 강제 없음 |
| 프로젝트 관리·케이스별 튜닝(P5) | 🟡 MED | 묶음 운영·예외 튜닝 UI 미구현 |
| 전역 자산 카탈로그·lineage | 🟡 MED | 재사용·발견·거버넌스 약함(현 팩 한정) |
| 관측·알림·SLA KPI | 🟡 MED | 운영 가시성 부재 |
| 변형(variant) 선택 UI | 🟡 MED | A사/B사 발산 증명됐으나 고객이 고를 길 없음 |
| 실 저작 API(Phase 3) | 🟢 LOW | 하니스가 오프라인 저작으로 대체 중 |

> **문서가 갭을 정직하게 명시**(P1~P5·"후속"·"mock"). "데모 기준" 프레이밍과 정합 — 단 **"범용 플랫폼" 주장엔 🔴 4종이 블로커**.

## 3. 목적 부합성

| 목적 | 부합도 | 평 |
|---|---|---|
| **(A) 회의 데모 기준 범용 플랫폼** | ◐ 70% | 8계층·5타입·flow 일반화로 **도메인 무관 구조는 증명**. 그러나 회의만 live, 계약·구매·경비는 baked/spec — **실 멀티도메인 운영은 미완**. 트랙이 (B) 증명에 70% 치우침 |
| **(B) harness/loop engineering·결정 시나리오** | ● 95% | **완전 달성**. 하니스 v0.2가 4도메인 무수정 1패스·게이트 통과·2중 자기검증(prose+실행). 블랙박스 0 |

→ 둘은 **상보**(A=프로세스/B=기술증명)이고 모순 없음. **부족분 = (A) 쪽**: 결정층을 스튜디오에 노출하고(C1), 게이트통과본을 실제 팩으로 배포하고(C4), 멀티도메인을 인박스에 띄우는 것. 이게 메워지면 "범용 플랫폼"이 데모 너머로 선다.

## 4. 조치

### 4.1 이 트랙 (app/ 무관 — 지금 가능)
- **A1. C5 수정**: 계약 P1 설계의 `decide()` → `evaluate()` 정합(canonical 명시). ✅ 본 검토와 함께.
- **A2. C4 공급**: `packs_baked_v2/pack_*.v2.json`(게이트통과·expectedOutcome·R6B)을 **app contract 팩 교체본**으로 핸드오프(README 명시).
- **A3. 스튜디오 결정-노출 설계**(C1) + **flow `next` 바인딩 설계**(C2): 화면 구현은 app 세션이지만 **설계·데이터 계약은 이 트랙이 스펙화** → 스튜디오 로드맵(⑥)·N3에 반영.
- **A4. 서비스 수준 로드맵**: §2 갭을 P-단계로 정리(영속→Connector→멀티테넌트→RBAC)해 후속 합의 자료화.

### 4.2 app/ 세션 핸드오프
- C1 스튜디오 결정 탭(도메인팩에 route/슬롯/임계·자산에 knowledge·워크플로우에 분기 시각화) / C2 flow `next`=verdict.outcome / C3 팩 스키마 마이그레이션·가드 / C4 contract 팩을 게이트통과본으로 / C6 프로젝트 거버넌스 바인딩.

## 5. 종합
- **정합성**: 문서 4/5·app 6.5/10. 핸드오프 작동(결정층 랜딩)이 큰 성과. **최대 갭 = 스튜디오가 결정층을 못 보여줌(C1)·흐름 분기 미연결(C2)·app 팩 구버전(C4).**
- **서비스화**: 핵심 런타임·하니스는 증명됐고, 블로커(영속·Connector·멀티테넌트·RBAC)는 정직히 후속으로 명시됨. "데모 기준"엔 충분, "범용 플랫폼 운영"엔 🔴 4종 필요.
- **목적**: (B) 완전, (A) 구조는 증명·운영은 미완. **결정층 노출+게이트통과본 배포+멀티도메인 인박스**가 (A)를 데모 너머로 끌어올리는 다음 레버.
