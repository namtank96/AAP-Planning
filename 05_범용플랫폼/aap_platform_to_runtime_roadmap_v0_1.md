# 데모 → 플랫폼 전환 로드맵 v0.1

> **배경**: `app/`(범용 플랫폼 구현체)은 회의 데모 `D_회의/aap_meeting_runtime_builder_v0_26.html`에서 **추출**돼 구조가 거의 동일하다. 본질이 **"각본 재생 뷰어"**이지 **"운영하는 플랫폼"**이 아니다. 이 문서는 데모 → 실제 플랫폼 형태로 가는 작업·개념·순서를 확정한다. (260619, 계획 확정 / 구현은 다음 세션)
> **결정**: 최종 실행성 목표 = **준-실제 작동(localhost + 실제 LLM + 모의 커넥터)**. 시작은 Phase 0+1.

---

## 1. 진단 — 왜 지금은 "플랫폼"이 아니라 "데모"인가 (코드 근거)

| # | 한계 | 근거 |
|---|---|---|
| 1 | **업무가 1건뿐** — 팩마다 하드코딩된 단일 케이스 재생, 여러 실제 업무 개념 없음 | `loadPack`이 `WORK[0]` 고정 시작, case=팩 더미 |
| 2 | **실제 실행이 없음** — `setTimeout`으로 미리 써둔 결과 문자열(`o.out`) 연출만 | `revealOps()` 640ms 타이머 텍스트 노출 |
| 3 | **사용자가 구조를 못 짬** — "Builder"인데 구성요소 읽기 전용 전시 | `renderDesign()`이 `COMPOSE` 데이터 그리기만 |
| 4 | **상태가 안 남음** — 새로고침 시 소멸, 결정·기록 비영속 | `STATE`=메모리 변수, 저장 없음 |

→ "AAP가 일하는 모습을 보여주는 영화" / 플랫폼 = "실제 업무를 받아 구조를 짜고 굴리고 통제하는 도구".

## 2. 개념 — 데모 → 플랫폼 3축 (+ 토대)

```
[토대] 상태 영속성 — 새로고침해도 남는다 (localStorage → 후에 DB)
 ① 인스턴스성   여러 실제 업무 건을 받아 동시에 굴린다     "영화 재생" → "인박스·케이스"
 ② 구성성       사용자가 실행 구조를 직접 짠다(Builder)   "전시" → "조립"
 ③ 실행성       각본 → 진짜(또는 준-진짜) 실행            "setTimeout" → "엔진·LLM·커넥터"
```

순서 근거: **① → ② → ③.** 구조를 짜도(②) 굴릴 업무가 없으면(①) 추상적, 진짜 실행(③)도 받을 업무·짜인 구조가 선행돼야 의미.

## 3. 작업 Phase (트랙 표시)

### Phase 0 — 토대: 운영 콘솔 셸 + 영속성  〔platform 골격 → design〕
- 3뷰 토글 → **운영 콘솔 IA** (좌측 글로벌 네비: 인박스 / 빌더 / 실행 / 관리 / 도메인)
- `STATE` localStorage 영속 (케이스·결정·trace 잔존)
- *이게 "앱"의 뼈대 — 이후 전부 여기 얹힘*

### Phase 1 — ① 인스턴스성: 업무 인박스 & 케이스  〔platform 기능 + design〕
- **업무 인박스**: 들어온 요청 여러 건, 상태별(접수/진행/검토대기/완료) 목록
- **케이스 생성**: "새 업무 요청" → 인스턴스 생성 → 8단계 런타임 투입
- 단일 하드코딩 case → case 객체 배열(각자 독립 진행 상태)
- design: 인박스·케이스 상세 레이아웃(Linear/Datadog 톤)
- *데모 아님을 가장 크게 만드는 단계 — **시작점***

### Phase 1.5 — 통합 인박스 (도메인 = 유형 속성으로 강등)  〔platform + design〕 ★260619 재설계
- **유저 지적**: 도메인 팩 셀렉터로 앱 전체를 갈아타는 "모드 전환"은 데모 잔재. 나뉜 이유는 시나리오별 포맷팅(surfaceSpec/WORK/compose). 실제 플랫폼 = **하나의 인박스에 여러 유형 업무가 섞여 들어옴.**
- **도메인 = 화면을 가르는 모드 ✕ → 각 업무 건의 유형(type) 속성 ✓**
- 통합 인박스: 모든 케이스 한 곳 + 유형 배지·유형 필터. 상태 × 유형 2축. 케이스 열기 = 그 케이스 packId로 런타임 컨텍스트 로드(팩=뒷단 처리기로 강등).
- "도메인" 네비 → 모드 스위치 ✕ → **업무 유형 카탈로그/레지스트리**(처리 가능한 유형)로 재정의.
- 효과: "하나의 런타임, 여러 유형 동시" = 플랫폼 명제를 데모보다 강하게 증명.

### Phase 2 — ② 구성성: 프로세스 → 자율운영 격상 파이프라인  〔platform + design〕 ★260619 재정의
- **유저 요구**: '업무 학습'(지식 축적) 외에, **AI가 업무 프로세스를 분해하고 실제 자율 운영 가능 수준으로 격상(HITL 필수 포함)**시키는 작업/플로우.
- 수동 빌더 ✕ → **AI가 제안하고 사람이 검토·수정·승인해 운영으로 올리는 파이프라인**(수동 편집은 폴백). '업무 학습'이 이 파이프라인의 입구.
- 5단계 (AAP 4대 역량 정합):
  ① **이해·분해(Breakdown)** 프로세스 → 단계·작업·데이터·결정점·리스크 (업무 의미화)
  ② **실행 구조 구성(Compose)** 각 작업 → 5타입 구성요소 매핑 (실행 구조)
  ③ **HITL 배치(Gate)** 책임·되돌리기 어려운·외부 영향 지점에 인간 게이트 강제 — **★필수, 없으면 격상 불가** (통제·권한)
  ④ **격상·검증(Operationalize)** 구성+게이트 → 8단계 캐논 런타임 적재 + Guardrail/Policy로 자율 안전성
  ⑤ **시운전→승격(Promote)** 샘플 케이스 dry-run → 사람 검토 → "운영 중" 승격 (운영 학습)
- **분해 엔진 = 지금은 결정론 시뮬**(genericAuthor 고도화, 규칙·템플릿 기반, file:// 유지) → 실제 LLM은 Phase 3에서 그 자리 교체. X1 디자인 계약 적용·같은 Domain Pack 스키마로 출력.
- design: 파이프라인 전용 플로우 UX(분해→구성→게이트→검증 단계 시각), 노드 그래프(오케 v2 차용·5타입 색).

### Phase 3 — ③ 실행성(진짜화 · 준-실제 작동 확정)  〔platform〕
- 결정론 엔진: 더미 → 규칙 기반 실제 계산(손해사정 PoC 패턴)
- **LLM 연동**: 추론·자동저작에 실제 모델 — **localhost 서버 + API 키 관리 필요**
- 커넥터: 최소 1~2개 모의/실제 API 연동
- *인프라 결정(서버 구조·키 관리)이 Phase 2 후 별도 착수*

### Phase 4 — 거버넌스 실체화  〔platform + design〕
- Run Trace 영속·필터·재현(replay), Policy 실제 차단(자동/확인/차단 분기), Evaluation·Skill Library 실데이터

### 가로지르는 디자인 작업  〔design〕
- **디자인 가이드 v0.2**: 기준선 v0.22 단일 HTML → 운영 콘솔 셸로 갱신(버전 드리프트 해소, Phase 0 동기)

## 5. 하이브리드 + 시연 모드 (260619 확정)

**유저 통찰**: 지금 산출물은 "실제 쓰는 AAP"가 아니라 **"AAP 작동을 보여주는 시뮬레이터"**다(분해=템플릿, 실행=각본 재생, 연동·실데이터·writeback 0). 이는 잘못이 아니라 Phase 3(실행성)를 미룬 의도된 상태.

**방향 = 하이브리드**: 범용 데모/시뮬레이터는 **설득·영업·개념 증명용으로 유지** + 한 업무를 **실제로 작동시키는 PoC**로 "진짜"를 증명.

**PoC 타깃 = 채용 스크리닝·코디네이션 (잠정 고정, 260619)** — 선정 기준 충족: ①도메인 지식 없이 이해 쉬움 ②진입 허들 낮음(입력=이력서·JD 텍스트, 외부연동 불필요·더미로 완결) ③인력 집약(HR 수백 건 1차 스크리닝·일정조율·커뮤니케이션). AAP 스토리 선명(분해: 스크리닝→매칭→일정→평가취합 / 5타입: 파싱 Agent·매칭 Module·캘린더 Connector·차별금지 Policy / HITL 자연 필수: 숏리스트 승인·최종 합격·오퍼 발송). **실작동 전환 최용이**(지능 자기완결적 — 이력서 넣으면 AI 추론→사람 승인, LLM만 붙이면 실작동, 외부시스템 의존 최소). 2순위=경비·지출 정산(룰+HITL 깔끔), 후보=CS/VOC(인력집약 최고·챗봇 리스크). 시연 모드 첫 시나리오로 사용.

**핵심 구조 = 하나의 플랫폼, 두 모드** (데모/실제를 별도 산출물로 가르지 ✕):
- **운영 모드(기본)**: 인박스·파이프라인·실행·관리·도메인 카탈로그 = 지향점 실제 AAP(지금 시뮬, Phase 3에서 한 업무 실작동).
- **시연 모드** = **좌측 네비 '시연' 메뉴 항목**(확정). 진입 → 시나리오 선택 → 운영 콘솔 화면 **위에 가이드 내레이션**(스포트라이트+툴팁+자동 진행)을 얹어 "프로세스 분해→구성→게이트→실행"을 한 흐름으로 시연.
- **왜**: 데모와 실제가 **같은 화면 공유** → 실작동으로 진화해도 시연 모드 그대로 동작(내레이션이 시뮬→실행 설명으로 바뀔 뿐). 두 벌 유지 불필요. 손해사정 v0.53+ "온디맨드 Guide Mode" 검증 패턴 차용.

## 6. "두 종류의 같음" + 채용 도메인 네이티브 surface (260619)

**유저 지적**: 지금 app/이 회의 데모와 거의 같은 포맷·플로우로 보임. → **같음에 2종 혼재**:
- **① 의도된 같음(정상)**: 인박스·3뷰 셸·8단계 캐논·HITL·5타입·거버넌스 = 공유돼야 표준화 증명.
- **② 의도 안 된 같음(문제)**: **실행(Run) 화면 자체**(좌 고객 서비스 화면+우 8계층+상단 업무순서) + 8단계 안무(…→**회의진행(시작/종료)**→…)가 **회의 전용 모양**인데 voc·quality 팩이 그 틀을 재사용 → 다 회의처럼 보임. surfaceSpec은 데이터 기반이라 도메인마다 다른 화면 가능하나, 팩들이 회의 모양에서 안 벗어남.

**표준 원칙(승격)**: 공유층(인박스·셸·캐논·HITL·거버넌스·5타입)=불변 공유 / **per-도메인 surface + 단계 안무는 도메인 네이티브로**(회의 리스킨 ✕). 손해사정 교훈("HR/툴처럼 안 보이게 ✕ → 진짜 업무 도구처럼 보이되 프로세스가 AAP식으로 작동")과 동일.

**순서 변경(유저 결정)**: **시연 모드 보류** → **채용 도메인 네이티브 surface 먼저.** 이유: 시연 첫 시나리오=채용인데 채용이 회의 리스킨이면 시연도 회의처럼 보임.

**다음 작업** (디자인 패스 완료·커밋 후):
1. **채용 도메인 네이티브 Pack/surface** (aap-platform 골격 → aap-design 시각): 좌측이 회의 포털 리스킨 ✕ → **ATS형**(후보 파이프라인 보드·이력서/JD 뷰·스코어카드·숏리스트). 단계 안무도 채용 고유(요건 접수→요건 분석→스크리닝 계획→★숏리스트 기준 승인→이력서 파싱·매칭·랭킹→면접 일정 조율→★합격·오퍼 승인→공유·학습), '회의진행' 단계 대체. 통합 인박스·X1·5타입·캐논 매핑 유지. 이 작업이 surfaceSpec이 진짜 도메인 네이티브 화면을 낼 수 있는지도 검증.
2. (이후) **시연 모드** — 네비 '시연' 항목 + 시나리오 선택(첫=채용) + 가이드 투어 엔진(운영 화면 위 오버레이).
